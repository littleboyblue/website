# ClaudeCode vs opencode vs Hermes Agent —— 三款AI编程智能体深度对比

---

*写作时间：April 15, 2026*

## 引言：AI编程正在发生变化

最近这两年，AI 编程工具发展得非常快。从最早的代码补全，到后来的对话式编程，再到现在能够完整完成任务的「AI 编程智能体」，整个领域演进的速度超出了很多人的预期。

当大家都在讨论大模型能力越强越好的时候，一个有趣的现象发生了：**三家不同背景的团队几乎在同一时间段推出了三个定位截然不同的 AI 编程智能体**——来自 Anthropic 的 ClaudeCode、开源社区的 opencode，还有 Nous Research 开源的 Hermes Agent。

这三件产品放在一起特别有意思：这三款都是2025年底到2026年初推出的新品，代表了三个不同方向的最新探索。

- 一个是主流大模型厂商亲自下场做的产品
- 一个是社区驱动的开源模块化设计
- 一个是独立研究机构开源的全自研框架

它们解决的其实是同一个问题——**让 AI 帮你写代码**，但走的却是三条完全不同的技术路线。

为什么会有这么大的差异？这些差异背后反映了什么样的设计思考？作为开发者或者研究者，我们应该如何选择？

这篇文章我会带你深入了解这三款工具，从定位、架构、设计理念到适用场景，做一个全面的对比分析，希望能帮助你找到最适合自己的那一款。

## ClaudeCode 是什么？

### 背景与定位

ClaudeCode 是 Anthropic 官方推出的 AI 编程智能体，随着 Claude 3.5 Sonnet 和 Claude 4 模型一起发布。作为大模型厂商亲自做的编程工具，它的定位非常清晰：**发挥 Claude 模型自身的长上下文和推理优势，给开发者提供开箱即用的一体化 AI 编程体验**。

简单说就是：Anthropic 做了最好的大模型，现在把这个模型直接包装成一个解决编程问题的完整产品，你不用折腾，装上就能用。

### 核心特点

ClaudeCode 最突出的几个特点：

**1. 深度集成 Claude 原生能力**

作为官方产品，ClaudeCode 第一个享受到 Claude 模型的所有新特性：
- 200K 起步，最大支持 1M+ token 的上下文窗口
- Claude 标志性的长文档理解能力，能一次性读懂整个代码库
- 强大的推理能力，特别是在复杂逻辑问题上表现出色

**2. VS Code 原生体验**

ClaudeCode 以 VS Code 扩展的形式交付，安装非常简单，登录你的 Anthropic 账号就能用。交互设计也做得很流畅：
- 侧边栏对话界面
- 代码 diff 预览
- 一键应用修改
- 支持多文件编辑

**3. 内置工具调用**

ClaudeCode 已经预置了常用的编程工具：
- 读取文件
- 编辑文件
- 运行终端命令
- 搜索代码
- Git 操作

这些都是内置做好的，不需要你自己配置。

### 架构设计

ClaudeCode 的架构其实相对简洁，它遵循 "Model First" 的设计思路：

```
┌─────────────────────────────────────────────┐
│  VS Code Extension (ClaudeCode)            │
│  ┌─────────────────────────────────────────┤
│  │ User Interface + Code Editor Integration │
│  └───────────────┬─────────────────────────┘
│                  │
│  ┌───────────────▼─────────────────────────┐
│  │  Tool Calling Layer (built-in)         │
│  │  - Read/Edit files                      │
│  │  - Terminal commands                    │
│  │  - Search                              │
│  └───────────────┬─────────────────────────┘
│                  │
└──────────────────┼─────────────────────────────┘
                   │
                   ▼
            ┌──────────────────┐
            │  Claude API      │
            │  (Claude 3.5/4)  │
            └──────────────────┘
```

从架构图可以看出几个关键点：

1. **工具能力内置**：所有工具都是预定义在扩展里的，不支持用户自定义扩展
2. **直连 Claude API**：核心推理完全在 Claude 模型端，客户端主要做界面和文件操作
3. **端到端闭环**：从用户提问到代码修改，整个流程产品化做好了

**设计决策**：ClaudeCode 选择把能力做在 VS Code 扩展里，而不是把智能体逻辑放在服务端。这样做的好处是：
- 文件操作都在本地，响应更快
- 代码不会全部传给 Anthropic 服务端（只有你让 AI 看的才传），隐私性更好
- 模型升级直接受益，不需要客户端改架构

### 优缺点总结

**优点**：
- 官方出品，体验流畅，开箱即用
- 充分发挥 Claude 模型长上下文优势
- 隐私性好，代码数据控制在本地
- 不需要自己部署，付费就能用

**缺点**：
- 闭源，不能二次开发
- 绑定 Claude 模型，不能换其他模型
- 不支持扩展能力，没法加自定义 Skill
- 需要付费调用 Anthropic API

总的来说，ClaudeCode 走的是**厂商一体化**路线：Anthropic 把模型、工具、体验都给你做好，你只管使用。对于普通开发者来说，这确实是最省心的选择。

## opencode 是什么？

### 背景与定位

opencode 是一个社区驱动的开源 AI 编程助手项目，它的定位和 ClaudeCode 完全不同。如果说 ClaudeCode 是一个开箱即用的产品，那么 opencode 更像是一个**模块化的 AI 智能体开发框架**，专注于 Skill 生态建设。

opencode 的核心理念是：**AI 智能体的能力应该是可插拔、可扩展、社区共建的**。每个人都可以开发自己的 Skill，任何人都可以随时使用这些 Skill。

### 核心设计理念

opencode 从一开始就遵循几个设计原则：

1. **模块化**：核心是调度引擎，所有能力都以 Skill 形式提供
2. **标准化**：通过 MCP（Model Context Protocol）统一接口，Skill 之间可以互操作
3. **开源开放**：完全开源，社区共建，任何人都可以贡献 Skill
4. **模型中立**：不绑定特定模型，可以搭配任意大模型使用

### 架构设计

opencode 的架构是典型的"核心+插件"设计：

```
┌─────────────────────────────────────────────────────┐
│                     opencode                         │
│  ┌─────────────────────────────────────────────────┤
│  │  Core Agent (调度引擎)                            │
│  │  - 用户需求理解                                   │
│  │  - 任务规划                                       │
│  │  - MCP 路由转发                                   │
│  └───────────────┬─────────────────────────────────┘
│                  │
│  ┌───────────────▼─────────────────────────────────┐
│  │  MCP Protocol Layer（标准化接口）                 │
│  └───────────────┬─────────────────────────────────┘
│                  │
│  ┌───────────────┼───────────────┐                 │
│  │               │               │                 │
│  ▼               ▼               ▼                 │
│ Skill A      Skill B      Skill C...              │
│ (code-review) (summarize) (design-to-code)        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

这个架构和 ClaudeCode 最大的区别在于：

- ClaudeCode：工具能力内置在核心里，不可扩展
- opencode：核心只做调度和路由，所有能力都是 Skill 外挂的

### Skill 机制详解

opencode 的 Skill 系统是它最核心的创新，让我详细解释一下。

**什么是 Skill？**

Skill 就是**一个独立封装的能力模块**。如果把 opencode 比作一个手机，那么 Skill 就是手机上的 App。手机本身提供通信和操作系统，你需要什么功能就装什么 App。

举几个 opencode 官方预置的 Skill 例子：

| Skill | 能力描述 |
|-------|----------|
| `design-to-code` | 设计图转代码，根据 Figma/图片生成像素级前端代码 |
| `summarize` | 文本摘要，能总结网页、PDF、视频文字内容 |
| `weather` | 查询实时天气 |
| `skill-vetter` | 安全审查，检查安装的 Skill 是否有安全问题 |

**Skill 的特点：**

1. **独立开发**：Skill 可以独立开发、测试、版本管理
2. **按需加载**：只用你需要的 Skill，不浪费资源
3. **社区共享**：你开发的 Skill 可以分享给其他人用
4. **不侵入核心**：新增 Skill 不需要修改 opencode 核心代码

**MCP 的作用**

MCP（Model Context Protocol）是 Skill 和核心 Agent 之间的通信标准。为什么需要标准？

想象一下，如果每个人开发 Skill 都用自己的接口格式：
- 核心 Agent 要记住每种不同的调用方式
- 换一个 Skill 就要改代码
- 开发者要适配不同的 Agent 框架

MCP 解决了这个问题：**大家都遵守同一个标准，只要符合标准，任何 Skill 都能被任何 Agent 使用**。

这就像 USB 接口标准：只要是 USB 设备，就能插任何 USB 接口，不用管设备是什么牌子。

### 使用方式

opencode 本身是一个命令行工具，以交互式 CLI 的方式工作：

```bash
# 安装
git clone https://github.com/anomalyco/opencode
cd opencode && npm install

# 使用
opencode "帮我 review 一下这个 PR 的代码"
```

因为是完全开源的，你可以：
- 自由修改源码
- 自己开发 Skill
- 部署在自己的服务器上
- 搭配任意支持工具调用的大模型使用（OpenAI、Anthropic、通义千问、文心一言等）

### 优缺点总结

**优点：**
- 完全开源免费
- 模块化设计，Skill 生态可扩展
- 模型中立，支持任意大模型
- 社区驱动，每个人都能贡献
- 架构清晰，适合二次开发

**缺点：**
- 需要自己部署，有一定使用门槛
- 目前社区 Skill 还在建设中，数量不如成熟生态多
- CLI 交互，不如 IDE 集成体验流畅

opencode 走的是**开源模块化**路线，它不试图做一个大而全的产品，而是提供一个灵活的框架，让社区一起建设生态。如果你是开发者，想要定制自己的 AI 助手，这是一个非常好的选择。

## Hermes Agent 是什么？

### 背景与定位

Hermes Agent 是 [Nous Research](https://github.com/NousResearch) 开源的一款**面向代码场景的大语言模型智能体框架**，项目地址：<https://github.com/NousResearch/hermes-agent>。它和 ClaudeCode、opencode 的定位又不一样。

如果说：
- ClaudeCode = 开箱即用的产品
- opencode = 模块化生态框架

那么 Hermes Agent = **研究性的全栈智能体解决方案**。它基于 Nous Research 在大模型微调领域的经验，更侧重于展示"如何用一个专门训练过的智能体来解决编程问题"，从规划、搜索到执行，整个流程都是自研设计。

Hermes Agent 的核心定位是：**给开源社区提供一个完整的、可复现的AI编程智能体基线，方便学术界和工业界在此基础上继续研究。**

### 设计思路与核心特点

Hermes Agent 的设计思路和前两者有明显区别，它从一开始就考虑"什么样的架构最适合复杂编程任务"，提出了一个分层设计：

```
┌─────────────────────────────────────────────────────┐
│                   Hermes Agent                      │
│  ┌─────────────────────────────────────────────────┤
│  │  Planner (规划器)                                 │
│  │  - 理解用户需求                                   │
│  │  - 分解任务步骤                                   │
│  └───────────────┬─────────────────────────────────┘
│                  │
│  ┌───────────────▼─────────────────────────────────┐
│  │  Retriever (检索器)                               │
│  │  - 代码库搜索                                     │
│  │  - 相关文档检索                                   │
│  └───────────────┬─────────────────────────────────┘
│                  │
│  ┌───────────────▼─────────────────────────────────┐
│  │  Executor (执行器)                               │
│  │  - 文件读写                                     │
│  │  - 代码生成与编辑                               │
│  │  - 命令执行                                     │
│  └───────────────┬─────────────────────────────────┘
│                  │
│  ┌───────────────▼─────────────────────────────────┐
│  │  Critic (审查器)                                 │
│  │  - 结果检查                                     │
│  │  - 错误修复                                     │
│  └─────────────────────────────────────────────────┘
└─────────────────────────────────────────────────────┘
```

这个设计相比 ClaudeCode 和 opencode，分工更细：

| 模块 | 职责 | Hermes 的特点 |
|------|------|--------------|
| Planner | 任务拆解 | 使用专门微调过的模型做规划，更懂编程任务分解 |
| Retriever | 信息检索 | 针对代码库做了向量检索优化，能快速找到相关代码 |
| Executor | 执行操作 | 支持多种工具调用，安全沙箱执行代码 |
| Critic | 结果审查 | 自检纠错，减少AI幻觉 |

### 针对编程场景的优化

Hermes Agent 在几个关键地方针对编程场景做了特别优化：

**1. 分层专业化**

ClaudeCode 基本是"一个模型走到底"，靠模型本身的能力搞定一切。Hermes 则是把不同阶段交给不同模块，每个模块可以用不同的模型，甚至不同模型能力可以专门微调。

好处是：每个模块可以专业化，分工更精细，整体效果更好；坏处是架构更复杂，部署成本更高。

**2. 代码检索增强**

对于大仓库，AI 需要快速找到和任务相关的代码文件。Hermes 在这方面做了专门优化，支持：
- 基于向量的代码语义检索
- 基于符号（函数名/类名）的检索
- 混合检索策略

这比纯靠大模型上下文窗口要更高效，特别适合大型项目。

**3. 自我纠错机制**

Hermes 设计了专门的 Critic 模块，生成代码后会自己检查一遍：
- 语法对不对？
- 逻辑合理吗？
- 测试能通过吗？
- 有安全问题吗？

发现问题自己先尝试修复，再给用户最终结果。这个思路能显著提高成功率。

### 使用场景

Hermes Agent 更适合这些场景：

1. **学术研究**：如果你是做 AI 智能体研究的，Hermes 给了你一个完整的可复现基线，你可以在它基础上改某个模块做实验。

 2. **企业内部二次开发**：Nous Research 开源这个框架本来就是给大家参考的，如果你的公司想要自己搭建一套内部的 AI 编程智能体，可以拿 Hermes 当起点。

3. **大型代码库任务**：如果你经常需要在大项目上做跨文件修改，Hermes 的检索增强设计在这种场景下更有优势。

### 优缺点总结

相比 ClaudeCode 和 opencode，Hermes Agent 的优缺点很明显：

**优点：**
- 架构设计清晰，职责分离，每个模块都可以独立优化
- 针对大型代码库做了检索增强，处理大项目更好
- 内置自我纠错机制，成功率更高
- 完全开源，Nous Research 技术背书
- 研究和工业界都能用

**缺点：**
- 架构相对复杂，部署和运维成本更高
- 需要更多依赖（比如向量数据库）
- 定位是框架/基线，开箱即用体验不如 ClaudeCode
- 生态还在初期，不如成熟生态丰富

Hermes Agent 走的是**研究驱动的全栈自研路线**，它展示了一种"专业化分工"的智能体架构思路——把复杂任务拆成多个阶段，每个阶段用专门的模块处理。这种思路在复杂问题上可能会有更好的表现。

## 核心维度对比

看完三个工具的分别介绍，我们来做一个系统的对比。我整理了一个表格，从多个关键维度对比三者的差异：

| 对比维度 | ClaudeCode | opencode | Hermes Agent |
|----------|------------|----------|--------------|
| **开发方** | Anthropic（厂商） | 开源社区 | Nous Research（研究机构） |
| **定位** | 开箱即用的产品 | 模块化生态框架 | 全栈研究型解决方案 |
| **开源** | ❌ 闭源 | ✅ 完全开源 | ✅ 完全开源 |
| **模型绑定** | 绑定 Claude | 不绑定，支持任意模型 | 不绑定，可换模型 |
| **架构风格** | 一体化，单模型到底 | 核心调度 + Skill 模块化 | 分层专业化（四模块） |
| **扩展能力** | ❌ 不可扩展 | ✅ Skill 生态，高度可扩展 | ✅ 模块级可扩展 |
| **Skill 支持** | 无，工具内置 | 第一等公民，MCP 标准 | 支持自定义工具，无标准 |
| **开发 Skill 成本** | - | 低，标准化接口 | 高，需改框架 |
| **标准协议** | 私有协议 | 支持 MCP | 私有架构 |
| **落地路径** | 直接用，零落地成本 | 自部署，按需加 Skill | 自部署，二次开发 |
| **部署方式** | VS Code 插件，直连 API | 本地部署，CLI | 本地/服务端部署 |
| **依赖要求** | 极低（只需账号） | 低（Node.js + API Key） | 中高（需要向量库等） |
| **使用门槛** | 极低，新手友好 | 中等，需会部署 | 较高，需技术基础 |
| **目标用户** | 普通开发者 | 定制化需求者/社区开发者 | 研究者/企业二次开发 |

下面我对几个关键维度做进一步分析。

### 架构设计对比

三款工具代表了三种完全不同的架构哲学：

**ClaudeCode：Monolithic（一体化）**
- 所有功能做在一起
- 工具内置，不可扩展
- 依赖模型本身能力
- 优点：简单，体验一致；缺点：封闭，不能进化

**opencode：Modular（模块化）**
- 核心只做调度和路由
- 所有能力都是 Skill 外挂
- 遵循 MCP 标准
- 优点：灵活，生态可生长；缺点：核心简单，依赖社区建设

**Hermes Agent：Layered（分层专业化）**
- 按流程阶段拆成多个模块
- 每个模块专业分工
- 每个模块可单独优化
- 优点：复杂任务效果好；缺点：架构复杂，部署成本高

这三种架构没有绝对的对错，只是目标不同：
- 如果你追求**用户体验**，一体化最简单
- 如果你追求**生态生长**，模块化最有潜力
- 如果你追求**效果最优**，分层专业化可能更好

### 开放性对比

开放性是这三者最显著的差异：

| 项目 | 开源 | 修改 | 二次开发 | 扩展能力 |
|------|------|------|----------|----------|
| ClaudeCode | ❌ | ❌ | ❌ | ❌ |
| opencode | ✅ | ✅ | ✅ | ✅（Skill） |
| Hermes Agent | ✅ | ✅ | ✅ | ✅（模块） |

ClaudeCode 作为商业产品，闭源是正常商业选择，它给你体验，但不给你控制权。opencode 和 Hermes 都是开源，但开放的粒度不同：
- opencode 开放粒度是**Skill**，鼓励大家贡献能力模块
- Hermes 开放粒度是**整个框架**，鼓励大家改进架构本身

### 使用成本对比

从普通用户的使用成本来看：

```
Cost: ClaudeCode < opencode < Hermes Agent
```

- **ClaudeCode**：打开 VS Code，装插件，登录就能用，成本几乎为零，只用付 API 调用费。
- **opencode**：需要克隆代码，安装依赖，配置 API Key，有一点门槛，但对开发者来说不算难。
- **Hermes Agent**：需要部署多个组件，可能还要向量数据库，环境配置相对复杂，门槛最高。

### 生态对比

目前三者的生态都还在发展早期，但走的路线完全不同：

- **ClaudeCode**：生态在 Anthropic 内部，由官方负责更新，用户只需要用，不能自己加东西。
- **opencode**：生态在社区，每个人都可以写 Skill 分享，社区共建。
- **Hermes Agent**：生态在研究领域，作为基线给学术界和工业界二次开发。

## 设计理念差异分析

看完表格对比，我们再深入一层——这三者设计理念的差异，本质上反映了什么？

### 是谁在做，决定了怎么做

有趣的是，这三个项目背后的开发主体，正好对应了三种不同的开发哲学：

**大模型厂商做产品——用户体验优先，闭环最重要**

Anthropic 作为 Claude 模型的开发者，它做 ClaudeCode 的天然优势就是模型+产品一体化，把体验做到极致，让更多人用 Claude，这对它来说就是最正确的战略。闭源不开放是商业选择，很合理。

**社区做开源项目——开放优先，生态最重要**

opencode 从一开始就是社区项目，它的思路就是"框架比产品重要"，我不给你做一个大而全的产品，但我给你一个灵活的架子，大家一起往上加能力。这种思路能不能成，取决于社区愿不愿意一起玩。

**研究机构做项目——探索优先，技术最重要**

Nous Research 做 Hermes Agent，目的不是做一个消费品给普通人用，而是基于他们在大模型微调领域的积累，探索"AI 编程智能体究竟能做到什么程度"，把研究成果开源给社区，证明分层专业化这条路走得通。它的价值更多在于技术示范，而不是直接给终端用户用。

可以说：**屁股决定脑袋，定位决定设计**。不同的主体，有不同的目标，自然就走出了不同的技术路线。

### 对"AI 能做多少"的判断不同

三款设计背后，其实隐含了设计者对 AI 能力的不同判断：

**ClaudeCode：相信大模型本身就能搞定一切**

ClaudeCode 的"单模型走到底"设计，背后的信心是——现在大模型已经足够强了，不需要搞那么多复杂的模块拆分，让模型自己搞定一切就好了。用户要的就是结果，越少折腾越好。

**opencode：AI 需要靠工具生态扩展能力**

opencode 的模块化思路，背后的判断是——大模型再强，也不可能搞定所有领域的任务。专业的事情要专业的模块来做，通过标准化接口把各种能力组合起来，才能应对无限多样的用户需求。

**Hermes Agent：复杂任务需要专业化分工**

Hermes 的分层设计，背后的判断是——复杂编程任务太复杂了，一个模型搞定容易出错，把规划、检索、执行、审查分开，每个环节用专门的模块来做，这样整体成功率更高。

三种判断，三种选择，目前来看，哪种对？可能都对——在不同的规模、不同的场景下，各有优劣。

### 路线之争：一体化 vs 模块化

这几年 AI 工具领域一直有个争论：**到底是一体化产品好，还是模块化框架好？**

ClaudeCode 代表了一体化路线：
- 优点：体验流畅，不用折腾
- 缺点：锁死在厂商，你不能改

opencode + Hermes 代表了开源路线：
- 优点：开放，自由，可定制
- 缺点：需要自己折腾，门槛高

我的观察是：这个市场最后会是**分层共存**。

- 普通用户大多数会选择开箱即用的一体化产品
- 开发者和有定制需求的人会选择开源框架
- 不同需求，不同选择

简单总结就是：
- ClaudeCode：厂商给你做好一切，你只用就行
- opencode：社区共建生态，每个人都能贡献
- Hermes：研究院探索技术边界，给出最佳实践基线

### Agent 落地与 Skill 生态：产品经理视角

从产品落地的角度看，Skill 机制本质上解决了一个核心问题：**如何用有限的研发投入，满足无限多样的用户需求？**

我们从落地的各个环节对比一下：

#### 1. 从需求到落地：路径长度对比

**ClaudeCode**：路径最短，需求 → 使用，没有落地成本。但如果你的需求不在内置工具里，就只能等厂商更新，落地周期以月计。

**opencode**：需求 → 开发 Skill → 所有人可用，落地周期以天计。因为有 MCP 标准，Skill 开发好就能被所有人用，边际成本为零。

**Hermes Agent**：需求 → 修改框架 → 测试集成，落地周期以周计。没有统一标准，每个新能力都要改框架，边际成本较高。

```
落地速度：ClaudeCode > opencode > Hermes Agent
（有现成满足你需求的工具时）

落地速度：opencode > Hermes > ClaudeCode
（需要定制新能力时）
```

#### 2. Skill 开发成本对比

| 项目 | Skill 地位 | 标准化 | 开发成本 | 复用性 |
|------|-----------|--------|----------|--------|
| ClaudeCode | N/A（无Skill） | - | - | - |
| opencode | 第一等公民 | MCP 标准统一 | 低，只需要实现业务逻辑 | 高，任何支持 MCP 的 Agent 都能用 |
| Hermes | 工具散在模块里 | 无统一标准 | 高，需要理解框架架构 | 中，只能在 Hermes 内用 |

opencode 的设计思路在这里优势非常明显：**Skill 是第一等公民**，框架只做调度，所有能力都是 Skill。因为 MCP 标准统一，开发者只需要关心自己的业务逻辑，不需要关心怎么和核心 Agent 对接，开发成本大大降低。

#### 3. 生态建设的可能性

Skill 生态能否做起来，关键看三点：
- **准入门槛**：会不会开发 Skill 太难？
- **收益成本比**：开发一个 Skill 得到的收益够不够覆盖成本？
- **网络效应**：越多人用，越多人开发，正向循环能否建立？

| 项目 | 准入门槛 | 收益成本比 | 网络效应 |
|------|----------|------------|----------|
| ClaudeCode | 封闭生态，不允许第三方 | 厂商独占收益 | 不可能形成第三方生态 |
| opencode | 低，标准清晰，容易上手 | 开发一次，所有人可用 | 容易形成网络效应 |
| Hermes | 高，需要理解整体架构 | 二次开发为主，共享难度大 | 难形成大规模 Skill 生态 |

从产品经理角度看，**opencode 的设计其实最符合生态生长的规律**：低准入门槛 + 标准化 + 开放共享，这是让生态自发生长的最佳架构。

#### 4. 企业落地场景的差异

对于企业想要落地 AI 编程智能体，三者的适合程度也不同：

- **ClaudeCode**：适合作为开发者个人工具直接采购使用。如果企业有内部定制需求（比如连接内部系统），因为不支持扩展，没法满足。
- **opencode**：适合企业搭建自己的 AI 助手平台。企业可以把内部各种能力封装成 Skill（比如连接内部代码库、调用内部服务），然后让所有开发者使用，非常灵活。
- **Hermes Agent**：适合有一定技术能力的中大型企业，想要从头搭建自己的智能体系统，Hermes 提供了一个很好的分层架构参考，可以在此基础上改造。

#### 5. 我的产品判断：Skill 化是未来方向

从产品演化的角度看，**Skill 化是必然趋势**。原因很简单：

1. **需求多样性**：用户需求千差万别，厂商不可能内置所有能力
2. **迭代速度**：Skill 独立迭代比整体版本迭代快得多
3. **生态杠杆**：社区一起开发 Skill，比厂商一个团队开发效率高得多
4. **用户定制**：不同用户可以加载不同 Skill，个性化满足需求

ClaudeCode 现在不开放 Skill，是商业策略选择，它需要保证体验一致性。但从长期看，开放 Skill 生态是更大的趋势。

## 如何选择适合你的工具？

讲完了对比和分析，最实际的问题来了：**你应该选哪个？**

我从产品落地场景和 Skill 需求角度，重新整理一下推荐建议：

### 按用户角色选择

| 用户角色 | 推荐 | 理由 |
|----------|------|------|
| 普通开发者 | ClaudeCode | 开箱即用，不用折腾，付费就能用 |
| 喜欢折腾的开发者 | opencode | 可以自己加 Skill，定制自己的 AI 助手 |
| AI 研究者 | Hermes Agent | 完整可复现基线，方便做实验 |
| 企业架构师 | opencode / Hermes | 需要定制内部能力，选开源方案 |

### 按落地需求选择

### 如果你只需要直接使用，没有定制需求 → 选 ClaudeCode

推荐理由：
- 开箱即用，装上 VS Code 插件就能干活
- Anthropic 官方优化，体验真的流畅
- Claude 本身模型能力就强，长上下文优势明显
- 不用折腾部署、配置、依赖，落地成本为零

不推荐的情况：
- 你需要定制特定能力（比如连接内部系统）
- 你不希望用闭源商业产品
- 你想做二次开发

### 如果你需要定制能力，想要搭建自己的Skill生态 → 选 opencode

推荐理由：
- 完全开源免费
- Skill 机制设计成熟，MCP 标准统一
- 新增 Skill 成本低，开发一次就能所有人复用
- 模型中立，你想用什么模型就用什么模型
- 非常适合企业把内部能力封装成 Skill 给团队使用
- 社区生态逐渐成型，可以直接用别人开发好的 Skill

从产品落地角度看，如果你的组织有定制化需求，opencode 是目前三者中设计最合理的选择。

不推荐的情况：
- 你想要开箱即用的极致体验
- 你完全不想自己动手配置

### 如果你在做 AI 智能体研究，或者要从头搭建框架 → 选 Hermes Agent

推荐理由：
- Nous Research 出品，分层架构设计经过研究验证
- 对大型代码库的检索增强做了专门优化
- 四个模块职责分离，方便你针对某个模块做研究改进
- 完全开源，可以作为企业自研的起点

不推荐的情况：
- 你只是普通用户，想找个现成工具用
- 你想要快速落地定制 Skill，嫌架构复杂

### 按 Skill 需求选择

| 你的需求 | 选这个 |
|----------|--------|
| 不需要自定义能力 | ClaudeCode |
| 需要快速开发可复用Skill | opencode |
| 需要定制整个架构 | Hermes Agent |

### 选择决策树

一句话帮你决策：

```
你需要开箱即用？
    → ClaudeCode
你需要定制Skill能力？
    → opencode
你需要研究/重构整个Agent架构？
    → Hermes Agent
```

## 总结与展望

我们来回顾一下这三款工具：

| 项目 | 路线 | 最适合 |
|------|------|--------|
| ClaudeCode | 厂商一体化 | 普通开发者，追求体验 |
| opencode | 开源模块化 | 定制化需求，社区生态 |
| Hermes Agent | 研究型分层 | 学术研究，企业二次开发 |

这三个项目放在一起看特别有意思，它们代表了当前 AI 编程智能体领域三个完全不同的探索方向：

1. **商业产品路线**：大模型厂商直接做产品给终端用户，体验优先，闭源商业化
2. **开源社区路线**：社区做框架，大家一起贡献能力模块，生态优先
3. **研究探索路线**：独立研究机构探索技术边界，开源基线推动领域进步

### 未来发展趋势

从我个人的观察来看，AI 编程智能体这个领域接下来会往几个方向走：

**1. 模块化会成为共识**

不管是商业还是开源，把能力拆成可插拔的模块是大势所趋。用户需求太多样了，大而全的内置工具永远满足不了所有人，开放生态才能走得更远。

**2. 专业化分工越来越细**

Hermes 走的分层专业化路线，我觉得这个方向会越来越重要。复杂任务真不是一个模型能搞定的，规划、检索、执行、审查分开，每个环节做专业优化，整体效果会更好。

**3. 标准协议的重要性提升**

MCP 这类标准协议会越来越重要，当大家都遵守同一个标准，Skill 才能跨框架复用，生态才能真正做起来。

**4. 开源闭源会长期共存**

不用争论开源一定比闭源好，或者反过来。它们服务不同的用户，满足不同的需求，自然会长期共存。用户用脚投票，市场会选出赢家。

### 最后

ClaudeCode、opencode、Hermes Agent 这三个项目虽然路线不同，但目标其实是一样的——**让 AI 真正帮开发者写代码，把开发者从重复劳动中解放出来**。

不管你是普通开发者、研究者、还是开源爱好者，相信这三个项目中总有一款适合你。我也期待看到这个领域接下来更快的发展，未来 AI 编程真的可能彻底改变我们写代码的方式。
