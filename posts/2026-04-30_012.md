# AI Agent 的尽头，为什么可能还是数据库

最近读到一篇关于图灵奖得主 Mike Stonebraker 的访谈，标题很抓人：他不确定今天是否还会建议 18 岁的人学计算机，并且认为 AI Agent 最后会变成数据库问题。

如果只看表层，这篇访谈最容易被传播的部分，是他对 Oracle、Google、AWS 这些公司的尖锐评价：Oracle 当年靠销售能力赢过技术更强的对手；Google 的 MapReduce 和最终一致性被他认为是错误方向；AWS 同时维护太多数据库，很多并没有足够存在价值。

但我觉得真正值得讨论的，不是他骂了谁，而是他反复提醒的一件事：**很多看起来很新的技术问题，最后都会回到一些非常老的系统工程问题。**

尤其是在 AI Agent 这个方向上，这个判断很值得技术人认真想一想。

## 1. 不要迷信大厂光环

技术行业有一种很常见的心理：只要某个方案来自 Google、Amazon、Meta、OpenAI，就天然觉得它一定代表未来。

这当然有原因。大厂有顶级工程师，有超大规模业务，有真实流量，有足够资源做复杂系统。它们的技术实践通常很值得学习。

但问题在于，大厂的方案首先是为它自己的约束服务的。

Google 当年推 MapReduce，是为了解决它自己的大规模数据处理问题。Amazon 有很多数据库产品，也和它的云业务、客户分层、商业模式有关。一个技术方案在大厂场景下成立，不代表它在普通公司、普通团队、普通业务里也成立。

这也是 Stonebraker 批评 MapReduce 和最终一致性的核心：行业里有太多人不是因为真正理解了问题，而是因为“Google 很聪明，所以 Google 的方案一定对”。

技术人最需要警惕的，就是这种“权威替代理解”。

看到一个新方案时，真正该问的不是：

“这是谁做的？”

而是：

- 它解决的具体问题是什么？
- 它牺牲了什么，换来了什么？
- 它的前提条件在我的场景里是否存在？
- 如果规模、团队能力、业务要求不同，它还适用吗？

大厂经验值得学习，但不能照搬。否则学到的可能不是先进性，而是不适配。

## 2. Agent 从只读到读写，才是真正分水岭

现在很多 AI Agent 看起来很强：能读文档、查资料、总结会议、生成代码、分析客户、写报告。

但这些能力大多还停留在“只读”阶段。也就是说，Agent 只是读取信息、加工信息、给出建议。即使它说错了，最多是建议不准、报告有误、代码需要修改，损失相对可控。

真正的分水岭，是 Agent 开始进入“读写”世界。

比如：

- 自动批准退款
- 自动修改库存
- 自动发起转账
- 自动调整用户信用额度
- 自动变更订单状态
- 自动调用内部系统完成审批

到了这一步，问题就完全不一样了。

如果一个 Agent 要把 A 账户的 100 元转给 B 账户，它不能只“看起来理解了任务”。它必须保证：A 扣款成功，B 入账成功；中间任何一步失败，都要能回滚；并发情况下不能重复扣款；系统崩溃后也不能出现半完成状态。

这些问题的名字，数据库领域早就讲了几十年：事务、原子性、一致性、隔离性、持久性、并发控制、故障恢复。

所以 Stonebraker 说 Agentic AI 最后会变成数据库问题，我认为是非常准确的。

很多人讨论 Agent 时，把重点放在 prompt、模型、推理能力、工具调用上。但只要 Agent 进入真实业务系统，最关键的问题就会变成：**它的动作能不能被正确提交、回滚、审计和约束。**

这不是模型聪不聪明的问题，而是系统能不能保证正确性的问题。

## 3. text-to-SQL 的落差，提醒我们不要被 benchmark 麻醉

访谈里还有一个很刺眼的例子：公开 text-to-SQL benchmark 上，大模型准确率可以做到 80% 以上；但在他们用真实数据仓库做的测试里，结果是 0%。即使用 RAG 增强，或者把 join 条件喂给模型，效果也远远不够生产可用。

这个数字很极端，但背后的现象并不陌生。

很多 AI demo 都有类似问题：演示环境很顺，公开榜单很好看，真实业务一接入就暴露大量边界条件。

为什么真实数据仓库这么难？原因并不神秘：

- 真实 schema 往往不干净，表名和字段名充满缩写、历史包袱和业务黑话
- 真实 SQL 往往很长，包含复杂 join、子查询、物化视图和本地规则
- 真实数据没有出现在模型训练语料里，模型无法靠“见过类似文本”来猜
- 真实业务里的概念高度本地化，只有组织内部人才知道含义
- 真实场景容错率很低，不能接受“差不多对”

这说明一个很重要的问题：**benchmark 只能说明模型在某个标准化任务上的表现，不能直接等同于生产可用性。**

尤其是企业系统，最难的往往不是语法，而是上下文。表为什么这么设计？字段真实含义是什么？哪些数据是脏的？哪些规则是历史遗留？哪些约束不能被打破？这些东西不在模型参数里，也不一定在文档里。

所以，text-to-SQL 很有价值，但短期内更适合作为辅助工具，而不是让非技术人员直接对复杂数仓“自然语言点菜”。

让模型生成 SQL 并不难，难的是让它在复杂业务语境中稳定生成正确 SQL。

## 4. Postgres 是好起点，但不是万能答案

Stonebraker 提到的“One size fits none”也很值得反复咀嚼。

他的意思不是通用数据库没有价值，而是任何试图“一套系统解决所有问题”的想法，在高要求场景下都会遇到边界。

Postgres 是一个非常好的起点。它免费、成熟、生态强、扩展多、社区大，适合绝大多数中小规模业务。很多项目从 0 到 1，甚至从 1 到 10，用 Postgres 都完全没问题。

但如果场景变成每秒百万级事务、PB 级数据仓库、大规模向量检索、低延迟流处理、复杂图计算，继续用一个通用系统硬扛，往往会付出巨大代价。

这不是 Postgres 不好，而是场景不同。

对技术选型来说，最重要的不是追求“最先进”，也不是追求“一个工具打天下”，而是理解负载特征：

- 是读多还是写多？
- 是事务型还是分析型？
- 是强一致还是最终一致可以接受？
- 是低延迟优先还是吞吐优先？
- 是结构化数据还是非结构化数据？
- 是单机能解决，还是必须分布式？

真正成熟的工程判断，不是“我喜欢哪个数据库”，而是“这个 workload 应该由什么系统承载”。

## 5. 计算机科学不再是增长型行业？我只同意一半

访谈里最容易引发情绪的部分，是 Stonebraker 说他不确定今天还会不会建议 18 岁的人学计算机，因为计算机科学未来可能不再是增长型行业。

我觉得这句话要拆开看。

如果所谓“学计算机”，只是学几门语言、会写 CRUD、会调 API、会照着教程搭项目，那这种技能的稀缺性确实会下降。AI 编程工具、低代码平台、云服务和成熟框架，会不断压缩低复杂度开发的价值。

过去很多岗位靠“会写代码”就能获得不错收入。未来，仅仅会写代码可能不够了。

但这不意味着计算机不值得学。

相反，越是 AI 普及，越需要真正懂系统的人：

- 懂数据建模，知道业务对象如何落到结构化系统里
- 懂分布式系统，知道一致性、可用性、延迟之间如何权衡
- 懂可靠性，知道失败一定会发生，系统必须能恢复
- 懂安全和权限，知道自动化执行不能没有边界
- 懂业务流程，知道技术动作背后对应什么现实后果
- 懂工程落地，知道 demo 到生产之间隔着多少坑

AI 会降低“写代码”的门槛，但不会消灭“构建可靠系统”的难度。

甚至可以说，AI Agent 越强，系统工程越重要。因为过去是人手动点按钮，人会在中间犹豫、确认、判断；未来如果 Agent 自动执行，系统本身就必须承担更多约束、审计、回滚和安全责任。

所以我更愿意把这句话改成：**浅层计算机技能不再稀缺，但深层系统能力会更值钱。**

## 6. 对 IT 从业者的启发

这篇访谈对普通 IT 从业者最大的启发，不是“赶紧去学数据库”，也不是“不要学 AI”。

真正的启发是：不要只追概念，要回到底层问题。

AI Agent、RAG、text-to-SQL、workflow、向量数据库、多模态，这些名词会持续变化。但真实业务系统里长期不变的问题仍然是：数据是否可信？状态是否一致？失败能否恢复？权限是否可控？成本是否可接受？用户是否真的获得价值？

几个具体建议：

- 学 AI，但不要只学 prompt，也要学数据、系统和工程可靠性
- 看 benchmark，但不要把榜单当生产结论，要看真实 workload
- 用 Agent，但要区分只读建议和读写执行，两者风险完全不同
- 做技术选型时，先理解负载，再选择工具，不要被宣传语牵着走
- 面对大厂方案，学习其思想，但不要默认适合自己
- 面对新工具，问它是否能进入生产闭环，而不只是 demo 好不好看

技术行业永远会有新名词，但底层问题不会因为换了包装就消失。

## 结语：AI 越强，越绕不开老问题

很多人以为 AI Agent 的未来，是模型越来越聪明，工具越来越多，自动化越来越彻底。

这当然是趋势。但另一条更隐蔽的趋势是：当 Agent 真正进入业务执行层，它就必须面对数据库和分布式系统那些老问题。

只读世界里，错了可以重试；读写世界里，错了可能就是钱、库存、订单、信用、权限和合规问题。

所以，AI Agent 的尽头不一定只是更大的模型，也可能是更严肃的系统工程。

真正长期有价值的技术能力，不是知道多少新概念，而是能把模型、数据、流程、权限和可靠性组合成一个可运行、可恢复、可验证的系统。

这也许正是这篇访谈最值得我们记住的地方：技术会不断换名字，但正确性、可靠性和工程判断，始终不会过时。
