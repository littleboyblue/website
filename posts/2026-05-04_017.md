# 向量库、知识图谱与本体论：RAG 落地到底该怎么做？

最近看到一篇虎嗅文章，标题很有冲击力：**向量库是 RAG 的伪命题，知识图谱是答案，本体论是灵魂。**

这个判断很容易引起讨论。

一方面，文章确实指出了很多企业知识库和 RAG 项目里的真实问题：只靠向量库做语义召回，很容易出现上下文割裂、例外条件丢失、证据链不完整和模型幻觉。另一方面，如果直接得出“向量库没用，知识图谱才是唯一答案”，又有点过度简化。

我更倾向于把这件事拆开看：**向量库不是伪命题，但它不是 RAG 的全部；知识图谱很重要，但它不是万能答案；本体论真正有价值的地方，是给复杂领域定义清楚业务边界和推理规则。**

真正可落地的方向，不是用某个技术替代另一个技术，而是把向量检索、关键词检索、结构化规则、知识图谱、证据溯源和大模型编排组合起来。

## 1. 为什么单靠向量库会出问题

早期做 RAG 系统时，向量库几乎是标配。

典型流程是：

```text
文档切分 -> 向量化 -> 相似度召回 -> 塞进 Prompt -> 大模型生成答案
```

这套流程的优点很明显：通用、简单、容易上线。只要把文档丢进去，系统就能根据用户问题召回语义相近的片段，再让大模型组织成自然语言答案。

但它的问题也很明显。

向量库解决的是“语义相似”，不是“逻辑正确”。它能找到和问题看起来相关的文本片段，但不能保证这些片段包含完整上下文、例外条件、约束关系和推理路径。

比如电商退款场景里，用户问“退款多久到账”。系统可能召回主条款：“退款 T+1 到账”，却漏掉下一段：“黑名单用户、已发货订单除外”。这时模型如果回答“一律 T+1 到账”，就可能带来真实业务风险。

医疗场景会更危险。如果药品说明里，“可用于高血压治疗”和“孕妇禁用”被切到了不同 chunk，系统只召回前者，就可能给出非常危险的建议。

所以，传统向量 RAG 的问题并不是“向量库完全没用”，而是它天然容易遇到几个风险：

- 上下文割裂
- 例外条款丢失
- 表格和层级结构被破坏
- 多跳关系无法表达
- 证据链不完整
- 高风险结论缺少校验

这些问题不是换一个更大的向量库就能彻底解决的。

## 2. 知识图谱真正解决的是什么

知识图谱的价值，不是“比向量库更高级”，而是它处理的是另一类问题。

向量检索关注的是：

```text
这段内容和用户问题像不像？
```

知识图谱关注的是：

```text
这个实体是什么？
它和其他实体有什么关系？
这个关系是因果、适应证、禁忌、并发，还是弱相关？
这个结论能不能被推导出来？
```

比如医疗场景中，可以把知识组织成这样的结构：

```text
糖尿病 -> 表现为 -> 口渴
糖尿病 -> 表现为 -> 频繁排尿
糖尿病 -> 治疗药物 -> 胰岛素
二甲双胍 -> 治疗 -> 2 型糖尿病
二甲双胍 -> 可能导致 -> 体重下降
```

这和把医学指南切成一段段文本再做相似度召回，是两种不同的知识组织方式。

图谱更适合处理：

- 多实体关联
- 多跳推理
- 因果链路
- 约束条件
- 风险校验
- 证据追踪

例如用户输入“咳嗽、发热、胸痛、呼吸急促”，系统可以沿着图谱推理：

```text
症状 -> 呼吸系统疾病
呼吸系统疾病 -> 肺炎 / 支气管炎 / COPD
肺炎 -> 需要检查 -> 血氧、白细胞、胸部影像
胸部影像浸润阴影 -> 高度怀疑 -> 肺炎
```

这比单纯召回几段相似文本更有结构，也更容易解释给医生或业务人员听。

但这不代表图谱可以替代所有检索。开放问答、模糊表达、长文档搜索、泛知识总结，向量检索依然有价值。图谱更适合那些实体清晰、关系重要、推理链条明确、答案需要可解释的场景。

## 3. 本体论不是口号，而是领域 Schema

文章里说“本体论是灵魂”，这个说法听起来有点抽象，但背后的工程含义其实很朴素。

在构建知识图谱之前，必须先定义清楚这个领域里的对象类型、关系类型和推理规则。否则图谱只是很多三元组的堆积，看起来有结构，实际可能会乱推理。

例如：

```text
二甲双胍 -> 治疗 -> 2 型糖尿病
二甲双胍 -> 可能导致 -> 体重下降
二甲双胍 -> 通过 AMPK 通路影响 -> 代谢调节
```

这三条关系都可能是对的，但它们的医学含义完全不同。

第一条是明确适应证，第二条是副作用或临床观察，第三条是机制解释。如果没有本体论约束，系统可能会粗暴推导出“二甲双胍可以用于减肥”。这显然是不可靠的。

所以，本体论在工程上可以理解为领域 Schema。它至少要回答这些问题：

- 有哪些实体类型
- 哪些实体之间允许建立关系
- 每种关系代表什么语义
- 关系的证据等级是什么
- 哪些关系可以传递
- 哪些结论不能直接推导
- 什么情况下必须人工复核

医疗、法律、金融这类高风险行业尤其需要这层约束。否则大模型自动抽取出来的三元组，可能只是把幻觉从“回答阶段”提前到了“知识构建阶段”。

## 4. 文章的观点可落地吗

我的判断是：方向可落地，但不能按口号落地。

可落地的部分包括：

- 不要只靠向量库
- 关键知识要结构化
- 高风险答案要有证据链
- 复杂关系要用图谱或规则表达
- 领域知识必须有 Schema 约束
- 低置信度问题要转人工复核

这些都很现实，也符合生产系统的经验。

但难落地的部分也很明显。

真正难的不是选 Neo4j、NebulaGraph，还是某个 GraphRAG 框架。真正难的是知识工程：

- 数据清洗
- 实体标准化
- 关系定义
- 冲突消解
- 版本管理
- 专家审核
- 持续更新

这些工作都很重，而且很难完全自动化。

很多团队容易低估这一点，以为“让大模型自动抽取三元组，再存进图数据库”，就完成了知识图谱。实际上，这只是开始。如果没有领域专家校验，没有证据等级，没有更新机制，没有推理边界，图谱越大，错误传播的风险也越大。

所以，知识图谱不是一个买来就能解决幻觉的组件，而是一套长期的数据治理和知识建模工程。

## 5. 更现实的落地路径

如果从企业实际项目出发，我不建议一开始就建设“大而全知识图谱”。更现实的做法，是分阶段推进。

第一阶段，先把 RAG 基础做好。

很多系统效果差，不是因为没有图谱，而是基础 RAG 做得太粗糙。应该先处理好：

- 文档清洗
- 合理切分
- 保留标题层级
- 保留表格结构
- 增加元数据
- 关键词和向量混合检索
- 重排模型
- 答案引用原文

这一步成本相对低，收益通常很明显。

第二阶段，结构化关键规则。

不要把所有知识都图谱化。优先处理高频、高风险、强规则的部分。

比如：

- 退款例外条件
- 药品禁忌
- 合同限制条款
- 审批条件
- 产品适用范围
- 风控黑名单规则

这些内容如果只靠向量召回，很容易漏条件；一旦结构化，就可以直接参与判断。

第三阶段，建设轻量知识图谱。

围绕核心业务实体建立关系，而不是追求完整世界模型。

例如企业知识库可以从这些实体开始：

- 产品
- 客户类型
- 政策
- 限制条件
- 流程节点
- 责任部门
- 风险等级

对应关系可以是：

```text
产品 -> 适用于 -> 客户类型
政策 -> 约束 -> 流程节点
风险等级 -> 需要 -> 人工审核
条款 -> 例外条件 -> 特定场景
```

第四阶段，引入领域 Schema 或轻量本体。

不一定要做成复杂的学术 Ontology，但至少要定义清楚实体类型、关系类型、证据等级、推理边界和人工复核条件。

第五阶段，让 LLM 做语言理解和编排。

大模型适合做自然语言理解、意图识别、候选实体抽取、查询规划、结果整合和答案表达。但它不应该独自决定高风险结论。

更合理的流程是：

```text
用户问题
-> 意图识别
-> 查询路由
-> 关键词 / 向量 / 图谱 / 规则混合检索
-> 证据聚合
-> 置信度校验
-> LLM 生成答案
-> 输出证据链
-> 低置信或高风险问题转人工
```

这比单纯“向量库 + Prompt”可靠，也比一上来建设完整知识图谱更务实。

## 6. 文章有没有给出实际验证

严格来说，文章没有给出严谨的效果验证。

它主要提供的是案例推演和架构判断，而不是可复现实验。比如它没有给出：

- 测试数据集
- 准确率对比
- 幻觉率下降比例
- 召回率变化
- 专家评分
- 线上 A/B 测试
- 系统延迟和维护成本

文章中对“大模型”“大模型 + RAG”“大模型 + 图谱 + RAG”的对比，更像是概念性演示，不是严格实验报告。

如果要真正验证这个方向，至少应该设计四组对照：

```text
纯 LLM
LLM + 向量 RAG
LLM + 混合检索
LLM + 图谱 + RAG
```

然后评估：

- 答案准确率
- 证据可追溯率
- 幻觉率
- 例外条款召回率
- 多跳推理正确率
- 专家评分
- 响应延迟
- 维护成本

只有这样的评估，才能证明图谱方案到底有没有带来真实收益，以及收益是否值得它的建设成本。

## 7. 我的结论

这篇文章最大的价值，是提醒大家不要把 RAG 简化成“文档切块 + 向量库 + 大模型”。

这个提醒是必要的。很多企业知识库效果不好，根本原因不是模型不够强，而是知识组织方式太粗糙。复杂业务不是靠相似度就能解决的，必须把规则、关系、约束和证据链纳入系统设计。

但我不认为向量库是伪命题，也不认为知识图谱就是唯一答案。

更准确的判断应该是：

```text
向量库适合召回相似内容。
关键词检索适合精确匹配。
规则系统适合确定性判断。
知识图谱适合关系推理。
本体论适合定义领域边界。
LLM 适合理解语言和生成表达。
```

真正可落地的 AI 知识系统，不是用某个技术替代另一个技术，而是把它们组合成一个可靠的工程体系。

如果要做企业级落地，我会优先选择：

```text
混合检索 + 关键规则结构化 + 答案引用溯源 + 小范围核心图谱 + 领域 Schema + 人工复核机制
```

这比喊“知识图谱是答案”更现实，也更容易产生业务价值。
