# Vibe Coding 不能只靠感觉：如何约束 AI 输出

> Vibe Coding 的问题从来不是“AI 会不会写代码”，而是“AI 写出来的代码，能不能长期被人维护”。想让 AI 稳定产出可用代码，关键不是写更长的 prompt，而是给它一套可执行、可验证、可迭代的约束系统。

## 一、为什么 Vibe Coding 容易失控

Vibe Coding 最迷人的地方，是你可以用自然语言快速把想法变成代码。

过去要花半天搭的页面，现在几句话就能生成；过去要翻文档、查 API、写样板代码，现在 AI 可以直接补齐。开发者从“逐行编码”变成了“描述意图、查看结果、继续调整”。

但它的问题也在这里。

当你只是凭感觉不断让 AI 改：

- “帮我优化一下”
- “再美观点”
- “顺手把这个功能也加上”
- “这里报错了，修一下”

代码很快就会变成一团看似能跑、但没人敢碰的东西。

变量命名不统一，组件边界混乱，异常处理时有时无，重复逻辑越堆越多，测试缺失，样式散落各处。短期看，功能好像做出来了；长期看，项目开始进入“神秘屋模式”：你知道门口在哪，但不知道里面发生了什么。

所以 Vibe Coding 真正需要补的，不是灵感，而是约束。

## 二、先把“什么是好代码”写下来

很多人用 AI 写代码时，只告诉它“要做什么”，很少告诉它“做到什么程度算合格”。

这会导致 AI 默认采用一种“尽快满足当前请求”的策略：能跑就行，能显示就行，能通过当前报错就行。它不会天然知道你的项目偏好，也不会自动遵守团队里那些没有写下来的隐性规则。

因此第一步，是把代码规范显性化。

可以在项目根目录放一个 `AGENTS.md`、`CLAUDE.md`、`CONTRIBUTING.md` 或类似文件，写清楚：

```md
# Coding Rules

- 使用 TypeScript，禁止使用 any，除非有明确注释说明原因
- React 组件保持单一职责，超过 200 行必须拆分
- 所有 API 调用必须经过 services 层，不允许在组件里直接 fetch
- 错误必须显式处理，不允许静默吞掉异常
- 新增功能必须补充测试，至少覆盖核心路径和异常路径
- 提交前必须通过 npm run lint、npm run test、npm run build
```

这类文件不是写给人看的形式主义，而是写给 AI 的“项目宪法”。

你每次让 AI 改代码时，都应该让它先读取这些规则，再开始动手。否则 AI 只能根据通用经验发挥，而通用经验往往和具体项目不完全匹配。

## 三、把约束分成三层：风格、架构、行为

代码规范不只是格式化。真正有效的约束，至少要分三层。

### 第一层：风格约束

这是最基础的一层，包括缩进、命名、文件组织、注释风格、import 顺序、格式化规则等。

这些规则不要靠口头提醒，应该交给工具：

- Prettier 负责格式
- ESLint / Ruff / Checkstyle 负责静态检查
- EditorConfig 统一编辑器行为
- TypeScript / mypy 负责类型约束

风格约束的原则是：**能让工具管的，就不要让 AI 自觉。**

因为 AI 的“自觉”不稳定。今天它按你的风格写，明天换一个上下文，可能又回到默认风格。工具不会忘，CI 也不会心软。

### 第二层：架构约束

架构约束决定代码放在哪里、模块之间怎么依赖、什么逻辑不能跨层调用。

例如：

- UI 层只负责展示，不直接访问数据库
- 业务逻辑放在 domain / service 层
- 基础设施代码放在 infra 层
- 前端组件不能直接拼 SQL 或调用私有接口
- 后端 handler 只做参数校验和响应封装，不堆业务逻辑

这类规则一定要写得具体，最好带反例。

不要只写“保持代码清晰”，而要写：

```md
不要在 React 组件里直接写复杂业务逻辑。
如果需要处理订单状态流转，应放到 `src/domain/order/` 下，组件只接收处理后的 view model。
```

AI 对抽象形容词很容易误解，但对具体路径、具体边界、具体禁止事项理解得更稳定。

### 第三层：行为约束

行为约束关注 AI 修改代码时的工作方式。

例如：

- 修改前先说明计划，不要直接大面积重写
- 优先做最小变更，不要顺手重构无关文件
- 不确定时先搜索现有实现，不要重新造一套
- 修 bug 必须先解释根因，再给修复方案
- 不允许为了通过测试而删除测试或降低断言强度
- 不允许引入新依赖，除非说明必要性和替代方案

这一层非常重要，因为 Vibe Coding 失控往往不是 AI 单行代码写错，而是它“太热心”。

你让它修一个按钮样式，它顺手重构了整个组件；你让它改一个接口字段，它顺手引入一个新库；你让它修测试，它直接把失败用例删了。

所以要明确告诉 AI：什么可以做，什么不能做。

## 四、不要只写 prompt，要建立检查点

很多人把约束理解成 prompt 工程：只要开头写一段“你是资深工程师，请写高质量代码”，就以为问题解决了。

这不够。

Prompt 是软约束，检查点才是硬约束。

一个更可靠的 Vibe Coding 流程应该是：

1. AI 先读项目规范和相关文件
2. AI 给出实现计划
3. 人确认范围
4. AI 做最小变更
5. 自动运行格式化、lint、测试、构建
6. AI 根据错误继续修复
7. 人看 diff，而不是只看最终效果

其中第 5 步是关键。

如果没有自动检查，AI 输出质量只能靠肉眼判断；而肉眼最容易被“功能好像能跑”欺骗。真正能挡住问题的，是类型系统、测试、lint、构建和代码审查。

可以把常用检查命令写进规范文件里：

```md
# Verification

每次修改后必须运行：

- npm run lint
- npm run test
- npm run build

如果命令失败，先修复失败原因；如果无法修复，必须说明原因和影响范围。
```

这样 AI 不只是生成代码，还必须对代码负责。

## 五、用模板限制 AI 的发挥空间

AI 最适合在清晰边界里发挥，不适合在完全开放的空间里自由发挥。

如果你希望它稳定产出某类代码，最好提供模板。

比如新建接口时，要求它按固定结构输出：

```ts
export async function getUserProfile(userId: string): Promise<UserProfile> {
  if (!userId) {
    throw new Error('userId is required');
  }

  const response = await userClient.get(`/users/${userId}`);
  return userProfileSchema.parse(response.data);
}
```

同时规定：

- 入参必须校验
- 返回值必须有类型
- 外部响应必须经过 schema parse
- 错误不能静默吞掉
- API client 必须复用已有实例

这样 AI 就不会每次都发明一种新写法。

模板的本质，是把“代码风格”变成“生成轨道”。轨道越清晰，AI 越不容易跑偏。

## 六、给 AI 一个“禁止清单”

规范里除了应该做什么，还要写不能做什么。

建议每个项目都准备一份 AI 禁止清单：

```md
# AI 禁止事项

- 禁止删除测试来让 CI 通过
- 禁止把类型改成 any 来绕过类型错误
- 禁止吞掉异常后只打印 console.log
- 禁止大面积格式化无关文件
- 禁止未经确认引入新依赖
- 禁止修改公开 API 的行为而不更新文档和测试
- 禁止用 mock 掩盖真实集成问题
- 禁止在不了解上下文时重写已有模块
```

这份清单看起来很基础，但非常有用。

因为 AI 经常会选择“最容易让当前任务成功”的路径，而不是“对项目长期最健康”的路径。禁止清单的作用，就是提前堵住那些短期有效、长期有害的捷径。

## 七、让 AI 先解释 diff，再接受代码

Vibe Coding 很容易变成一种“看效果开发”：页面出来了，测试绿了，就合并。

但 AI 生成代码时，真正应该看的不是它说了什么，而是它改了什么。

每次改完后，可以要求 AI 用固定格式解释 diff：

```md
## 变更说明

- 修改了哪些文件
- 每个文件为什么要改
- 有没有改动公共接口
- 有没有引入新依赖
- 有没有影响已有行为
- 运行了哪些检查命令
- 还有哪些风险或未覆盖场景
```

这一步会迫使 AI 回到工程视角，而不是只停留在“任务完成”的叙事里。

如果它说不清为什么改某个文件，那个改动大概率就值得警惕。如果它无法说明风险，也不代表没有风险，只代表它没有认真审视。

## 八、把“人审 AI”变成流程，而不是临时感觉

AI 写代码之后，人类的角色不是退场，而是从“亲手写每一行”转向“设计边界、制定标准、审核结果”。

人审 AI 时，至少看五件事：

1. **范围是否收敛**：有没有改无关文件？
2. **抽象是否过度**：有没有为了一个小需求设计一套大框架？
3. **错误是否真实处理**：是解决问题，还是把错误藏起来？
4. **测试是否有效**：测试是在验证行为，还是只是在迎合实现？
5. **可维护性是否变好**：三个月后别人还能不能理解？

这和传统 code review 没有本质区别，只是 AI 把产出速度提高了，所以 review 更不能省。

AI 越快，人越要慢一点。

## 九、一个可直接复制的约束模板

如果不知道从哪里开始，可以先在项目里放一个最小版 `AGENTS.md`：

```md
# Project Rules for AI Agents

## Coding Style
- Follow existing code style before introducing new patterns.
- Keep changes minimal and focused on the requested task.
- Do not reformat unrelated files.

## Architecture
- Reuse existing utilities, services, components and conventions.
- Do not create a new abstraction unless at least two real call sites need it.
- Keep UI, business logic and data access separated.

## Safety
- Do not delete or weaken tests to make them pass.
- Do not use any / unknown / type ignore to bypass type errors unless explicitly justified.
- Do not introduce new dependencies without explaining why existing code cannot solve it.

## Workflow
- Before editing, inspect related files and summarize the plan.
- After editing, explain every changed file.
- Run lint, tests and build when available.
- If verification fails, report the exact failure and fix it when possible.
```

这份模板不复杂，但足够把 AI 从“随手生成”拉回“工程协作”。

后续可以根据项目继续补充：目录结构、命名规则、测试要求、性能红线、安全红线、接口兼容要求、日志规范、错误码规范等。

## 十、结语

Vibe Coding 不是不需要规范，恰恰相反，它比传统编码更需要规范。

传统开发里，很多规则存在于工程师的经验和肌肉记忆里；但 AI 没有你的项目记忆，也不会天然理解你的团队偏好。如果不把规则写下来，不把检查自动化，不把审查流程固定下来，AI 就会按照自己的默认模式一路发挥。

好的 Vibe Coding，不是放任 AI 自由生成，而是给 AI 一条清晰的跑道：

- 用规范告诉它什么是好代码
- 用架构边界限制它不要乱改
- 用测试和 lint 检查它是否合格
- 用禁止清单堵住危险捷径
- 用 diff review 保留人的判断

AI 可以提高编码速度，但代码质量仍然来自约束。

没有约束的 Vibe Coding，只是更快地制造技术债；有约束的 Vibe Coding，才可能变成真正可持续的软件开发方式。
