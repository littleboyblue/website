# Markdown 没死，HTML 也不是新王

![Markdown 没死，HTML 也不是新王](images/post-025/cover.png)

> 当 AI 开始从“回答问题”走向“交付成果”，Markdown 和 HTML 的关系正在被重新定义。前者更像 AI 的后台工作语言，后者更像面向人的前台界面。真正重要的不是谁取代谁，而是 AI 内容生产开始分层了。

## 一、这场争论为什么突然重要了

最近，Claude Code 工程师 Thariq 提出一个很有传播力的观点：他已经越来越少让 AI 输出 Markdown，而是更倾向于让 AI 直接生成 HTML。

这个说法之所以能引起讨论，是因为它击中了很多 AI 用户的真实体验。

过去我们让 AI 写东西，默认得到的是一段 Markdown：标题、列表、加粗、小结。它很清楚，也很方便复制。但当 AI 的任务从“解释一个概念”变成“交付一个完整成果”时，Markdown 的边界就明显了。

比如你让 AI 做一份深度研究报告、一套产品方案、一个代码审查总结，或者一个可交互原型。纯 Markdown 可以表达结构，却很难表达布局、视觉、导航和交互。它能把内容写出来，但未必能把内容“交付”出来。

HTML 正是在这个位置重新被看见。

它不只是另一种文本格式，而是浏览器时代的原生交付容器。它可以有 CSS、有图片、有链接、有响应式布局、有 SVG、有 Canvas，也可以加入按钮、滑块、动效和交互逻辑。一个 HTML 文件，打开就是页面；部署到线上，发出去就是链接。

所以这场争论表面上是 Markdown 和 HTML 的格式之争，本质上是 AI 从聊天工具走向生产工具之后，输出形态必须升级。

## 二、为什么 Markdown 会成为 AI 时代的默认语言

在判断 HTML 会不会取代 Markdown 之前，先要理解 Markdown 为什么会在 AI 时代这么流行。

Markdown 的优势不是“漂亮”，而是“刚刚好”。

它诞生于 2004 年，目标是让人用接近纯文本的方式写作，同时可以渲染成结构化页面。一个 `#` 就是标题，一个 `-` 就是列表，两个 `**` 就能加粗。它不像 Word 那样复杂，也不像 HTML 那样标签密集。

这套设计对大语言模型非常友好。

模型处理的不是文件本身，而是 token。PDF、Word、PPT 进入模型之前，都要先被解析成文本，再切成 token。解析过程越复杂，越容易引入噪声。

PDF 的目标是“打印出来好看”，不是“机器读起来顺”。它内部常常记录字符坐标，而不是文章逻辑。两栏排版、复杂表格、扫描件 OCR，都会让文本顺序和结构变得混乱。

Word 和 PPT 也不轻松。`.docx` 和 `.pptx` 本质上是压缩包，里面有大量 XML、样式、主题、修订记录和版式信息。对人类协作来说，这些信息很重要；对模型理解内容来说，其中相当一部分只是 token 噪声。

TXT 足够干净，但又太干净。它没有标题、列表、代码块、引用这些结构信号。长文档一旦变成纯文本，模型就要靠语义自己猜层级。

Markdown 正好处在中间：它是纯文本，所以噪声少；它又带有轻量结构，所以模型知道哪里是标题、哪里是列表、哪里是代码、哪里是引用。

这就是 Markdown 成为 AI 默认语言的核心原因：

- 它省 token；
- 它容易生成；
- 它容易解析；
- 它足够结构化；
- 它可以被人直接阅读；
- 它也可以被程序继续处理。

从这个角度看，Markdown 不是偶然流行，而是非常适合充当 AI 的工作格式。

## 三、HTML 的优势不在“理解”，而在“展示”

HTML 的强项和 Markdown 不一样。

如果说 Markdown 适合让 AI 工作，那么 HTML 更适合让人消费结果。

一份 Markdown 报告，结构可以很清楚，但阅读体验仍然取决于你在哪里打开它。不同编辑器、不同平台、不同渲染器，对它的展示都不完全一样。更重要的是，Markdown 很难自然承载复杂交互。

HTML 则不同。它天然属于浏览器。浏览器本身就是世界上最成熟、最普及的内容展示环境。

当 AI 输出 HTML 时，它可以直接把内容组织成一个可阅读、可导航、可分享的页面：

- 长报告可以有目录、锚点和折叠区；
- 数据分析可以有图表、筛选器和可视化组件；
- 方案对比可以做成卡片、表格和标签页；
- 产品原型可以加按钮、动画和状态切换；
- 代码审查可以用颜色、分区和跳转链接帮助阅读。

这类成果如果只用 Markdown 表达，会显得像草稿；用 HTML 表达，则更像成品。

这也是 Thariq 那个观点真正有价值的地方：当 AI 的输出对象是“人”，而且这个输出需要被认真阅读、展示、传播或交互时，HTML 的体验明显更好。

但这并不意味着 HTML 就应该成为 AI 的全部工作语言。

## 四、不要把“最终交付格式”和“工作记忆格式”混为一谈

很多关于 Markdown 和 HTML 的争论，容易混淆两个问题：AI 内部怎么工作，以及用户最终看到什么。

这是两个层级。

AI 在内部拆任务、记忆上下文、交换信息、写计划、存规则时，需要的是低成本、稳定、可读、可编辑的格式。Markdown 在这里非常合适。

比如一个 Agent 项目里的 `AGENTS.md`、`CLAUDE.md`、`MEMORY.md`、`SKILL.md`，本质上都是给 AI 读的工作文件。它们不需要动画，不需要布局，也不需要花哨样式。它们需要的是清晰、轻量、可长期维护。

但当 AI 要把最终结果交给人时，情况就不同了。

用户未必想看一堆 Markdown 源码，也未必愿意打开编辑器预览。他可能只想点开一个链接，看到一份排版良好的报告，或者直接操作一个页面。

所以更合理的分工应该是：

**Markdown 是后台协议，HTML 是前台界面。**

后台需要 Markdown，因为它适合 AI 读写；前台需要 HTML，因为它适合人类阅读和交互。

把 HTML 说成 Markdown 的替代者，其实有点像说网页会取代数据库。用户确实看网页，但系统仍然需要数据库。界面和底层协议不是同一类东西。

## 五、AI 输出正在从“答案”变成“产品”

过去我们使用 AI，主要是在聊天窗口里问答。

这个阶段，Markdown 就够了。它可以让回答有标题、有重点、有列表，比纯文本更清楚，也比复杂网页更快速。

但现在 AI 的角色变了。它越来越像研究员、产品经理、程序员、设计师、运营助理，甚至是一个可以独立执行任务的 Agent。

用户对它的期待也变了。

不只是“告诉我结论”，而是：

- 给我一份可以发给老板的报告；
- 给我一个能演示的网页；
- 给我一个可以点击的原型；
- 给我一份可视化的竞品分析；
- 给我一套能直接落地的执行方案；
- 给我一个能让团队继续协作的工作空间。

当任务变成这样，AI 的输出就不能只是“回答”，而要接近“产品”。

这时 HTML 的价值会越来越大。

它让 AI 的结果从一段文本，变成一个可访问、可展示、可交互、可分享的对象。某种意义上，HTML 是 AI 交付物产品化的第一站。

但也正因为如此，HTML 适合的是“最终交付”，而不是每一次中间思考。

如果 AI 每次回答一个小问题，都要生成一个精美网页，那就像朋友回你一句话之前先化妆一小时。体验不一定更好，反而可能变慢、变重、变烦。

## 六、未来不是单一格式胜出，而是格式自动分层

真正成熟的 AI 产品，不会让用户在 Markdown 和 HTML 之间二选一。

它应该根据任务自动选择格式。

如果用户要快速解释、总结、改写，Markdown 就很好。

如果用户要结构化数据，JSON 更合适。

如果用户要长期维护的项目规则，Markdown 仍然合适。

如果用户要一份漂亮的研究报告、一个交互原型、一个可视化页面，HTML 更合适。

如果用户要演示给别人看，可能最终应该生成 PPT、网页或视频。

也就是说，未来的关键不是“Markdown 还是 HTML”，而是 AI 能否理解不同阶段需要不同格式：

- 思考阶段：轻量、低成本、可编辑；
- 协作阶段：结构清晰、便于版本管理；
- 交付阶段：美观、可读、可分享、可交互；
- 归档阶段：稳定、可检索、可迁移。

Markdown 很适合前两层，HTML 很适合第三层。它们不是竞争关系，而是流水线关系。

## 七、Markdown 最后的胜利，可能是变得不可见

一个技术格式真正成功的标志，往往不是被所有人挂在嘴边，而是变成基础设施。

普通用户不关心 HTTP，但每天都在用网页。很多人不关心 JSON，但无数应用都靠它交换数据。真正活下来的协议，常常会从用户视野里消失。

Markdown 未来也可能如此。

普通用户看到的，可能是一份 HTML 报告、一个交互式页面、一个 AI 生成的仪表盘，甚至是某种 3D 空间或视频化结果。但在这些前台界面背后，AI 仍然可能先用 Markdown 组织内容、拆解任务、存储规则、传递上下文。

它不一定总是站在前台，却会继续留在后台。

所以，Markdown 没死，HTML 也不是新王。

这场讨论真正提醒我们的，是 AI 时代的内容生产正在出现新的分层：

**Markdown 负责让 AI 想清楚，HTML 负责让人看明白。**

未来的优秀 AI 产品，不会迷信某一种格式，而会在后台保持简单，在前台提升体验。

真正需要被取代的，不是 Markdown，而是那种把所有输出都当成一段聊天文本的旧思路。
