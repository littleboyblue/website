# Oh My OpenAgent 教程：把 OpenCode 变成一支 AI 开发小队

![Oh My OpenAgent 教程：把 OpenCode 变成一支 AI 开发小队](images/post-026/cover.png)

> Oh My OpenAgent 不是另一个“更会聊天”的 AI 编程工具，而是一个基于 OpenCode 的 Agent 编排插件。它的核心价值，是把单个模型的代码能力，变成多模型、多角色、可规划、可验证的开发流程。真正上手时，不要先纠结所有概念，先装好、跑通 `ultrawork`，再逐步理解 Prometheus、Atlas、Sisyphus 和 Team Mode。

## 一、先说清楚：Oh My OpenAgent 是什么

Oh My OpenAgent，简称 OMO，官方定位是一个面向 OpenCode 的 agent harness，也可以理解为“Agent 编排套件”。

它不是一个独立 IDE，也不是简单的 prompt 包。更准确地说，它是在 OpenCode 之上增加了一套多 Agent 工作流：

- 用不同 Agent 承担规划、执行、检索、架构评审、前端视觉、文档查询等角色；
- 根据任务类型，把不同模型放到更适合的位置；
- 通过 `ultrawork`、`@plan`、`/start-work` 等入口，把复杂开发任务拆成可执行流程；
- 支持后台 Agent、生命周期 hook、模型 fallback，以及可选的 Team Mode。

如果把普通 AI 编程工具看成“一个很强的程序员坐在你旁边”，那 OMO 更像是“给 OpenCode 配了一套小型工程团队”。

它的价值不在于每次都生成更长的代码，而在于：当任务变复杂时，它能把“我让一个模型硬做到底”变成“让合适的 Agent 做合适的事”。

## 二、适合谁用，不适合谁用

OMO 更适合下面几类人：

1. 已经在用 OpenCode，想把它升级成更系统的 AI 开发工作流；
2. 经常处理多文件、多模块、多步骤任务，而不是只改一两行代码；
3. 愿意为不同模型配置账号或订阅，希望降低单一供应商锁定；
4. 希望 AI 不只是回答问题，而是能探索代码库、写计划、执行任务、验证结果。

它不太适合下面几种情况：

- 你只想偶尔问 AI 一个语法问题；
- 你不想安装或配置任何命令行工具；
- 你完全不愿意看 diff、跑测试、处理认证；
- 你的任务都是非常短的小改动，用普通 OpenCode 或 Claude Code 已经足够。

一句话：**OMO 是给“想认真用 Agent 做工程”的人准备的，不是给“只想点一下生成代码”的人准备的。**

## 三、安装前需要准备什么

上手前，至少要有三样东西：

1. **OpenCode**
2. **Bun**
3. **至少一个可用模型来源**

先检查 OpenCode：

```bash
opencode --version
```

如果还没安装 OpenCode，先去 OpenCode 官方文档完成安装。OMO 是 OpenCode 插件，不是替代 OpenCode 的独立工具。

再检查 Bun：

```bash
bun --version
```

官方安装文档里明确建议：安装 OMO 时使用 Bun，不要用 npm、yarn 或 pnpm。安装后，CLI 会使用对应平台的独立二进制，日常运行不需要你手动维护 Node/Bun 运行时。

还要想清楚你有哪些模型或订阅。安装器会根据你的回答配置不同 Agent 的模型来源，例如：

- Claude Pro / Max
- ChatGPT Plus / Pro 或 OpenAI 访问
- Gemini
- GitHub Copilot
- OpenCode Go
- OpenCode Zen
- Z.ai Coding Plan
- Kimi for Coding
- Vercel AI Gateway

不需要一开始全部都有，但要知道：OMO 的很多能力来自“多模型分工”。如果你只有一个模型，也能用，但体验会打折。

## 四、最推荐的安装方式：让 Agent 帮你装

OMO 的官方文档其实很直接：推荐你把安装说明交给一个 LLM Agent，让它帮你完成配置。

可以把下面这段粘贴到 Claude Code、OpenCode、Cursor 或其他你正在用的 Agent 里：

```text
Install and configure oh-my-openagent by following the instructions here:
https://raw.githubusercontent.com/code-yeongyu/oh-my-openagent/refs/heads/dev/docs/guide/installation.md
```

为什么官方更推荐这种方式？

因为安装过程不只是“下载一个包”，还涉及：

- 检查 OpenCode 是否存在；
- 判断你有哪些订阅；
- 生成对应安装参数；
- 写入 OpenCode 插件配置；
- 引导你做 provider auth；
- 最后运行 doctor 检查。

这些步骤让 Agent 来做，比人手动复制配置更不容易漏项。

如果你想自己安装，也可以运行：

```bash
bunx oh-my-openagent install
```

如果是在非交互环境里，可以使用 `--no-tui` 加参数，例如：

```bash
bunx oh-my-openagent install \
  --no-tui \
  --claude=yes \
  --openai=yes \
  --gemini=no \
  --copilot=no
```

如果你没有 Claude，但有 OpenCode Go，可以类似这样：

```bash
bunx oh-my-openagent install \
  --no-tui \
  --claude=no \
  --openai=no \
  --gemini=no \
  --copilot=no \
  --opencode-go=yes
```

这里不要机械照抄，关键是按自己的订阅情况设置参数。

## 五、关于名字：OpenAgent 和 OpenCode 的过渡期

安装时你可能会看到两个名字：

- `oh-my-openagent`
- `oh-my-opencode`

这是项目重命名过渡造成的。当前项目已经叫 Oh My OpenAgent，但历史上使用过 Oh My OpenCode。npm 包和命令也处在兼容阶段。

一般来说，新配置里优先使用：

```json
{
  "plugin": ["oh-my-openagent"]
}
```

但旧的 `oh-my-opencode` 配置仍可能被识别，只是可能出现 legacy warning。

如果运行 doctor 时看到“Using legacy package name”之类提示，可以把 OpenCode 配置里的插件名从：

```json
"oh-my-opencode"
```

改成：

```json
"oh-my-openagent"
```

配置文件名也有兼容关系。你可能会看到：

- `oh-my-openagent.jsonc`
- `oh-my-opencode.jsonc`

新项目建议使用前者；已有项目如果还在用后者，先不要慌，确认 doctor 通过之后再逐步迁移。

## 六、安装后第一件事：跑 doctor

安装完成后，不要急着让它写代码，先验证：

```bash
bunx oh-my-openagent doctor
```

这个命令会检查系统、配置、工具和模型解析情况。它比“能不能打开 OpenCode”更重要，因为 OMO 的核心是多 Agent、多模型协作。如果某个 Agent 没有解析到可用模型，后面复杂任务就可能退化成普通聊天。

同时检查 OpenCode 配置：

```bash
cat ~/.config/opencode/opencode.json
```

你应该能看到插件数组里包含 `oh-my-openagent`，或者在迁移期看到兼容的旧名称。

还可以检查可用模型：

```bash
opencode models
```

检查认证状态：

```bash
opencode auth list
```

如果某个 provider 没登录，就用：

```bash
opencode auth login
```

然后按提示选择 Anthropic、OpenAI、Google、GitHub Copilot、OpenCode 等 provider。

## 七、第一个任务：直接输入 ultrawork

OMO 的入门体验非常简单：打开 OpenCode，在项目里输入：

```text
ultrawork
```

或者简写：

```text
ulw
```

它的含义不是“执行一个固定脚本”，而是让 OMO 自动判断当前项目和任务上下文，进入更完整的探索、计划、执行、验证流程。

适合 `ultrawork` 的场景包括：

- “帮我实现这个功能”；
- “这个项目有一堆报错，帮我修”；
- “把这块代码重构一下”；
- “给这个模块补测试”；
- “我懒得解释太多，你先自己读代码理解一下”。

它不适合特别简单的任务。比如改一个按钮文案、修一个错别字、问一个 API 用法，直接正常 prompt 就够了。

可以按这个经验判断：

- 简单任务：直接问；
- 中等任务：让 OMO 先探索再做；
- 复杂任务：进入规划流程。

## 八、复杂任务：用 @plan 和 /start-work

当任务比较大、你又不想让 AI 一边想一边乱改时，推荐使用规划流程。

大致是两步：

```text
@plan
```

先让 Prometheus 进入规划状态。它会像顾问一样追问目标、边界、验收标准和技术约束，并可以调用探索类 Agent 了解代码库。

计划完成后，再执行：

```text
/start-work
```

这时 Atlas 会读取计划，拆分任务，调度不同 worker 执行，并在过程中积累经验和验证结果。

这个流程的好处是：

- 先想清楚，再动代码；
- 人可以在计划阶段控制范围；
- 执行阶段有更明确的任务边界；
- 更适合多文件、多模块、风险较高的修改。

如果你做的是重构、迁移、架构调整、测试体系补齐，尽量不要一上来就说“帮我全改了”。先让它计划，再让它执行。

## 九、理解几个核心 Agent

OMO 里有很多 Agent，入门不需要全部背下来，但要理解几个核心角色。

### Sisyphus：偏协调的主力 Agent

Sisyphus 更像一个会沟通、会协调、会推进的 lead developer。它适合处理复杂上下文、拆解任务、组织执行。

### Prometheus：规划者

Prometheus 负责把模糊需求变成计划。它的重点不是直接写代码，而是问清楚目标、约束和验收标准。

### Atlas：执行指挥者

Atlas 像执行阶段的 conductor。它读取计划，安排任务，检查结果，但很多具体代码修改会交给 worker 完成。

### Hephaestus：深度技术执行者

Hephaestus 更像沉浸式解决复杂技术问题的 specialist。它适合深度编码、复杂推理和长时间自主探索。

### Oracle、Librarian、Explore

这几个更像辅助专家：

- Oracle 偏架构和深度判断；
- Librarian 偏文档、开源资料和外部知识；
- Explore 偏代码库探索和模式检索。

这些名字看起来有点中二，但背后的设计很实际：**不要让一个 Agent 同时承担所有认知负担。**

## 十、模型配置的关键：不是越贵越好，而是角色匹配

OMO 的一个重要思想是：不同模型适合不同工作方式。

官方文档里把模型类比成团队成员。Claude、GPT、Kimi、GLM、Gemini 这类模型，不只是“谁更聪明”的差异，而是“谁更适合哪种任务”。

例如：

- Claude / Kimi / GLM 更适合长指令、流程遵循、协调式工作；
- GPT 更适合原则驱动、深度推理、复杂技术执行；
- MiniMax、Haiku 一类模型更适合快速检索、轻量任务或辅助场景；
- Gemini / Qwen 等模型在视觉、前端、长上下文场景可能有优势。

所以不要简单地把所有 Agent 都改成一个你觉得最强的模型。

一个常见误区是：

> “我有某个最贵模型，那我把所有 Agent 都设成它。”

这未必更好。OMO 的 prompt 和 Agent 角色是按模型风格设计的。把模型乱换，可能会导致规划 Agent 太发散、执行 Agent 太啰嗦、检索 Agent 太慢，整体体验反而下降。

更稳妥的做法是：先用安装器默认推荐配置，跑 doctor 确认 fallback 正常，再针对具体问题微调。

## 十一、Team Mode：什么时候需要开

OMO 还有一个可选能力叫 Team Mode。它默认关闭。

Team Mode 的目标，是让多个 Agent 以“团队”的方式并行协作：有 lead，有成员，有任务列表，有 mailbox，有可选的 tmux 可视化。

开启方式是在用户级或项目级配置里加入：

```jsonc
{
  "team_mode": {
    "enabled": true,
    "max_parallel_members": 4,
    "max_members": 8,
    "tmux_visualization": false
  }
}
```

用户级配置通常在：

```text
~/.config/opencode/oh-my-openagent.jsonc
```

项目级配置可以放在：

```text
.opencode/oh-my-openagent.jsonc
```

开启后重启 OpenCode，相关的 `team_*` 工具才会出现。

但新手不建议一开始就开 Team Mode。原因很简单：普通 OMO 工作流已经足够复杂，Team Mode 又引入了更多并行状态、成员通信和任务协调。

更适合它的场景是：

- 大型重构需要多个方向并行探索；
- 安全研究需要多名“猎手”同时找问题；
- 代码库很大，单 Agent 探索太慢；
- 你希望用 tmux 观察多个成员的实时工作。

如果只是日常写功能、修 bug，先用 `ultrawork` 和 `@plan` 就够了。

## 十二、一个推荐的新手工作流

第一次使用 OMO，可以按这个顺序来：

### 1. 安装并验证

```bash
bunx oh-my-openagent install
bunx oh-my-openagent doctor
opencode auth list
opencode models
```

确认 doctor 没有关键错误。

### 2. 找一个真实但不致命的小任务

不要一上来就让它重构全项目。选一个范围可控的任务，例如：

- 修一个明确 bug；
- 给某个函数补测试；
- 给一个页面加小功能；
- 整理一段重复逻辑。

### 3. 让它先读项目规则

如果项目里有 `AGENTS.md`、`CLAUDE.md`、`README.md`、`CONTRIBUTING.md`，先让它读取。

例如：

```text
先阅读 AGENTS.md 和相关模块代码，再帮我完成这个任务。不要改无关文件，完成后说明验证方式。
```

### 4. 对中等任务使用 ultrawork

```text
ultrawork
```

然后描述目标和限制。

### 5. 对复杂任务使用计划流程

```text
@plan
```

计划确认后：

```text
/start-work
```

### 6. 最后一定看 diff 和测试结果

不要因为是多 Agent 系统就放弃审查。最终仍然要看：

```bash
git diff
```

以及项目对应的测试、lint、build 命令。

OMO 可以提高执行力，但不能替你承担工程责任。

## 十三、常见问题和排查

### 1. doctor 报模型不可用

先看你是否登录了对应 provider：

```bash
opencode auth list
```

再看可用模型：

```bash
opencode models
```

如果模型不存在，就不是 OMO 单方面能解决的问题，需要登录 provider、调整订阅或修改 fallback。

### 2. 插件名 warning

如果提示 legacy package name，检查：

```bash
cat ~/.config/opencode/opencode.json
```

把旧的 `oh-my-opencode` 改成 `oh-my-openagent`。

### 3. 配置改了但没生效

重启 OpenCode。很多插件配置、Team Mode 开关、工具注册都需要重启后才会加载。

### 4. Agent 做得太多

这是所有 Agent 编程工具的通病。解决方法不是“骂它”，而是明确边界：

```text
只修改 src/auth 目录下与登录流程相关的代码。不要重构无关模块。完成后列出所有修改文件和验证命令。
```

对于高风险任务，先用 `@plan`，不要直接 `ultrawork` 一把梭。

### 5. 结果看起来很强，但我不确定能不能合并

把判断标准拉回工程基本功：

- 是否通过测试；
- 是否有清晰 diff；
- 是否改了无关文件；
- 是否引入新依赖；
- 是否破坏现有接口；
- 是否有可回滚路径。

Agent 再强，也不能跳过这些检查。

## 十四、我的建议：把 OMO 当工作流，而不是魔法

Oh My OpenAgent 最容易被误解的地方，是它的命令看起来太简单：装好以后输入 `ultrawork`，似乎就可以让 AI 自动把活干完。

但真正稳定的用法不是“交给它然后不管”，而是把它当成一个更强的工程工作流：

- 用 Prometheus 把需求问清楚；
- 用 Atlas 组织执行；
- 用 Explore 和 Librarian 收集上下文；
- 用 Oracle 做关键判断；
- 用 doctor 和测试验证环境；
- 用 diff review 保住代码质量。

也就是说，OMO 的意义不是让人退出开发过程，而是让人从“逐行写代码”退到“定义目标、控制边界、审查结果”的位置。

这也是 AI 编程工具正在发生的变化：

过去我们关心的是单个模型能不能补全代码；现在更重要的是，它能不能进入真实工程流程。

OMO 给出的答案是：不要迷信一个模型，也不要迷信一次 prompt。把任务拆开，把角色分开，把验证补上，让 AI 像团队一样工作。

如果你已经在用 OpenCode，又经常被复杂任务、模型切换和上下文管理折磨，Oh My OpenAgent 值得一试。

但请记住：

**`ultrawork` 是开始，不是结束。真正决定结果质量的，仍然是你的目标、边界、验证和审查。**
