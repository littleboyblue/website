# 干掉 Markdown？Token 不够的时候，没法和 Karpathy 共情

![干掉 Markdown？Token 不够的时候，没法和 Karpathy 共情](images/post-028/cover.png)

> 本文基于 InfoQ 文章《干掉 .md？兜里 Token 不够，没法和 Karpathy 共情》整理，原文围绕 Claude Code 团队成员 Thariq 的《The Unreasonable Effectiveness of HTML》以及 Theo Browne 的长视频，讨论了 AI Agent 时代 HTML 是否正在取代 Markdown 的争议。

最近开发者社区有一个话题持续升温：**Markdown 是不是已经过时了？**

起因是 Claude Code 团队成员 Thariq 写了一篇文章《The Unreasonable Effectiveness of HTML》，核心观点是：在 AI Agent 时代，HTML 正在成为比 Markdown 更高效的人机协作媒介。

随后，TypeScript 圈知名技术创作者 Theo Browne（Ping Labs CEO、t3.gg 创始人）发了一条 30 多分钟的视频，系统阐述了自己为什么开始放弃 Markdown，并总结出"HTML 优于 Markdown"的五大核心理由。

Theo 坦言，自己曾是 Markdown 的坚定支持者——过去近二十年里，它一直是开发者最自然的表达方式之一。但现在，他说自己几乎不再为任何东西写 Markdown 文件，而是全面转向让 Claude Code 生成 HTML。

## 一、HTML 的五大优势

Theo 结合 Thariq 的文章和自身经验，归纳出 HTML 优于 Markdown 的五个关键点：

### 1. 更高的信息密度

一个 HTML 文件可以承载表格、设计插图、代码、图像、空间工作流和交互组件。Markdown 能装的东西，HTML 都能装，而且装得更多。

不过 Theo 也承认，模型在输出图像方面依然表现糟糕——他最近尝试让 GPT-5.5 查找特定图片，模型却因系统提示生成了一张完全无关的图像。"这些模型在识别图像内容方面有所进步，但要真正找出一张图片并放到合适位置，祝你好运。"

### 2. 视觉清晰，易于阅读

随着 Claude 处理的工作日趋复杂，它产出的规范和计划也越来越庞大。Theo 坦言，自己通常不会真正阅读超过 100 行的 Markdown 文件，也无法让团队中的任何人去读。

但 HTML 文档可以通过标签页、插图、链接等方式组织成清晰的结构，甚至支持移动端响应式布局。

### 3. 便于分享

Markdown 文件在大多数浏览器中无法良好原生渲染，通常需要以附件形式发送。而 HTML 文件只需上传到 S3 即可生成链接，同事可在任何设备上打开并引用。

### 4. 强大的交互能力

HTML 允许用户与文档进行双向交互——添加滑块、下拉菜单来调整设计或算法参数，并将更改复制回提示词中，粘贴回 Claude Code 继续操作。这是纯文本的 Markdown 做不到的。

### 5. 令人愉悦的创作体验

Theo 强调，用 HTML 制作文档"更有趣"，让他感觉更投入、更专注。"如果某样东西能让你更兴奋地去构建、去迭代，那就是好事。"

## 二、Token 成本和版本控制：HTML 的两个硬伤

尽管整体支持 HTML，Theo 并未全盘接受 Claude Code 团队的论述。他在视频中多次提出质疑。

### Token 太贵了

Claude Code 团队认为，在百万 Token 上下文时代，HTML 的冗余结构不再是问题。但 Theo 认为这种代价依然值得认真权衡。

他的评论区经常有人阴阳怪气："Thariq 写这篇东西，纯粹是为了让大家多烧 Token，好让 Anthropic 多赚钱。"

Theo 觉得这种想法挺蠢的——有趣的玩笑，但绝对不是事实。他也承认："确实，Markdown 通常更省 Token。但对我来说，HTML 带来的额外表现力，以及我更有可能认真读完它的事实，意味着在拥有 100 万 Token 上下文窗口和 Opus 4.7 的情况下，我其实根本感觉不到 Token 消耗的增加。不过这么说也有点傻。"

**关键矛盾在于**：HTML 的冗余结构确实天然会消耗更多上下文窗口。对上下文窗口充裕的人来说，这不是问题；但对 Token 预算有限的人，这就是真金白银的差距。

### 版本控制不太行

> "和 Markdown 相比，HTML 的 diff 非常杂乱，很难审查。"

Markdown 的 diff 十分清晰，而 HTML 因标签嵌套复杂，代码审查时可读性显著下降。

但 Theo 认为这并非完全无解：可以让 Claude 参考你的代码库，先生成一个统一的设计系统 HTML 文件，然后把它作为其他 HTML 文件的基准。这样仍然基于 HTML，但能让 Agent 产出的内容更好看、更可移植，同时拥有更好的版本控制和托管能力。

## 三、一次性代码：Agent 时代的新逻辑

Theo 提出了一个颇具启发性的观点：开发者长期被一种旧观念束缚——代码必须是可复用的。但在 Agent 时代，大量代码的价值恰恰在于"一次性"。

他透露，自己现在写的代码里，**超过 70% 只运行一次便被丢弃**。

例如：

- 为某组 CSV 数据临时生成分析 UI
- 针对特定 JSON 创建编辑器
- 快速构建可视化检查面板
- 为某个 prompt 构建临时交互界面

这些 HTML 工件并非产品，而是辅助决策的临时工具。而 AI 的存在，使这种"即写即弃"的开发方式第一次变得经济可行。

## 四、Karpathy 的判断：未来属于视觉输出

视频尾声，Theo 引用了 Karpathy 的观点：

> "在查询结尾要求模型以 HTML 输出，然后在浏览器中查看，效果非常惊人。"

Karpathy 进一步提出一个判断：**未来，人类对 AI 的输入会越来越偏向语音，而 AI 对人类的输出，则会越来越偏向视觉。**

原因很简单——人脑近三分之一的处理资源用于视觉系统，视觉天然拥有更高的信息吞吐能力。在他看来，Markdown 只是从纯文本向视觉化过渡的第一站，而 HTML 只是中间阶段。

更远的未来，AI 输出可能直接演化为：

- 实时交互式模拟
- 动态视觉界面
- 神经生成视频
- 完全无代码渲染层

换句话说，**HTML 并不是终点，而只是迈向下一代 AI 交互界面的过渡形态。**

## 五、Markdown 真要死了吗？

Theo 最终并未给出明确答案。他承认，彻底抛弃 Markdown 仍显激进——他自己可能把"HTML 最大化"这条路走得太极端了。

但他认为，这场讨论的真正意义不应该是"HTML 是否取代 Markdown"，而是开发者终于开始重新思考一个长期被忽视的问题：**在 AI 原生时代，人与模型之间究竟应该通过什么样的界面协作。**

在 Theo 看来，行业距离答案还很远。"我们甚至还没真正搞清楚 AI 的正确 UI 长什么样。"

YouTube 评论区里，更多的声音不是对"HTML 新时代"的兴奋，而是对 Token 消耗、效率成本以及这一观点实际必要性的质疑。最具代表性的调侃是：

> "Anthropic：兄弟们，把输出格式从 Markdown 换成 HTML，Token 消耗直接翻倍，收入也翻倍。"

## 写在最后

这场争论的核心，其实不是格式之争，而是**协作界面之争**。

Markdown 的优势在于轻量和可编辑——当人还需要亲手写文档的时候，这些优势是决定性的。但当文档越来越多地由 AI 生成、AI 修改，"人能不能方便地手写"就不再是核心价值了。

HTML 的优势在于表现力和交互性——它让 AI 的输出从"一段文字"变成"一个可探索的界面"。这种转变会改变人与 AI 协作的方式：从被动阅读到主动探索。

但 HTML 也有真实的代价：Token 消耗更高、版本控制更差、移动端适配不稳定。这些不是"百万上下文时代"一句话就能消解的。

Karpathy 的判断或许指出了更远的方向：Markdown 和 HTML 都不是终点，AI 与人的协作界面还会继续演化。但在今天，对大多数开发者来说，更务实的态度可能是——

**不是二选一，而是各归其位：Markdown 做后台的工作语言，HTML 做前台的交付界面。**

毕竟，兜里 Token 不够的时候，确实没法和 Karpathy 共情。
