# Vibe Coding 并入 Agent 工程：代码不值钱之后，什么才值钱

> 本文基于 InfoQ 对 Heavybit 播客《High Leverage》中 Simon Willison 访谈的整理，结合 AI 编程工具的真实使用变化，讨论 Vibe Coding 与 Agent 工程的边界为什么正在模糊，以及这对软件工程师、团队流程和软件公司价值意味着什么。

过去一年，很多开发者对 Vibe Coding 的态度很明确：可以玩，但别上生产。

所谓 Vibe Coding，就是你不太关心代码具体怎么写，也不逐行审查实现，只要 AI 生成的东西能跑，就先用起来。它很适合个人小工具、一次性脚本、内部原型。坏了也就坏了，最多坑自己。

而 Agentic Engineering，听起来就严肃得多：专业工程师借助 Coding Agent 写代码，但仍然对安全、性能、可维护性、测试和长期演进负责。AI 可以参与生产，但责任不能外包。

这个区分曾经很清楚。

但 Simon Willison 最近提出了一个更刺耳的观察：**这两条线正在重叠。**

不是因为 Vibe Coding 突然变得严谨，而是因为 Agent 写出的代码越来越可靠，可靠到连资深工程师也开始减少逐行审查。

## 一、最吓人的变化：专业工程师也开始“不看代码”

Simon Willison 过去对 Vibe Coding 的态度很强硬：如果你给别人写软件，却完全不看 AI 生成的代码，这是极其不负责任的。

这个判断现在仍然成立。

但问题在于，真实工作方式已经悄悄变了。

当你让 Claude Code 或 Codex 写一个 API 端点、补一组测试、改一个 SQL 查询、加一个页面交互时，它大概率能完成，而且完成得还不错。一次、两次、十次之后，你的信任阈值会被不断调低。

一开始你会认真看 diff。

后来你只跑测试。

再后来，你只看功能是否正常。

最后，你会把 Agent 当成一个“半黑盒”的协作者：像信任公司里另一个团队交付的模块一样，先接入、先使用，只有出问题时才去翻源码。

这就是 Simon 所说的“偏差正常化”：**AI 每次写对一次，都会让你下一次更容易盲目信任它。**

这件事真正危险的地方，不是 AI 不能被信任，而是人类会在不知不觉中把“它这次写对了”外推成“它以后也会写对”。

## 二、Vibe Coding 和 Agent 工程的差异，不再是“看不看代码”这么简单

以前我们可以用一个很简单的标准划线：

- 不看代码，只要能跑，就是 Vibe Coding；
- 看代码、写测试、做 Review，就是 Agentic Engineering。

但这个标准正在失效。

因为在真实工程里，人类本来也不会审查所有代码。

一个公司内部的缩略图服务、风控服务、数据平台、权限系统，很多调用方都不会逐行读源码。他们依赖的是文档、接口契约、SLA、测试、监控和组织信任。

如果 AI Agent 生成的模块也能提供这些东西，工程师自然会把它视为一个外部服务或内部团队交付物，而不是一段必须逐字符理解的代码。

所以，Vibe Coding 与 Agent 工程真正的分界线，不应该只是“有没有看代码”，而应该是：**有没有建立可靠的责任边界和验证机制。**

一个负责任的 Agent 工程流程，至少要回答这些问题：

1. 这段代码影响的是个人工具，还是用户数据、交易、权限、安全？
2. 它的输入输出契约是否明确？
3. 是否有测试覆盖关键路径和边界条件？
4. 是否能通过日志、监控、回滚机制发现和控制风险？
5. 人类是否清楚哪些部分必须亲自审查？

如果这些问题没有答案，即使你用了最先进的 Coding Agent，本质上也只是把 Vibe Coding 包装成了工程实践。

## 三、人类 Review 正在成为新的瓶颈

软件工程的历史，某种程度上就是不断消除瓶颈的历史。

云计算消除了等服务器、等机房、等网络配置的瓶颈。CI/CD 消除了手工部署的瓶颈。容器和 IaC 消除了环境不一致的瓶颈。

而在 AI Coding Agent 之后，新的瓶颈变成了：**人类能不能审得过来。**

过去一个开发者每天写几百行代码，团队流程、PR Review、测试节奏、发布窗口，都是围绕这个生产速度设计的。

现在一个开发者可以同时让多个 Agent 并行写代码，一天产出几千行甚至更多。原来的 SDLC 就会出现结构性失衡：代码生成速度暴涨，但需求澄清、架构判断、Review、测试、上线和运维没有同步增长。

于是团队会遇到一个尴尬问题：

**如果每一行 AI 代码都要人工 Review，效率提升会被吃掉；如果不 Review，又如何对生产质量负责？**

答案不会是极端的“全看”或“全不看”，而是分层。

安全相关代码必须看。

权限、支付、数据删除、隐私、加密、审计这些代码必须看。

核心数据模型和状态机必须看。

公共 API 和长期演进接口必须看。

但很多 UI 细节、胶水代码、一次性迁移脚本、内部原型、低风险自动化，可能会越来越接近“出了问题再看”。

这不是道德上的退步，而是工程上的再分配：**人类注意力必须从逐行审查，迁移到风险分级、边界设计和验证体系。**

## 四、真正的风险：开发者失去“代码本体感觉”

讨论 Vibe Coding 时，很多人担心 AI 会写坏代码。

但更深的风险是：开发者可能逐渐失去辨别坏代码的能力。

一个经验丰富的工程师，在长期编写和重构代码时，会形成一种近似“本体感觉”的能力。你看到一个抽象，就能感觉它会不会泄漏；看到一个依赖方向，就能感觉后面会不会拧巴；看到一个状态流转，就能感觉未来加需求会不会爆炸。

这种能力不是读几篇最佳实践得来的，而是在反复动手、踩坑、重构、上线、救火中长出来的。

如果未来大量开发者只负责给 Agent 下指令、验收结果、复制错误信息、反复提示“再改一下”，他们可能会更快做出东西，却更慢形成工程直觉。

这对新人尤其关键。

AI 可以极大降低入门挫败感。以前一个缺分号、一个环境变量、一个依赖版本冲突，都可能卡住新人几个小时。现在把错误贴给模型，基本能马上得到答案。

这是好事。

但如果学习路径只剩下“让 AI 写、让 AI 修、让 AI 解释”，新人可能会错过最重要的一段训练：亲手在代码里建立因果感。

更健康的方式，可能不是让 AI 永远当驾驶员，而是让 AI 回到“结对编程”的副驾驶位置：它查资料、解释错误、提出方案、指出风险，但键盘仍然由人来敲一部分。

只有这样，AI 才是在放大人的能力，而不是替代人的能力生长过程。

## 五、代码正在贬值，但软件并没有贬值

AI 编程工具最容易引发的误判是：既然 Claude、Codex、Qwen、DeepSeek 都能写代码，那软件公司的价值是不是会归零？

这个判断看似激进，其实混淆了两件事：

- 代码文本的生产成本在下降；
- 软件系统的长期价值并不等于代码文本。

代码确实正在商品化。

一个有 README、有测试、有提交历史的小项目，现在半小时就能生成得像模像样。过去我们看到一个仓库有上百次提交、文档完善、测试齐全，会默认作者投入了大量心血。现在这些外观信号都可能被快速伪造。

于是一个新标准变得更重要：**有没有人真正用过它。**

一个被作者每天使用两周的小工具，比一个刚生成出来、测试覆盖率 95%、README 精美的仓库更可信。企业采购也是一样：一个 CRM、工单系统、数据平台，最有说服力的不是代码有多漂亮，而是有没有真实客户稳定跑过半年。

当代码不再稀缺，真正稀缺的东西会变成：

- 对业务问题的理解；
- 稳定的数据模型；
- 清晰的 API 契约；
- 可验证的工作流；
- 可靠的权限和审计；
- 长期维护中的判断力。

换句话说，软件没有死，死的是“只靠写代码收费”的旧叙事。

## 六、未来软件的核心，可能更像数据库和接口边界

Simon 在访谈里举了一个很好的例子：如果今天从零做一个 Issue Tracker，他会把主要精力放在哪里？

不是 UI。

UI 可以用 Agent 快速生成，也可以让客户自己定制。

真正值得投入的是核心数据库 Schema：Issue、评论、标签、里程碑、权限、状态流转、通知关系。这些东西一旦设计错，后面生成再多 UI 都只是把混乱包装得更漂亮。

这也是 AI 时代很重要的架构原则：**越是上层交互，越可以让 Agent 灵活生成；越是底层数据和接口，越需要确定性。**

因为 Agent 带来的本质增量是非确定性生产力。它能快速生成不同版本、不同界面、不同实现路径。但系统里必须有一些稳定边界来吸收这种非确定性，否则灵活性会变成混乱。

这些稳定边界包括：

- 数据模型；
- API 设计；
- 权限模型；
- 事务一致性；
- 观测和审计；
- 回滚与迁移策略。

未来的软件工程师，价值可能不再是“我能比别人更快写出这 500 行代码”，而是“我能定义出一个让 50 个 Agent 同时工作也不会把系统搞乱的边界”。

## 七、并行 Agent 会改变“专注”的含义

过去的软件开发强调深度专注。一个程序员被打断后，可能需要很久才能重新加载上下文。

但 Agent 改变了这一点。

当具体执行交给模型，人类从“持续持有全部实现细节”转向“持续给出下一步方向”。这会让工作方式变得更像调度，而不是手工雕刻。

你可以让一个 Agent 探索方案 A，让另一个 Agent 做方案 B，再让第三个 Agent 查历史代码和约束。你本人则在多个结果之间做判断。

这种方式特别适合 Spike，也就是为了验证想法而快速搭建的实验性实现。

以前做一个 Spike 可能要两天。现在一小时可以跑十个。

但它也带来新的认知负担：你不再被写代码耗尽，而是被评估结果耗尽。大量方案同时出现后，真正困难的是判断哪个方向更稳、哪个抽象更干净、哪个风险被隐藏了。

所以，未来开发者的稀缺能力会越来越像技术评审和系统设计：**不是亲手把每块砖砌上去，而是判断这栋楼会不会塌。**

## 八、市场不会奖励“高质量代码”，但会惩罚失控系统

一个残酷现实是：高质量代码在市场上长期没有得到充分奖励。

AI 出现之前，很多公司就已经背着沉重技术债快速交付。业务压力、融资节奏、短期 KPI、组织惯性，都会把团队推向“先上线再说”。

AI 不会自动改善这个激励结构。

相反，它可能让低质量交付变得更快、更便宜、更大规模。

如果一个组织过去就不重视测试、不重视架构、不重视文档、不重视维护成本，那么引入 Agent 后，它得到的往往不是高质量软件，而是更高产的技术债机器。

所以，AI 编程真正考验的不是模型，而是组织治理。

团队需要明确哪些事情可以快，哪些事情不能快；哪些代码可以“先跑起来”，哪些代码必须先被设计清楚；哪些 Review 可以抽样，哪些 Review 必须强制。

没有这些规则，Vibe Coding 和 Agent 工程的融合不会带来生产力革命，只会带来债务膨胀。

## 写在最后

Vibe Coding 并入 Agent 工程，并不意味着“不负责任”变成了合理。

它真正意味着：软件开发的责任形态变了。

过去，负责任主要体现为“我写了这段代码，我理解每一行”。

未来，负责任会更多体现为：

1. 我知道哪些地方必须亲自审查；
2. 我知道哪些风险可以通过测试和监控兜住；
3. 我知道哪些边界不能交给非确定性系统随意生成；
4. 我知道如何验证一个系统真的被人用过，而不是只是看起来完整；
5. 我仍然保有判断坏代码、坏抽象、坏接口的能力。

AI 让代码变便宜了，但它没有让工程判断变便宜。

甚至恰恰相反：当代码生成速度提高十倍、百倍之后，判断力、边界感、数据建模、接口设计、安全意识和验证能力，都会变得更贵。

所以，Vibe Coding 最终能不能进入 Agent 工程，不取决于模型写得多像人，而取决于人类有没有建立足够清醒的工程边界。

代码不值钱之后，真正值钱的，是你知道哪些代码不能随便生成。
