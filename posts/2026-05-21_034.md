# 如何使用 AI 生成一份 PPT

> 做 PPT 的真正痛点不是"不会设计"，而是内容组织和排版的双重劳动。正确的 AI 方案不是"一句话出幻灯片"，而是 LLM 管内容、模板管排版、PptxGenJS 管输出——三层分离，各司其职。

做 PPT 这件事，大多数人的真实体验是这样的：

1. 打开 PowerPoint，选模板，花 20 分钟调母版；
2. 写内容，发现文字一多就溢出，开始调字号、行距、对齐；
3. 插图片，调位置，发现和文字打架，再调；
4. 同事说"这页太密了"，删内容，重排；
5. 领导说"加一页数据对比"，找图表，调数据源，再排。

两小时过去了，你花在"想清楚说什么"上的时间可能只有 15 分钟，剩下全在排版。

AI 可以把这两步拆开。但拆开的方式，比大多数人想的更工程化。

## 一、整体架构

先看全貌：

![AI 生成 PPT 的架构流程](images/ai-ppt-arch.svg)

四个阶段，两条输出路径：

1. **LLM 生成大纲 JSON**——所有 LLM 都能做，这一步没有技术壁垒；
2. **JSON + HTML 模板 → HTML 预览**——用 EJS 或 Handlebars 渲染，核心逻辑 10 行代码；
3. **JSON + PptxGenJS → PPTX 输出**——按布局参数直接调用 API，跳过 DOM 解析；
4. HTML 只做预览和排版验证，不参与最终生成。

为什么不让 HTML 直接转 PPTX？因为 HTML → PPTX 的转换（如 html2pptx、puppeteer + 转图片）需要解析 DOM，布局精度差、字体丢失、动画无法保留。PptxGenJS 原生生成，字体/动画/母版完整支持。

HTML 预览和 PPTX 输出共享同一份 JSON 数据源，保证一致性。

## 二、第一步：LLM 生成大纲 JSON

这是整个流程里最没有技术含量的一步——但最影响最终质量。

给 LLM 一个 JSON Schema 约束，让它输出结构化大纲，而不是自由文本。Prompt 大致如下：

```
你是一个 PPT 大纲生成器。根据以下主题，生成一份幻灯片大纲。

主题：2026年Q2产品复盘

要求：
- 8-12 页
- 每页必须包含 type 和 content 字段
- type 只能是：title | content | two-col | quote | chart | end
- 输出严格遵循以下 JSON Schema

{
  "slides": [
    {
      "type": "title",
      "content": {
        "heading": "string",
        "subtitle": "string",
        "date": "string"
      }
    },
    {
      "type": "content",
      "content": {
        "heading": "string",
        "body": ["string"],
        "notes": "string?"
      }
    },
    {
      "type": "two-col",
      "content": {
        "heading": "string",
        "left": ["string"],
        "right": ["string"]
      }
    },
    {
      "type": "quote",
      "content": {
        "text": "string",
        "author": "string"
      }
    },
    {
      "type": "chart",
      "content": {
        "heading": "string",
        "chartType": "bar|line|pie",
        "data": { "labels": ["string"], "values": [[number]] }
      }
    },
    {
      "type": "end",
      "content": {
        "heading": "string",
        "subtitle": "string"
      }
    }
  ]
}
```

LLM 返回的 JSON 大致长这样：

```json
{
  "slides": [
    {
      "type": "title",
      "content": {
        "heading": "2026 Q2 产品复盘",
        "subtitle": "增长、留存与下一步",
        "date": "2026年5月"
      }
    },
    {
      "type": "content",
      "content": {
        "heading": "核心指标",
        "body": [
          "DAU 从 12 万增至 18 万，增长 50%",
          "次日留存从 38% 提升至 44%",
          "付费转化率稳定在 4.2%"
        ]
      }
    },
    {
      "type": "chart",
      "content": {
        "heading": "DAU 趋势",
        "chartType": "line",
        "data": {
          "labels": ["4月", "5月", "6月"],
          "values": [[120000, 155000, 180000]]
        }
      }
    },
    {
      "type": "two-col",
      "content": {
        "heading": "做对了什么 / 下一步改进",
        "left": [
          "新用户引导流程优化",
          "Push 策略精细化",
          "社区运营冷启动完成"
        ],
        "right": [
          "长留存路径仍不清晰",
          "付费墙体验需重做",
          "数据基建待升级"
        ]
      }
    },
    {
      "type": "quote",
      "content": {
        "text": "增长不是目标，是结果。真正要盯的是留存曲线的斜率。",
        "author": "产品周会共识"
      }
    },
    {
      "type": "end",
      "content": {
        "heading": "谢谢",
        "subtitle": "Q3 目标讨论 →"
      }
    }
  ]
}
```

这一步的关键不是 LLM 能不能生成 JSON——GPT-4o、Claude、Gemini 都能。关键是你给的 Schema 是否足够约束，让输出可预测。

几个实操建议：

- **约束 type 枚举**：不让 LLM 自由发挥布局类型，6 种足够覆盖 90% 场景；
- **body 用数组不用字符串**：强制 LLM 拆成要点，而不是一坨段落；
- **notes 字段可选**：演讲者备注，LLM 很擅长写，但你不一定需要；
- **chart 的 data 要结构化**：labels + values，不要让 LLM 用自然语言描述数据。

## 三、第二步：准备 HTML 幻灯片模板

6 种布局，每种对应一个 EJS partial（Handlebars 同理）：

| type | 用途 | 关键元素 |
|------|------|----------|
| `title` | 封面 | 大标题 + 副标题 + 日期 |
| `content` | 正文 | 标题 + 要点列表 |
| `two-col` | 双栏 | 标题 + 左右两列要点 |
| `quote` | 引用 | 大字引言 + 署名 |
| `chart` | 图表 | 标题 + Chart.js 渲染 |
| `end` | 结尾 | 致谢 + 副标题 |

以 `title` 布局为例，EJS 模板长这样：

```html
<section class="slide slide-title">
  <h1><%= content.heading %></h1>
  <p class="subtitle"><%= content.subtitle %></p>
  <p class="date"><%= content.date %></p>
</section>
```

`content` 布局：

```html
<section class="slide slide-content">
  <h2><%= content.heading %></h2>
  <ul>
    <% content.body.forEach(item => { %>
      <li><%= item %></li>
    <% }) %>
  </ul>
</section>
```

`chart` 布局稍微特殊，需要 Chart.js 在浏览器端渲染：

```html
<section class="slide slide-chart">
  <h2><%= content.heading %></h2>
  <canvas id="chart-<%= index %>" data-type="<%= content.chartType %>"
    data-labels='<%= JSON.stringify(content.data.labels) %>'
    data-values='<%= JSON.stringify(content.data.values) %>'>
  </canvas>
</section>
```

CSS 部分用一套统一的幻灯片样式，16:9 比例，固定宽高。每个 `.slide` 占满一屏，`page-break-after: always` 方便打印预览。

模板只管视觉，不管数据——数据全在 JSON 里。这意味着换一套视觉风格，只需要换 CSS 和 EJS partial，JSON 不用动。

## 四、第三步：JSON + 模板 → HTML 预览

核心渲染逻辑，EJS 版本：

```javascript
const ejs = require('ejs');
const fs = require('fs');
const data = JSON.parse(fs.readFileSync('outline.json', 'utf-8'));
const html = ejs.render(fs.readFileSync('deck.ejs', 'utf-8'), { slides: data.slides });
fs.writeFileSync('preview.html', html);
```

`deck.ejs` 就是遍历 slides 数组，按 type 选择 partial：

```html
<!DOCTYPE html>
<html>
<head>
  <link rel="stylesheet" href="slide.css">
  <script src="https://cdn.jsdelivr.net/npm/chart.js"></script>
</head>
<body>
<% slides.forEach((slide, i) => { %>
  <%- include(`partials/${slide.type}.ejs`, { content: slide.content, index: i }) %>
<% }) %>
<script>
  document.querySelectorAll('canvas[data-type]').forEach(canvas => {
    new Chart(canvas, {
      type: canvas.dataset.type,
      data: {
        labels: JSON.parse(canvas.dataset.labels),
        datasets: [{ data: JSON.parse(canvas.dataset.values)[0] }]
      }
    });
  });
</script>
</body>
</html>
```

打开 `preview.html`，浏览器里直接看效果。调排版、调字号、调颜色，改 CSS 即时生效。

这一步的意义：**所见即所得地验证布局**，比生成 PPTX 再打开 PowerPoint 快得多。改一次 CSS 刷新一下，比在 PowerPoint 里点 8 次鼠标快。

## 五、第四步：PptxGenJS 输出 PPTX

这是整个方案最关键的设计决策：**跳过 HTML DOM 解析，直接用 JSON 中的布局参数调用 PptxGenJS API。**

HTML 只做预览，不承担输出职责。PPTX 的生成完全由 JSON 驱动：

```javascript
const PptxGenJS = require('pptxgenjs');
const pptx = new PptxGenJS();
const data = JSON.parse(fs.readFileSync('outline.json', 'utf-8'));

const layouts = {
  title: (c) => [{
    text: c.heading, options: { x: 0.5, y: 1.5, fontSize: 36, bold: true }
  }, {
    text: c.subtitle, options: { x: 0.5, y: 2.5, fontSize: 18, color: '666666' }
  }],
  content: (c) => [{
    text: c.heading, options: { x: 0.5, y: 0.3, fontSize: 24, bold: true }
  }, {
    text: c.body.map(b => ({ text: b, options: { bullet: true, fontSize: 14 } })),
    options: { x: 0.5, y: 1.2, w: 9 }
  }],
  twoCol: (c) => [/* 左右双栏布局 */],
  quote: (c) => [/* 引用布局 */],
  chart: (c) => [/* PptxGenJS 原生图表 */],
  end: (c) => [/* 结尾布局 */]
};

data.slides.forEach(slide => {
  const s = pptx.addSlide();
  const elements = layouts[slide.type](slide.content);
  elements.forEach(el => s.addText(el.text, el.options));
});
```

核心循环不到 20 行。每种布局是一个函数，接收 content，返回 PptxGenJS 的元素描述数组。

注意 `chart` 类型：PptxGenJS 有原生图表支持（`addChart`），不需要 Chart.js 的输出，直接用 JSON 里的 `data.labels` 和 `data.values` 生成：

```javascript
chart: (c) => [{
  type: c.chartType,
  data: [{ name: 'Series', labels: c.data.labels, values: c.data.values[0] }],
  options: { x: 0.5, y: 1.2, w: 9, h: 5 }
}]
```

最终输出：

```javascript
pptx.writeFile({ fileName: 'output.pptx' });
```

## 六、为什么"HTML 预览 + PptxGenJS 输出"比纯 HTML→PPTX 好

有人会问：既然已经有 HTML 预览了，为什么不直接把 HTML 转成 PPTX？

试过的人都知道，这条路有三个坑：

**1. 布局精度**

HTML 是流式布局，PPTX 是绝对定位。一个 `flex` 布局在浏览器里看着完美，转成 PPTX 后元素可能挤在一起或溢出。`html2pptx` 类工具只能做粗略映射，间距、对齐、嵌套关系都会丢失。

**2. 字体**

HTML 预览用的是系统字体，PPTX 需要嵌入或指定字体名称。转换工具往往无法正确处理字体回退链，导致 PPTX 打开后字体全变。

**3. 图表和动画**

Chart.js 渲染的是 Canvas 像素，转成 PPTX 只能当图片插入，无法编辑数据。PptxGenJS 的 `addChart` 生成的是原生 PowerPoint 图表，打开后可以修改数据、调样式、加动画。

所以正确分工是：

- **HTML**：快速预览、验证内容逻辑和排版感觉；
- **PptxGenJS**：精确输出、保证字体/图表/动画可编辑。

两者共享 JSON 数据源，HTML 改了内容就重新生成 JSON，PPTX 自动同步。

## 七、避坑与进阶

**LLM 幻觉**

LLM 生成的大纲里，数据和引用可能全是编的。"DAU 从 12 万增至 18 万"——这个数字从哪来的？必须人工核实。JSON 的结构化格式反而让核实更容易：遍历 `body` 数组，逐条确认。

**每页信息量**

PPT 不是文档。LLM 倾向于生成大量文字，因为它的训练数据里"详尽"比"精简"更受奖励。你需要在 prompt 里明确约束：每页不超过 5 个要点，每个要点不超过 15 字。

**企业模板**

有品牌规范的公司，只需要把 EJS partial 里的 CSS 和 PptxGenJS 的布局参数换成企业模板的数值。JSON 不用动，渲染逻辑不用动，换皮肤就行。

**扩展布局类型**

6 种布局覆盖 90% 场景。剩下 10% 可以按需扩展：

- `image`：全屏图片 + 标题叠加；
- `table`：数据表格，PptxGenJS 有 `addTable` 原生支持；
- `timeline`：时间线，用 PptxGenJS 的形状和连线画；
- `comparison`：对比布局，两列各有图标和文字。

每加一种布局，只需要写一个 EJS partial + 一个 PptxGenJS layout 函数，其他流程不变。

## 写在最后

AI 生成 PPT 的正确姿势，不是"一句话出幻灯片"。

那种方案听起来很酷，但生成的 PPT 往往内容空洞、排版失控、数据不可信，改起来比从零做还慢。

正确的做法是三层分离：

1. **LLM 管内容**：生成结构化 JSON 大纲，这是 LLM 最擅长的事；
2. **模板管排版**：EJS/Handlebars 渲染 HTML 预览，快速验证布局；
3. **PptxGenJS 管输出**：按 JSON 参数原生生成 PPTX，保证精度和可编辑性。

三层各司其职，互不耦合。换 LLM 不影响模板，换模板不影响输出逻辑，加布局类型不影响已有流程。

这才是工程化的 AI 工作流——不是让 AI 替你做 PPT，而是让 AI 把你从排版劳动里解放出来，让你专注在想清楚"说什么"这件事上。
