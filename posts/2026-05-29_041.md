# OpenClaw 架构拆解：从 Gateway 微内核到 Dreaming 记忆引擎

> OpenClaw 是一个 TypeScript 构建的 AI Agent 微内核平台——Gateway 做编排、Channel 做协议适配、Plugin SDK 做能力扩展、Dreaming 做记忆自动沉淀。本文从开发者视角拆解其核心架构：Gateway 五大角色、Channel 25+ 适配器契约、Agent 执行引擎的三层容错、Auth Profile 轮转降级、以及 Dreaming 三阶段加权晋升算法——把每个"不完美"背后的工程取舍看清楚。

![OpenClaw 架构拆解](images/post-041/cover.svg)

---

## 一、看山三境：从惊艳到理解

第一次见 OpenClaw，所有人都被惊艳了——24/7 后台常驻、跨多 IM 通道无缝流转、有人格长期记忆、自主完成开放性复杂任务。"这就是 AI 时代的私人助理操作系统"。

用了一段时间，光环褪色。OpenClaw 费 token（Bootstrap 每轮 push 几万 token）、健忘（Compaction 默认有损 + Dreaming 默认关）、复杂任务交付度低。两个都还在路上。

踩完坑再回头看源码，反而看懂了每个"不完美"背后的工程取舍。OpenClaw 用 4 个设计回答了 4 个重要问题——多协议可插拔契约（Channel 25+ Adapter）、LLM 上下文资源预算（可插拔 Context Engine + 多级 Compaction）、记忆自动沉淀不退化（Dreaming 三阶段加权晋升）、凭证失败与业务失败分治。

## 二、五层架构与核心设计理念

![OpenClaw 五层架构](images/post-041/01-architecture.svg)

OpenClaw 的架构分为五层：**触达层**（Channel Plugin）、**编排层**（Gateway）、**能力层**（Plugin SDK）、**记忆层**（向量引擎 + Dreaming）、**模型层**（9 种 LLM API + 降级链）。

四个核心设计理念：

**本地优先（Local-First）**：OpenClaw 不是云服务，而是运行在用户设备上的 Gateway 进程。所有会话数据、配置、媒体文件都存储在 `~/.openclaw/` 目录下。Gateway 是控制平面，Agent 是产品本身。

**万物皆插件（Everything is a Plugin）**：核心代码只负责编排——消息路由、会话管理、安全网关。所有具体能力（Discord 通道、Anthropic 模型、浏览器工具）都以插件形式实现，统一通过 Plugin SDK 注册。

**安全纵深（Defense in Depth）**：五层递进防御——从网络层 TLS 到认证层 Device Identity，从命令执行审批到插件安装扫描，再到沙箱隔离。执行策略默认为 deny，所有 shell 命令需要通过白名单或人工审批。

**记忆驱动（Memory-Driven）**：Agent 不仅有静态的工作区文件（SOUL.md, USER.md, MEMORY.md）定义人格与记忆，还有向量记忆引擎实现混合搜索、Dreaming 后台整合和 Active Recall 主动召回。

## 三、Gateway——系统的微内核中枢

Gateway 是整个系统的中枢，默认监听 `:18789`。它的职责远不止消息收发——聊天、会话、配置热加载、模型目录、执行审批、定时任务、远程节点、语音唤醒等几乎所有功能域都通过它的 RPC 方法暴露和调度。

### Gateway 的 5 大角色

**角色 1：唯一长驻进程（Single Source of Truth）**——避免多进程下的"WhatsApp 二次扫码、Telegram session 冲突"等致命问题。Channel session 天然是状态强相关的，不能多进程并发持有。

**角色 2：消息总线**——所有 channel, client, node 流量都走 Gateway 或由 Gateway 分发。不分协议入口——HTTP, SSE、私有 RPC 全部统一到 WS Schema。

**角色 3：多 Agent 路由的物理边界**——来自 Telegram@user1 的消息路由到 Agent A，来自 Discord@user2 的路由到 Agent B。不同 Agent 物理隔离（独立 workspace, SOUL, MEMORY, sessions）。这是 OpenClaw 最关键的差异化能力——解决了单 Agent 的上下文污染、工具链冲突、渠道风格差异三个瓶颈。

**角色 4：认证 + 信任边界**——Pairing v3 协议、idempotency key 必填、Token-based device identity。一个 Gateway 同时承担"消息路由 + 认证 + 信任根"。

**角色 5：嵌入式 HTTP Host**——Agent 可以主动构造 UI（canvas）让用户在浏览器看，不需要单独起 web server。

### "边界 vs 实现"哲学

Gateway 是边界，不是实现。协议定义、路由、认证由 Gateway 做；WhatsApp 会话生命周期、Channel 消息收发、Memory 整理、工具执行由插件做。Gateway 自己只做"协议 + 路由 + 信任"，其他全是插件——这才能保证微内核保持几千行核心代码。

## 四、Channel Plugin——25+ 适配器的完整契约

OpenClaw 的 Channel Plugin 不是简单的"消息适配器"——它同时承担协议适配、身份配对、安全审批、命令路由、配置生命周期、Gateway 协议绑定等角色，是一个完整的 IM 域协作单元。

Channel 接口包含 25+ 个可选槽位，分为 7 组：必选 4 项、Setup 三件套、Auth + Security 7 项、Messaging 7 项、协作能力 7 项、Gateway + 运维 6 项、反向工具。所有槽位都是可选的——Telegram/Discord 实现了 30+ 个，简单内部 webhook channel 只需实现 4 个必选 + 5 个可选。

### Channel 的核心价值：Per-channel Streaming Adapter

LLM 流式输出的"语义"在每个 IM 协议里完全不同——Telegram 用 `editMessageText` 反复编辑同一条消息、Discord 用 `interaction.followUp`、iMessage 不支持流式退化为分段发送。Channel 把这些差异封装掉。

### Channel Docking——跨 Channel 会话迁移

用户 Alice 在 Telegram 发起会话后想切到 Discord 继续，发 `/dock_discord`，Gateway 验证 identityLinks 确认两个账号属于同一用户后，保留 session 上下文不变，只换投递地址。不重建 session——相当于"AI 会话的呼叫转移"。这是单 channel 框架做不到的。

### 反向能力：Channel → LLM 工具

Channel 可以反向给 LLM 提供工具——Telegram Channel 提供 `telegram_get_chat_members`、Discord Channel 提供 `discord_react`、Slack Channel 提供 `slack_pin_message`。Channel 不只是消息通道，还是 LLM 的能力扩展源。

## 五、Agent 执行引擎——调度 + 容错 + 预算

OpenClaw 的 Agent Runtime 本质是一个"调度 + 容错 + 预算"的编排核——它不直接承担"如何思考"，而是通过 hook 和插件把具体能力外包出去，自己专注于三件事：决定调谁（调度）、失败了怎么办（容错）、花多少资源（预算）。

### 三层错误边界

- **内层**（runEmbeddedAttempt）：LLM 调用 + 工具执行的一次尝试，失败时抛 FailoverError
- **中层**（runEmbeddedPiAgent）：接住 FailoverError，决定是换 Auth Profile 重试还是向上抛
- **外层**（runWithModelFallback）：最终接住不可恢复的 FailoverError，遍历 `model.fallbacks[]` 切换模型

**可恢复错误的处理是静态可证明的，不靠 LLM 猜。**

### Auth Profile——不只是"API Key 数组"

你有 3 个 Anthropic 账号——个人 Pro、公司 Max、AWS Bedrock。Pro 额度用完自动切 Max，Max 被限频自动切 Bedrock，任何一个恢复了自动切回来。

OpenClaw 把每个账号建模为**带健康状态的对象**：支持三种凭据类型（ApiKey / Token / OAuth），每个 Profile 有独立的 cooldownUntil、cooldownReason、failureCounts。选取策略：类型偏好（oauth > token > api_key）→ 均衡轮转 → 冷却探针 → 用户锁定。

重启后磁盘 store 里 Profile A 还标着"billing 冷却到 14:30"→ 直接跳过 → 0ms 恢复。Hermes 重启后不记得哪个 key 上次挂了，又去撞已知欠费的 key。

### 双路径执行——把 Claude Code 当 Backend

OpenClaw 不把 Claude Code、Codex CLI 当"竞品"，而是当**可替换的执行 backend**：同一个 Gateway 管理、同一个 Agent 人格、同一套记忆系统、同一个会话转录格式，只是底层 LLM 调用换了个 Runner。

实际配置：Agent A 用 claude-cli backend（复用 Claude Code 的文件编辑/终端/浏览器等内置工具链），Agent B 用 embedded backend + DeepSeek（自有 API Key 直连），Agent C 用 codex-cli backend（走 ChatGPT Plus 订阅额度）。

### 三级 Compaction 策略

| 级别 | 触发条件 | 动作 |
|---|---|---|
| L1: Pre-request | 会话历史 > 阈值 | 主动调 compaction 生成摘要 |
| L2: Timeout-triggered | LLM 首 token 超时 且 prompt > 65% context | 紧急压缩后重试 |
| L3: Context overflow | API 返回 context_length_exceeded | 三级降级：compact → truncate → 抛错 |

L2 是容易忽视但很巧妙的一层——LLM 首 token 慢不一定是服务端慢，很可能是 context 太大导致 prefill 耗时过长。用"超时 + 大 context"双条件判断主动压缩后重试，比盲目切换 profile 更经济。

## 六、记忆系统——从静态文件到智能召回

![Dreaming 记忆晋升](images/post-041/02-memory-flow.svg)

记忆系统是 Agent 执行引擎的关键上游——Agent Engine 在每次 `buildPrompt()` 时从记忆系统获取相关上下文注入 System Prompt。

### 双层存储架构

- **召回层（pull）**：`memory/YYYY-MM-DD.md`——每天的记忆文件，不自动注入 LLM context，需要 LLM 调 `memory_search` 才能看
- **静态层（push）**：`MEMORY.md`——全局记忆文件，每轮注入 LLM context

4 路径流入召回层（`/new` 触发、Compaction Pre-Flush、LLM 主动写入、Dreaming 摄入），1 路径晋升到静态层（Dreaming Deep 阶段）。

### Dreaming 三阶段加权晋升

**Light Sleep**：读取近期短期召回、每日记忆和脱敏会话，去重后整理候选条目。仅整理候选，不写 MEMORY.md。

**REM Sleep**：提取反复主题 + 选候选"潜在真理" + 反馈 Deep 排名权重。REM 不看 diversity 和 recency，只关心"内在质量"——找的是"内容本身过硬"的稳固事实。

**Deep Sleep**：6 信号加权评分 + 三重门控后写入持久记忆。

Deep 晋升评分公式：

```
score = 0.30 × relevance + 0.24 × frequency + 0.15 × diversity
      + 0.15 × recency + 0.10 × consolidation + 0.06 × conceptual
```

三重门禁缺一不可：`score ≥ 0.75` AND `recallCount ≥ 3` AND `uniqueQueries ≥ 2`。单纯总分高可能一次召回特别准；单纯命中多可能同一 query 反复命中；三个都通过才是真正"重要"。

**Dreaming 默认 opt-in 关闭**——这是关键工程取舍：每次 sweep 跑 LLM 评分有成本，错误晋升会污染所有后续对话（MEMORY.md 每轮注入 LLM），cron + timezone + 阶段调度增加复杂度。

### 混合搜索算法

检索时采用 BM25 + 向量相似度混合搜索，经过三层后处理：加权融合 → 时间衰减（半衰期 30 天，"常青"文件豁免）→ MMR 多样性重排（避免返回高度相似的结果）。

## 七、多 Agent 路由——物理隔离 vs 逻辑隔离

OpenClaw 支持在同一 Gateway 下运行多个 Agent，通过 `agents.bindings` 配置将不同来源的消息路由到不同 Agent。每个 Agent 拥有独立的工作区（人格/记忆/Dreaming）。

路由匹配按优先级逐级尝试：精确用户 → 线程父级 → 同类型通配 → 服务器+角色 → 服务器/组织 → 团队 → Bot 账号 → 整个通道 → 兜底 main Agent。

Agent 间通信通过 `agentToAgent` 工具实现 4 种协作模式：Supervisor（中央统筹）、Router（分诊台）、Pipeline（串行传递）、Parallel（并行执行）。

**关键认知**：一个 Agent = 一份记忆 = 一个服务对象。同一 Agent 下多用户共享记忆会导致偏好串扰。正确做法是为不同用户配置多 Agent 路由绑定。

## 八、预算贯穿 Runtime——把稀缺资源显式量化

OpenClaw 对"预算"的理解不是"token 限额"那么窄，而是**把 runtime 里的每一种稀缺资源都显式量化，并配一条超限后的降级路径**：

| 稀缺资源 | 预算形式 | 超预算降级 |
|---|---|---|
| 上下文窗口 | contextTokenBudget | 三级 Compaction 降级 |
| 单次工具输出 | 30% 软限 + 16K 硬限 | head + tail 截断，error 关键词优先保留 |
| 启动上下文 | maxChars + totalMaxChars | 按优先级裁剪 + 截断告警注入 LLM |
| 循环迭代 | 按 profile 数动态算 | 抛 FailoverError 切模型 |
| 凭证可用性 | 分级退避（30s → 1min → 5min） | 轮换到下一个 profile |
| 子 Agent 递归 | spawnDepth + orchestrator/leaf | 达到深度后强制 leaf |

Bootstrap 截断策略是**头尾保留，砍中间**——保 head 70%（全局约定）+ 保 tail 20%（最新更新），砍 middle（历史细节，损失最小）。截断后注入告警到 prompt，LLM 知道自己被截断后会主动用工具补读完整文件。

所有 token 估算都乘以 1.2 安全边际——`estimateTokens()` 是估算不是精确 tokenizer，乘 1.2 留 20% buffer 防止"估算说够，实际溢出"。

## 九、双向连接——既能当主，也能当辅

OpenClaw 不试图取代任何现有工具，而是和它们互联。它提供三种粒度的暴露面：

**MCP Server（工具粒度）**：`openclaw mcp serve` 暴露 9 个具体工具——conversations_list、messages_read、messages_send、events_wait 等。Claude Code 可以通过 MCP 协议访问 OpenClaw 管理的所有 Channel 会话。

**ACP Server（Agent 粒度）**：`openclaw acp` 启动 ACP 协议前端，Zed 等 IDE 可以把 OpenClaw 当 Agent 用。

**Gateway HTTP API（系统粒度）**：Gateway 本身有完整的 HTTP API，任何能发 HTTP 请求的程序都能调它。

场景决定了谁是主谁是辅——用户在 QQ Bot 和 Agent 对话时 OpenClaw 是主；用户在 Claude Code 里写代码时 Claude Code 是主，OpenClaw 是消息通道扩展。

## 十、写在最后：工程取舍的清晰度

OpenClaw 的架构不是"完美"的——费 token、Dreaming 默认关、多用户场景需要手动配多 Agent。但它的价值在于：**每个"不完美"都是经过权衡的取舍，而不是疏忽。**

- Gateway 做边界不做实现 → 核心代码保持几千行，但扩展性极强
- Channel 25+ 适配器 → 重，但加上之后不用再操心认证/重载/诊断
- Dreaming 默认关 → 成本和副作用可控，启用后收益确定
- Auth Profile 轮转 → 凭证管理从"碰运气"变成"可证明的降级"
- 预算贯穿 Runtime → 每种稀缺资源都有独立的降级路径

**一个架构好不好，不是看它有多完美，而是看它的取舍有多清晰。** OpenClaw 的取舍是清晰的——这比"什么都支持但什么都做不好"强得多。
