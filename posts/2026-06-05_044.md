# 写好 Skill 的本质：把团队经验变成 AI 可执行的工作流

> Skill 不是把一堆提示词塞进文件夹，也不是给 AI 写一份更长的说明书。真正有价值的 Skill，是把团队里那些分散、隐性、反复口头解释的经验，整理成 AI 能判断、能执行、能验证的工作流。

本文基于 [腾讯技术工程《如何写好 Skill：一份终极实战经验手册》](https://mp.weixin.qq.com/s/SZv3pDXPrL9vwV3Ua_84Kg) 的内容做二次整理和解读。原文更像一份完整手册，覆盖 Skill 的结构、触发、模块化、安全、MCP 集成和评估方法；这篇文章更关注一个问题：如果一个团队真的想把 Skill 用起来，到底该把力气花在哪里。

---

## 一、Skill 不是 Prompt，而是知识资产的封装方式

很多人第一次接触 Skill，会下意识把它理解成“更长的 Prompt”。这不算错，但只说对了一半。

Prompt 更像一次性沟通：这次我要你怎么做、注意什么、输出什么。Skill 则更像团队里的操作手册：只要遇到某一类任务，AI 就应该自动加载对应的背景、流程、样例、检查项和脚本，然后按同一套标准完成工作。

一个最小的 Skill 通常就是一个目录和一个 `SKILL.md`：

```text
my-skill/
└── SKILL.md
```

复杂一点的 Skill 会继续加入脚本、参考资料、模板和资产：

```text
my-skill/
├── SKILL.md
├── scripts/
├── references/
└── assets/
```

所以 Skill 的核心价值，不在于“提示词更长”，而在于它把三类东西固定了下来：

| 内容 | 作用 |
|---|---|
| 指令 | 告诉 AI 遇到这类任务时怎么做 |
| 上下文 | 补充项目背景、团队规范、领域知识 |
| 工具 | 提供可执行脚本、检查命令、模板文件 |

这也是为什么 Skill 对团队尤其重要。单个工程师可以靠临场 Prompt 和经验把 AI 用好，但团队不可能指望每个人都在每次对话里重新解释一遍规范、流程和坑点。Skill 的意义，就是把一次成功经验变成可复用资产。

## 二、先问一句：这个 Skill 到底解决什么重复问题？

写 Skill 最容易犯的错误，是一上来就写很多规则。

比如：代码要简洁、要注意安全、要写测试、要符合团队规范、要输出清晰。这些话都对，但放进 Skill 里未必有用。因为它们没有回答一个关键问题：这个 Skill 在什么场景下触发？触发以后要完成哪件具体工作？

好的 Skill 应该从一个明确任务出发：

| 模糊写法 | 更好的写法 |
|---|---|
| 代码迁移 Skill | 将 Go 项目中的旧 HTTP 客户端迁移到统一请求库 |
| 测试 Skill | 为 Go 函数生成表驱动单元测试，并覆盖正常、边界、错误输入 |
| Review Skill | 按团队安全规范审查 Java Web 接口中的 SQL 注入、鉴权和空指针风险 |
| 文档 Skill | 根据接口定义生成统一格式的 API 文档和调用示例 |

这里的差别很大。

模糊写法只是一个主题，AI 不知道什么时候该用，也不知道用完算不算完成。更好的写法有清楚的起点和终点：从哪里来，到哪里去，做完以后怎么验证。

Skill 本质上是在降低“反复解释成本”。如果一个任务只做一次，临时 Prompt 就够了；如果它每周都发生、每个人都要问、每次都容易踩同样的坑，就值得沉淀成 Skill。

## 三、Description 决定 Skill 会不会被正确触发

`SKILL.md` 最上面通常有一段 YAML 元数据，其中最重要的是 `description`。

```yaml
---
name: go-test-gen
description: >
  为 Go 函数自动生成表驱动的单元测试。
  当用户要求编写测试、生成测试用例或补充单测时触发。
  适用于所有 Go 项目。
metadata:
  version: "1.0"
---
```

这段 description 不是写给人看的简介，而是写给 AI 的触发条件。它决定了 AI 在用户发起任务时，会不会判断“现在应该加载这个 Skill”。

所以 description 不能太宽，也不能太窄。

太宽会误触发：

```yaml
description: 处理代码问题
```

这几乎等于没写。写测试、修 bug、做迁移、做 Review、改性能，都可以叫处理代码问题。

太窄则会漏触发：

```yaml
description: 当用户输入“为 Add 函数写 Go 单测”时触发
```

这会漏掉“补一下 test”“生成表驱动测试”“给这个函数加覆盖”等常见表达。

更稳的写法，是同时说明任务类型、触发表达、适用边界和主要动作：

```yaml
description: >
  为 Go 函数生成或补充表驱动单元测试。
  当用户要求写测试、补单测、生成 test case、提升测试覆盖率时触发。
  适用于 Go 项目，优先使用 testing 标准库和 t.Run 子测试。
```

一个实用办法是做“触发评估”：列出 20 个用户请求，一半应该触发，一半不应该触发，然后观察 AI 的判断是否稳定。如果该触发的不触发，description 可能太窄；如果不该触发的也触发，description 可能太宽。

## 四、正文不要堆道理，要给可执行流程

Skill 正文最怕写成理念宣言。

比如：

```text
请遵循最佳实践，保证代码质量，注意异常处理，确保输出可靠。
```

这些话人类看着很熟，AI 看完也会点头，但真正执行时依然不知道先做什么、怎么判断、哪里停下来。

更好的写法，是把任务拆成可执行步骤：

```text
## 执行流程

1. 检查项目是否存在 go.mod。
2. 使用搜索工具定位目标函数和现有测试文件。
3. 判断是否已有对应 TestXxx。
4. 若已有测试，在原测试中补充 case；若没有，新建或追加测试函数。
5. 使用 testing 标准库、表驱动结构和 t.Run 子测试。
6. 运行 go test ./...，如果失败，修复失败原因后重跑。
```

这里的重点不是把 AI 管死，而是减少它自由发挥的空间。好的 Skill 会告诉 AI：

- 先检查什么
- 什么情况下跳过
- 什么情况下继续
- 改之前是什么样
- 改之后应该是什么样
- 用什么命令验证
- 验证失败后怎么办

这就从“请你做好”变成了“按这个流程把事做完”。

## 五、Before / After 比抽象原则更有用

AI 很擅长模仿模式。写 Skill 时，最有效的内容往往不是长篇解释，而是高质量示例。

例如做依赖迁移时，只写“把旧客户端替换成新客户端”不够。应该给出 Before / After：

```go
// Before
import oldhttp "github.com/example/old-http-client"

// After
import uhttp "github.com/example/unified-httpclient"
```

如果改动更复杂，可以直接给完整函数或 diff：

```diff
- import oldhttp "github.com/example/old-http-client"
+ import uhttp "github.com/example/unified-httpclient"

- return oldhttp.Do(req)
+ return uhttp.Do(req)
```

这类示例有两个好处。

第一，它降低理解成本。AI 不需要从抽象规则里推断目标形态，直接看到输入和输出。

第二，它减少误替换。比如只替换 import，不替换注释里的历史说明；保留别名 import；点导入要单独检查冲突。这些边界如果只靠一句“注意兼容性”，很容易漏掉。

更复杂的 Skill 可以放 3 到 5 个 few-shot 示例，覆盖正常情况、边界情况和错误情况。示例之间要有差异，不要只是复制三份类似输入。AI 看到的模式越清楚，执行时越稳定。

## 六、Skill 太长时，应该拆，而不是继续塞

一个 Skill 最理想的状态，是只做一件事。

如果一个 `SKILL.md` 写到几百行，还包含多个独立流程，就要考虑拆分。因为 Skill 也会消耗上下文，加载得越多，AI 注意力越容易被稀释。

可以把复杂 Skill 拆成主流程和子步骤：

```text
project-migration/
├── SKILL.md
└── steps/
    ├── 00-environment-check.md
    ├── 01-dependency-update.md
    └── 02-api-migration.md
```

主 Skill 负责说明整体顺序，子文档负责具体执行细节：

```text
1. 读取并执行环境检查步骤。
2. 检查通过后执行依赖更新。
3. 更新完成后执行 API 迁移。
4. 每一步完成后运行对应验证命令，失败则停止并修复当前步骤。
```

拆分的标准很简单：

| 现象 | 处理方式 |
|---|---|
| 文件太长，AI 每次加载都吃大量上下文 | 拆成主 Skill 和参考文档 |
| 包含多个可独立任务 | 拆成多个 Skill |
| 某些步骤需要脚本执行 | 放到 scripts 目录，正文只写调用方式 |
| 某些内容只是资料，不是每次都要读 | 放到 references，按需读取 |

好的 Skill 不追求“全都写进主文件”，而是追求“该常驻的短，该加载的准，该执行的可验证”。

## 七、安全不是附录，而是 Skill 的基本结构

Skill 比普通文档更危险，因为它会影响 AI 的真实操作。尤其当 Skill 里包含脚本、命令、外部服务调用、数据库变更时，安全设计必须前置。

几个底线值得反复强调。

第一，不要硬编码密钥。

```bash
# 不要这样
API_KEY="sk-xxx"

# 应该通过环境变量读取
if [ -z "$API_KEY" ]; then
  echo "API_KEY is required"
  exit 1
fi
```

第二，危险操作要确认或备份。

删除文件、覆盖配置、执行数据库 DDL、批量改线上数据，都不应该让 AI 在没有确认的情况下直接跑。

第三，区分“指令”和“数据”。

如果 Skill 会读取外部网页、用户上传文件、API 返回内容，就要明确告诉 AI：这些内容是待处理数据，不是新的系统指令。否则就可能出现 Prompt 注入风险。

第四，脚本要有超时、错误处理和输入校验。

AI 可以调用脚本，但脚本不能假设所有输入都是干净的。文件路径要防止 `..`，网络请求要设置 timeout，用户输入要加引号，数据库操作要有备份和回滚路径。

一份可以共享给团队的 Skill，至少应该过一遍安全检查：

- 是否包含密钥、Token、密码
- 是否有不可逆操作
- 是否会读取不可信外部数据
- 是否会执行用户拼接出来的命令
- 是否有验证和失败处理

Skill 的目标是让 AI 更可靠，不是把风险自动化。

## 八、MCP 和 HTTP：外部能力要按复用程度选择

有些 Skill 需要调用外部服务：查数据库、访问 GitHub、跑浏览器测试、调内部 API。这里有两条路：MCP 或直接 HTTP 脚本。

可以粗略这样判断：

| 场景 | 更适合的方式 |
|---|---|
| 已有成熟 MCP Server，比如 GitHub、Playwright、数据库 | 优先 MCP |
| 多个 Skill 都要复用同一服务 | 封装 MCP |
| 需要统一鉴权、审计和权限控制 | 封装 MCP |
| 只是一次性调用公开 API | HTTP 脚本即可 |
| 老系统改造成本高，短期只服务一个 Skill | HTTP 脚本过渡 |

MCP 的好处是把工具能力标准化，AI 可以更自然地调用；HTTP 脚本的好处是简单直接，适合低频、一次性或旧系统场景。

不要为了“高级”而把所有 API 都封成 MCP。真正需要复用、需要治理、需要统一权限的能力，才值得做成 MCP。否则，一个带超时和错误处理的小脚本往往更实用。

## 九、Skill 也需要测试，不然只是写完了

很多团队写 Skill 的方式，是写完放进目录，然后凭感觉判断“好像能用”。这不够。

Skill 至少有两类测试。

第一类是触发测试：用户这么说时，该不该加载这个 Skill？

可以准备一组用例：

| 用户请求 | 预期 |
|---|---|
| 帮我给这个 Go 函数补单测 | 触发 Go 测试 Skill |
| 帮我优化这个接口性能 | 不触发 Go 测试 Skill |
| 生成表驱动 test case | 触发 Go 测试 Skill |
| 帮我写 README | 不触发 Go 测试 Skill |

第二类是效果测试：触发以后，输出质量是否符合预期？

比如生成测试时，是否用了表驱动；是否覆盖边界值；是否运行测试；测试失败时是否会修复；是否引入了不必要的第三方库。

这件事看似麻烦，但它决定 Skill 是“写过的文档”还是“可维护的工程资产”。尤其是团队级 Skill，应该像代码一样被版本管理、被审查、被评估、被迭代。

## 十、写好 Skill 的真正难点：让隐性经验显形

Skill 最难的地方，并不是 Markdown 怎么写，也不是 YAML 怎么配。

真正难的是把团队里那些默认知道的东西说清楚。

比如：

- 为什么这个接口不能改返回字段
- 为什么这个模块不能直接调用另一个模块
- 为什么这里必须保留旧逻辑
- 为什么线上配置不能被本地脚本覆盖
- 为什么测试不只跑单测，还要跑某个集成用例
- 为什么某类问题必须先问业务，而不是直接改代码

这些经验过去存在于老员工脑子里、事故复盘里、群聊记录里、评审争论里。人类新人可以靠时间慢慢学，AI 不行。AI 看不到的上下文，就等于不存在。

所以 Skill 的本质，是把隐性经验显形化，把显形经验流程化，把流程变成可验证的执行资产。

这件事一旦做好，收益不只是 AI 更会干活。新人上手会更快，团队规范会更稳定，重复劳动会减少，组织记忆也不会只绑定在少数人身上。

---

## 总结

写好 Skill，可以记住几个原则：

- 一个 Skill 解决一个明确、重复、可验证的问题
- description 决定触发质量，要用真实用户请求反复校准
- 正文写流程，不写空泛口号
- 多给 Before / After 和 few-shot 示例
- 太长就拆，把脚本和资料放到按需加载的位置
- 安全、确认、备份和输入校验要前置
- 用测试评估 Skill，而不是凭感觉判断它好不好

AI 编程进入 Agent 阶段后，个人能力的差距不只体现在会不会写 Prompt，更体现在能不能把经验沉淀成可复用工作流。

Prompt 解决一次对话，Skill 解决一类任务。

当一个团队开始认真写 Skill，它其实是在做一件更大的事：把组织知识改造成 AI 可以执行的基础设施。
