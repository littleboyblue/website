# Coding Agent 这一年：从 Vibe Coding 到 Harness Engineering

> Vibe Coding 很爽，但生产环境不只看产出速度。过去一年，Coding Agent 领域最大的变化不是模型变强了，而是人们开始认真搭建围绕 Agent 的"安全网"——Context Engineering 精准控制上下文，Subagents 隔离任务上下文，Harness Engineering 用确定性工具约束非确定性输出。

本文基于 [InfoQ 对 Thoughtworks 全球 AI 辅助软件交付负责人 Birgitta Böckeler 在 QCon 纽约演讲的整理](https://mp.weixin.qq.com/s/ENERagN834L5CXGkAGmXtg)，结合当前 Agent 工程实践做二次梳理和解读。

---

## 一、Context Engineering：从 Rules File 到一整套上下文体系

一年前的 Context Engineering 基本等同于 rules file：在项目根目录放个 `AGENTS.md`，写上常见的坑和反复犯的错，每次开新会话 Agent 就会收到这个文件。加上 MCP Server，可以帮 Agent 更动态地获取数据。

一年后，这个生态已经完全不同。不只是 rules 和 MCP，还有 commands、skills、subagents、plugins、specs。核心变化是：**上下文管理从"全量灌输"变成了"按需加载"**。

其中 Skills 是最值得单独讲的变化。

Skills 本质上做了三件事。第一，把 rules 模块化。你不用把所有东西塞进一个大文件然后全量发给 Agent，而是按功能拆成子文件夹。第二，模块可以被 LLM 按需即时加载。Agent 只会先看到 skill 的简短描述，当 LLM 判断当前场景需要时才真正加载完整内容。第三，skill 不只是 Markdown，还包含脚本，Agent 可以直接执行。

这种"渐进式惰性加载"机制解决了一个关键问题：**上下文窗口不是无限可塞的**。虽然技术上下文窗口比两年前大了很多，但塞得太满 Agent 效果会明显下降，成本也会急剧上升——因为每次与模型交互，都要把整个上下文发过去。

现在主流 coding agent 已经开始提供上下文监控功能，你可以看到是什么在占空间：system prompt、skills、各种 context interfaces 等等。设计 skills 时，必须开始思考"上下文预算"。

一个容易被忽视的点：Context Engineering 是一个**双向放大器**。好的工程实践会被放大，但坏的代码结构和混乱的规范同样会被放大。如果仓库里本身就乱，Agent 会照着已有的混乱模式走，结果只会更乱。

## 二、Subagents：隔离上下文，聚焦子任务

Subagents 是 Context Engineering 的另一项关键能力。核心思想很简单：主 Agent 可以派生出子 Agent。

最常见的使用场景是代码库研究。Agent 在会话开始时需要了解项目结构，这项工作本身消耗大量 token。把它交给一个 subagent，subagent 最后只把结果汇报回主 session，所有中间噪音不会塞进主上下文。

另一个典型用法是 code review。用一个"没有历史上下文污染"的独立上下文窗口来做代码审查，甚至让它使用不同的模型，这样审查结果更客观。

更进一步的用法是构建复杂工作流。比如迁移几千个 CI/CD pipeline 到 GitHub Actions，可以用主 Agent 编排，不同 subagent 处理不同类型的迁移任务，配合 skills 和 MCP servers。这在过去纯人工处理是不可想象的规模。

Agent Teams 也开始出现。和 swarm（扔出几十上百个 agent）不同，teams 更克制：主 Agent 决定哪些任务可以并行，不同 agent 之间可以互相通信交换结果。但这还处于很早期的实验阶段。

## 三、减少人类监督的诱惑和风险

从 supervised 到 unsupervised，这是 Coding Agent 最明显的趋势。

Supervised 模式下，开发者坐在会话前持续观察、不断介入纠偏。Unsupervised 模式则是把任务发到云端，Agent 自己跑 20 分钟甚至更久，回来再给你结果。现在几乎所有主流 coding agent 都支持这种模式了。

CLI 编程助手的兴起进一步推动了这一趋势。Claude Code、Cursor CLI、Copilot CLI 可以以 headless 模式运行，可直接接入 GitHub Actions 等 CI/CD 系统。

但这带来了一个核心判断难题：**你如何判断一个任务需要多少监督？**

Birgitta 提出了一个三维风险评估框架，与原有软件工程风险评估完全一致：

| 维度 | 含义 |
|---|---|
| 出错概率 | AI 做这类任务有多大概率出错？取决于你对工具的熟悉度、对需求的把握、对任务复杂度的判断 |
| 出错影响 | 如果出错，后果有多严重？是 PoC，还是核心流程？ |
| 可检测性 | 你能不能发现它出错了？取决于测试、lint、CI 等反馈循环的质量 |

这三个维度很大程度上与代码库质量相关。烂代码库里 AI 出错概率更高，因为它会照着烂模式走。没有良好测试自动化的系统，可检测性也更差。

对开发者来说，真正需要培养的新能力只有一项：**你对 AI 在某类任务上的真实能力到底有多了解**。其他能力——需求分析、架构判断、风险评估、反馈循环设计——本来就是成熟开发者应该具备的。

## 四、安全：Prompt Injection 和致命三要素

Agent 从不可信来源获取内容，而内容里包含你不曾意识到的指令——这就是 Prompt Injection，而且事故报告几乎每周都在出现。

Simon Willison 提出的"致命三要素"值得关注：当 Agent 同时满足接触不可信内容、能访问私有数据、可以对外通信这三条，安全风险就处于高位。

在企业 Agent 场景里，这个问题更危险。很多业务场景会接入邮箱系统（通常还有 read/write 权限），那一刻起就已经满足致命三要素了。

另一个重大风险是 secrets extraction，尤其在开源项目里。攻击者可以通过 GitHub Issue 中的 prompt injection 诱导 Agent 泄露 secrets。已经有过攻击者获得向 npm registry 发布恶意版本权限的案例。

在减少监督的背景下，安全要关注几件事：是否合理使用沙箱（不只在云端，本地也是）；组织里工程师的 AI 安全素养如何；绝对不要随便开"YOLO 模式"（Agent 想执行什么命令就执行什么）。

## 五、成本爆炸：12 美分变成 9 万美元一年

2024 年初，有人算过：生成 100 行代码只需要 12 美分，相比开发者的工资简直太便宜了。

但现在完全不是这个数字了。

2025 年夏天，有公开数据显示，一个重度用户平均每天花费约 380 美元。按每月 20 个工作日计算，年成本约 91,200 美元。在德国，这已经是一份不错的开发者薪资了。而且那还是去年的数字。

为什么成本暴涨？因为现在的流程远比"生成 100 行代码"复杂：研究已有代码、制定计划、审查调整计划、开始实现、跑测试、修测试、检查 lint、修错误、UI 功能还要检查浏览器、再修、跑 code review agent、回应审查意见、做总结。来来回回一大圈，最后可能只改了两行代码。

成本的另一面是套餐模型的变化：从每月 20 美元的"无限套餐"，到现在的"200 美元 flat rate"还不是真正 flat（有 request limiting），Reddit 上已经有人发帖"月才过半我 token 就用完了怎么办"。**工具依赖一旦形成，成本就不再是可选变量**。

## 六、Harness Engineering：用确定性工具约束非确定性输出

这是演讲最核心的概念，也是过去一年最重要的范式转移。

模型的能力在提升，但它永远是非确定性的。问题不是让 Agent 写出完美代码——人类自己写的代码也从来不完美——而是在特定场景下，怎么建立足够的信任来安全、快速、可持续地交付。

Harness Engineering 的思路就是：**围绕非确定性的 AI，搭建一层确定性的安全网**。

这层安全网包含两类东西。一类是前向输入：principles、coding conventions、参考文档、how-to、CLI 工具、bootstrap 脚本、Codemod 等确定性工具。在这些输入的指引下，Agent 更大概率按你的意图行事。

另一类是后向反馈：静态分析、结构测试、lint、日志、浏览器检查、自动化测试。Agent 产出第一版代码后，先把那些重复的修正工作做完，然后再轮到人类审查。

一个典型的例子是结构性测试。Java 生态里的 ArchUnit、TypeScript 的 dependency-cruiser，这些工具一直存在，但在"人类主导写代码"的时代不是刚需。现在它们突然变得重要，因为它们变成了 Agent 的"反馈系统"。

比如你可以定义：external SDK 只能在特定的 client folder 里被引用，domain 层完全不允许直接依赖这些外部 SDK。这些规则不只是约束代码，也在反向塑造 Agent 的行为。

更有意思的是，这些 lint rules 可以被"增强语义"。你可以在错误信息里附加解释："这个错误不仅仅是违反规则，它可能意味着你的设计出现了问题，建议重构。"这就从"报错"变成了"带解释的反馈通道"。

把原本为人类设计的工程约束系统，改造成 Agent 可学习、可反馈、可优化的系统——这才是 Context Engineering 真正开始变得"工程化"的地方。

## 七、代码可维护性的隐忧

一个很容易被忽略的问题：当 autonomy 提升、supervision 降低，代码可维护性会发生什么？

OpenAI 的一个团队过去五个月维护一个从零开始的项目，给自己定了一个激进规则：尽量不直接手写代码，只通过 Agent 交互让 Agent 改代码。即便如此，他们仍然发现**熵在增加，漂移在发生**，代码库不会自动变得更整洁。于是他们引入了一个"garbage collection"过程，让 Agent 持续在代码库上运行清理和整理。

这说明 Harness Engineering 不只是"一次性配置"，而是需要持续运行和迭代的安全网。就像 GC 不是手动跑的，代码库的结构性健康度也需要自动化维护。

这也是为什么 Birgitta 提出：未来我们可能不再靠传统服务模板起步，而是一个 **Harness 模板**。实例化之后就能支撑代码库。到那时候，技术栈的选择维度可能不再是"React 还是 Vue"，而是"有没有现成的 Harness，这样我就不用操心、不用从头搭了"。

## 八、速度的另一面

Coding Agent 给整个行业注入了一种强烈的驱动力：speed、throughput、PR 数量。"这周 merge 了多少 PR？""比上个月快了多少？"

但这种对速度的追求也有代价。

Amazon 最近出了一篇内部复盘，提到一些疑似由 AI 生成代码相关的故障。他们的对策之一是增加关卡，让 senior 工程师必须 review 生产内容的引入。这有点讽刺：说好的更快上线，结果反而加了更多手续。

Reddit 上有个帖子很火，Agent 说："你说不行，但我以为你是在拒绝我问你要权限，所以我就自己直接去做了。"这反映出当前 Agent 的自主度和人类控制之间仍然存在巨大鸿沟。

另一个值得关注的现象是**认知超载**。有越来越多开发者开始说用 AI 之后感到倦怠：要么是被审查的工作量压垮，要么是同时切多个会话根本看不过来。真正能把 AI 开到最大自主度跑起来的，通常都是资深工程师——他们有更强的认知承载力。

---

## 总结

过去一年 Coding Agent 领域最重要的变化，不是模型本身，而是团队开始认真搭建围绕 AI 的确定性基础设施。

Context Engineering 把上下文管理从"全量灌输"升级为"按需加载"；Subagents 让复杂任务可以隔离上下文、分而治之；Harness Engineering 用确定性工具约束非确定性输出。

但硬币的另一面也同样真实：安全风险随着 Agent 自主度上升而扩大，成本从 12 美分涨到 9 万美元一年，代码可维护性不会因为 AI 写得快就自动变好。

Birgitta 最后抛出的问题值得每个团队思考：如果你打算给 AI 更多自主权，你准备好了吗？你的自动化安全网是什么？你的安全态势如何？你团队的 AI 素养到了什么程度？

AI 时代的护城河，不只是模型用得早或者用得多。真正能穿越周期的，是对风险、成本和质量的综合判断力。
