# AI Agent 测评体系化实践：从"跑通一次"到"生产可靠"

> 当 AI Agent 从"Demo 可用"走向"生产可靠"，测评就是那道必须跨过的门槛。腾讯 TEG 云架构平台部网关测试团队在 AI Agent 测评领域建立了一套"确定性评分器 + Rubric 评分器 + 人工评分器"三类组合的完整测评框架，覆盖功能正确性、过程质量、效率成本、鲁棒性安全、体验对齐五大维度，并已在 TPerf 性能平台智能分析 Agent 项目中落地验证。无论你是刚开始构建 Agent 测评体系，还是已有初步实践希望系统化升级，都可以从中找到可直接复用的方法论、评分模板与工程实现方案。

本文基于 [腾讯技术工程《AI Agent 测评体系化实践》](https://mp.weixin.qq.com/s/PUbGqheJhFMmb6hGj1ZtOw) 做二次梳理，提炼核心框架与落地经验。

## 一、为什么要做测评？

### 1.1 三大痛点

Agent 的自主性带来了三个传统软件没有的问题：

- **非确定性**：同一 prompt 多次执行结果不同，"跑通一次"不代表"稳定能跑"
- **黑盒化**：模型升级、Prompt 微调、工具链变化都可能导致行为漂移，肉眼难以察觉
- **错误级联放大**：一次任务涉及几十步工具调用，前序步骤的一个小偏差会沿链路逐级放大，最终导致结论完全偏离

没有测评的后果：

| 痛点 | 后果 |
|------|------|
| 主观性强 | 依赖"感觉变好了"的直觉判断，缺乏量化依据 |
| 悄悄退化 | 改了 Prompt 或升级了依赖，旧场景悄悄变差却无人知晓 |
| 人工验证成本高 | Skill 越多、模型迭代越快，靠人肉回归的成本指数级增长 |
| 模型不敢升级 | 新模型发布时没有对比数据支撑切换决策 |
| 缺少效率基线 | 没有延迟/Token/费用的历史基线，线上变贵变慢时无法定位原因 |
| 过程易忽略 | 最终答案可能碰巧正确但推理路径是错的 |

### 1.2 核心理念

核心可以概括为一个公式：

**Eval（评估）= Agent 输入 → 执行 → 捕获执行过程（Trace + 产物） → 一组检查规则 → 可对比的分数**

测评的目标是建立一个**可重复、可量化、可持续演进**的评估闭环，而不是追求完美覆盖。关键在于：每次变更都能快速跑出一个可比较的分数，用数据代替直觉，用全量代替抽查。

## 二、测评框架：谁来评？评什么？

### 2.1 三类评委：谁来打分？

Agent 的输出既有可程序化验证的硬指标（文件存在、调用正确），也有只能靠语义理解才能判断的软指标（推理合理性、建议质量）。单一评分手段无法兼顾两者，因此 **Agent 测评没有"银弹评分器"，必须三类组合使用**：

**确定性评分器**（脚本 / 断言 / Lint / AST）—— 快、便宜、客观、可复现，负责所有"能用代码判断"的事

**Rubric 评分器**（LLM-as-Judge + Prompt + Schema）—— 灵活、可扩展，负责"代码搞不定但能结构化描述"的场景

**人工评分器**（专家）—— 昂贵、慢、黄金标准，负责"校准、诊断、兜底"

| 维度 | 确定性评分器 | Rubric 评分器 | 人工评分器 |
|------|-------------|--------------|-----------|
| 谁来评 | 脚本（Bash/Python） | 大模型（固定版本） | 领域专家 |
| 成本 | 毫秒级 / 免费 | 秒级 / API 费用 | 分钟–小时级 / 人力 |
| 稳定性 | 100% | 有抖动（需降噪） | 取决于标注员水平 |
| 门禁角色 | 硬门禁 | 分级门禁 | 不进门禁，做采样审查 |

**选择优先级：确定性评分器 > Rubric 评分器 > 人工评分器** —— 能用代码判断的绝不用模型，必要时用模型，人工用于校准。

确定性评分器典型用例：

- **工具调用检查**：是否调用了指定工具、参数是否正确
- **产物检查**：文件是否存在、内容是否符合预期
- **关键词匹配**：响应中是否包含/不包含特定内容
- **执行指标**：工具调用次数、token 消耗是否在阈值内
- **基线对比**：本次执行的过程和结果与基线快照逐项对比

人工评分器只在六个必要场景投入：校准 LLM 评委（一致率 ≥ 85%）、主观任务打分、诊断通过率异常、建立 Ground Truth、Trace 采样审查、高风险兜底。

### 2.2 五个维度：评什么？

从"做对了吗"到"好用吗"逐层递进：

| 大类 | 子维度 | 主要评委 | 优先级 |
|------|--------|---------|--------|
| 1. 功能正确性 | 结果正确性、任务完成度、指令遵循、工具调用正确性 | 代码 | P0 |
| 2. 过程质量 | 推理合理性、步骤最优性、信息完整性、上下文利用率 | Rubric + 人工 | P1 |
| 3. 效率与成本 | Token消耗、工具调用次数、延迟、失败重试率 | 代码 | P1 |
| 4. 鲁棒性与安全 | 一致性（pass^k）、异常恢复、抗对抗、幻觉率、越权风险 | 代码 + 人工 | P0 |
| 5. 体验与对齐 | 语气风格、清晰度、主动澄清、同理心、品牌一致性 | Rubric + 人工 | P2 |

> **pass^k**：k 次试验中每次都通过的概率，衡量稳定性；**pass@k**：k 次中至少 1 次通过的概率，衡量峰值能力。

**P0 先落地、P2 按需补充** —— 优先保证"做对了"和"靠谱"，再逐步覆盖过程、成本和体验。

#### 功能正确性（Functional Correctness）

回答"这次任务到底做成了没？"—— P0，必须自动化、全覆盖。确定性评分器的主战场。

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| 结果正确性 | 代码比对、单元测试、数据库校验 | pass@1 / pass^k |
| 任务完成度 | 子目标打点 | 完成率 % |
| 指令遵循度 | JSON Schema 校验、正则、字段检查 | 遵循率 % |
| 工具调用正确性 | 调用日志断言 | 调用准确率 % |

#### 过程质量（Process Quality）

回答"即便做对了，过程合理吗？"—— 最能体现"智能"水平，Rubric 评委的主战场。

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| 推理合理性 | Rubric 评委 | 合理性评分 1–5 |
| 步骤最优性 | 比较实际步数 vs 参考解法 | 步数比 |
| 信息完整性 | Rubric 按要点核查 | 要点覆盖率 % |
| 自我纠错能力 | Trace 分析 | 纠错成功率 |

#### 效率与成本（Efficiency & Cost）

回答"结果对了，但划得来吗？"—— 被很多团队忽视但极其重要。一个 pass@1 高但 token 花 10 倍的方案，在生产上是不可接受的。

#### 鲁棒性与安全（Robustness & Safety）

回答"它会不会在关键时刻翻车？"—— P0，尤其是涉及金融、医疗、企业数据的 Agent，必须前置。

#### 体验与对齐（Experience & Alignment）

回答"用户愿意继续用它吗？"—— P2，但决定产品生死。早期用 Rubric + 人工抽样，成熟后引入线上 A/B + 用户反馈闭环。

### 2.3 不同类型 Agent 的测评侧重

| Agent / Skill 类型 | 测评侧重点 | 典型检查项 |
|-------------------|-----------|-----------|
| 知识库问答 | 准确性、幻觉检测、引用溯源 | 回答是否基于知识库而非编造 |
| 代码编写 | 产物正确性、可运行性 | 代码是否能编译/运行通过 |
| 功能工具 | 过程合规性、工具调用正确性 | 是否按预期步骤调用了正确的工具 |
| 问题定位 | 推理链路、根因准确性 | 推理过程是否逻辑清晰 |

## 三、测评实施：怎么做？

### 3.1 设计测评用例集

用例集从 **触发 → 核心逻辑 → 产物质量 → 异常容错** 四个场景层层递进，每个场景包含正向和负向用例。

| 场景 | 关注点 | 正向用例 | 负向用例 |
|------|--------|---------|---------|
| 触发 | Skill 是否在正确场景被激活 | 各种合法表述都能触发 | 相似但不相关的 prompt 不误触发 |
| 核心逻辑 | 执行过程是否按预期步骤进行 | 工具调用序列、参数、顺序正确 | 缺少关键步骤、调用错误工具 |
| 产物质量 | 最终产物是否完整、准确、可用 | 响应内容准确、文件完整 | 内容缺失、格式错误、幻觉/编造 |
| 异常容错 | 异常输入/环境下能否合理处理 | 优雅降级、明确提示 | 崩溃、死循环、静默失败 |

> **为什么需要负向触发用例？** 只测正例，Agent 可能学会"什么都触发"——过度触发比不触发更难发现。

核心逻辑用例设计三步法：

1. **梳理分支流程图** —— 画出 Skill 的所有核心分支路径
2. **每条分支至少 1 个用例** —— 覆盖所有核心分支
3. **补充组合和边界** —— 分支间组合、阈值边缘、跨分支冲突

经验法则：核心逻辑用例的数量通常是其他三类场景用例之和的 **2–3 倍**。

### 3.2 设计评分规则

评分按 **负分制** 进行计算，初始总分 100 分，每有一个不符合预期的轨迹或结果扣除对应分数，扣至 0 分。达标标准为 **80 分**（推荐值）。

| 评分维度 | 计分项 | 描述 |
|---------|--------|------|
| 结果 | 任务结果 | 任务结果和预期不符合，扣除 100 分 |
| 过程 | 步骤遵循性 | 每有一步不符合预期，就扣 10 分 |
| 过程 | 步骤输出结果 | 每有一步不符合预期，就扣 10 分 |
| 过程 | 调用工具/知识库 | 每有一项不符合预期，就扣 10 分 |
| 效率 | 分析耗时 | 基准时长 5 分钟，每多 1 分钟扣 10 分 |
| 稳定性 | 一致性 | 对同一个任务进行 N 次试验，若均达标则取平均分 |

### 3.3 建立用例基线

用例基线是单个用例执行 1 次后，经人工确认的**预期过程**和**预期结果**的快照。

**预期过程**：思维链（CoT）、工具调用序列、中间产物、完整 Trace

**预期结果**：最终响应、输出文件、输出报告

基线建立流程：先跑出来 → 再确认 → 确认后就是预期。**用例设计阶段只需定义 prompt 和检查规则，不需要手写预期过程和结果。**

基线在评分中同时服务于确定性评分器（可程序化的客观判定）和 Rubric 评分器（作为参考答案进行语义对比）。

### 3.4 执行测评

用例设计完毕、评分规则就位、基线确认通过后，正式执行测评。核心是将准备工作串联成可自动化运行的流水线：加载用例 → 逐个执行 → 评分 → 汇总报告。

| 触发场景 | 触发方式 | 说明 |
|---------|---------|------|
| Agent/Skill 提示词变更 | 自动触发 | PR 合入前执行回归测评 |
| 模型版本升级 | 手动触发 | 验证新模型对现有 skill 的兼容性 |
| 定期巡检 | 定时触发 | 周期性全量回归，发现潜在退化 |

### 3.5 用例集维护

测评体系不是"搭完就完"的一次性工程。必须区分两种套件：

| 类型 | 核心问题 | 期望通过率 | 维护频率 |
|------|---------|-----------|---------|
| 能力测评（Capability） | "Agent 能把什么做好？" | 起步 20–50%，逐步提升 | 主动拓展、频繁迭代 |
| 回归测评（Regression） | "原有能力是否还在？" | 接近 100% | 只增不减，CI 每次运行 |

**能力测评通过率稳定 = 100% 后，把用例"毕业"到回归套件，从此每次 CI 都跑。**

## 四、工程落地：如何自动化？

### 4.1 前置依赖：Trace 输出能力

过程评测的前提是**能拿到结构化的执行轨迹**。如果被测 Agent 只输出最终回答，没有可解析的中间过程，那么过程评测就无从下手。

对被测 Agent/Skill 的要求：

- **输出结构化 Trace**：以可解析的格式（如 JSONL、JSON）输出，包含工具调用、思维链、中间产物等
- **Trace 内容完整**：至少包含每步的工具名称、入参、返回值、时间戳
- **Trace 格式稳定**：字段命名和结构在版本间保持一致

> 建议：在 Agent/Skill 设计阶段就将结构化 Trace 输出作为标准能力纳入，而非事后补救。

### 4.2 环境隔离

每次测评在隔离环境中执行，避免状态污染：Clone 仓库到临时目录，每个用例执行前重置环境，测评产物统一归档。

### 4.3 稳定性评估

通过多轮执行（N 次）检测 AI 的幻觉和非确定性。核心原则：**只要 N 次中有 1 次不通过，就说明该用例存在稳定性风险**。

| Agent/Skill 使用类型 | 容忍阈值 | 说明 |
|---------------------|---------|------|
| 关键决策类 | 0%（N/N 全部通过） | 任何一次失败都可能导致误判 |
| 辅助分析类 | ≤ 10% | 偶发偏差可接受，但需持续观察 |
| 创意生成类 | ≤ 40% | 输出多样性本身是特性 |

> 调试技巧：0% pass^N 通常说明任务定义有问题，而不是 Agent 能力不行。先检查用例描述是否有歧义、评分器是否配置错误。

### 4.4 测评报告

报告应包含：全局概览（通过率、平均分、模型版本、Token 消耗、总费用）、用例列表（按任务分组）、单用例详情（Prompt、多 Trial 聚合、稳定性评分）、执行记录（逐 Trial 详情、步骤评分明细）。

## 五、实战案例：TPerf 性能 AI 分析 Agent

### 5.1 Agent 概述

TPerf AI 分析 Agent 属于「功能工具」类型，通过 MCP 工具调用 TPerf-API 获取各维度性能数据，结合知识库检索进行深度分析，经决策树判断资源瓶颈点和压测有效性，输出结构化性能分析报告。

测评侧重三个方面：**操作步骤合规性 + 执行效率 + 输出报告质量**。

### 5.2 评分标准

采用 **确定性评分 + 模型评分** 两层组合，满分 100 分扣分制。

**确定性评测维度**：

| 维度 | 关注点 | 最大扣分 | 评分方法 |
|------|--------|---------|---------|
| 操作步骤 | 工具调用流程是否与基线一致 | -10 | LCS 算法对齐步骤，不一致/多余/缺少/顺序错误各扣 2 分 |
| 效率 | 分析耗时和 Token 消耗是否合理 | -10 | 耗时以 300s 为基准每超标 10% 扣 1 分；Token 同理 |
| 稳定性 | 同一任务多次分析，结论是否一致 | — | 并发 N 次执行，全部达标取平均分，任一不达标则 0 分 |

**模型评分维度**：

| 维度 | 关注点 | 最大扣分 | 评分方法 |
|------|--------|---------|---------|
| 关键判定 | 压测有效性和性能结果判定是否与基线一致 | -80 | 任一项不一致直接扣 80 分 |
| 章节内容 | CPU/内存/网络/磁盘IO/业务指标等 | 每项 -5，上限 -80 | 大模型逐项对比基线报告与 trial 报告 |

综合评分公式：

```
单次 Trial:  trial_score = max(0, 100 + 步骤扣分 + 效率扣分 + 报告扣分)
达标判定:    trial_pass  = (trial_score ≥ T)，T = 80
用例总分:    case_score  = avg(所有 trial_score) 若全部 trial_pass = true
                         = 0                      若任一 trial_pass = false
```

### 5.3 用例集设计

按性能瓶颈场景分组，覆盖 9 大类共 30+ 个用例，每个用例对应一个真实的 TPerf 压测记录，支持多模型对比评测（7 个模型）。

| 场景分组 | 用例数 | 典型场景 |
|---------|--------|---------|
| CPU 瓶颈 | 6 | 中断绑核异常、RSS 哈希配置异常、大象流 |
| 网卡瓶颈 | 5 | RSS 队列收包不均、大象流、网卡硬件丢包 |
| 磁盘瓶颈 | 4 | 磁盘繁忙 IO Util > 80%、await > 10ms |
| 磁盘 IO | 2 | 磁盘 IO 高负载 + iowait 高 |
| 内存 | 1 | 内存占用率 > 85% |
| TCP 队列 | 3 | Accept 队列溢出、CPU 高负载导致 accept 不及时 |
| Nginx | 1 | Nginx 代码问题导致核间不均衡 |
| 配置问题 | 1 | 压测配置问题导致压力不足 |
| 正常场景 | 5 | 全核高负载但无异常、RSS 均衡、无丢包等 |

### 5.4 工程实现

**Trace 获取**：通过 Knot API 查询获取完整的 AG-UI 事件流，包含工具调用事件、文本消息事件、时间和消耗、最终分析报告。

**框架结构**：项目按职责分层组织——API 封装层、配置层、用例定义层、采集层、评分层、执行层、工具层、解析层、结果层。

**流水线集成**：两条流水线——基线确认流水线（日常开发调试）和测评流水线（pytest 实现全部测评逻辑，用例管理复用 TCase 平台）。

**执行模式**：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 快速分析 | 每用例跑 1 轮 | 日常开发调试、确定基线 |
| 全量测评 | 每用例跑 5 轮 × 7 模型 | 版本验收、模型对比 |
| 按模型筛选 | 只运行指定模型 | 单模型验证 |
| 重新评分 | 对已有结果重新评分 | 评分逻辑调整后 |

### 5.5 模型对比

全量测评后可横向对比多个模型的综合表现，帮助团队做出模型选型决策。关键指标包括 pass@1、pass^5、用例通过率、得分、Token 使用量、开销、耗时。

## 六、关键启示

总结这篇文章的核心方法论：

1. **三类评委组合是必须的** —— 没有银弹评分器，确定性 > Rubric > 人工，优先级不能反
2. **五个维度缺一不可** —— P0 先行（功能正确性 + 鲁棒性安全），P1 跟进（过程质量 + 效率成本），P2 按需（体验对齐）
3. **基线是测评的锚点** —— 先跑出来再确认，确认后就是预期，后续每次执行都参考它来评分
4. **能力测评和回归测评是两套逻辑** —— 能力测评指导优化方向，回归测评防御漂移，通过率 100% 后"毕业"
5. **稳定性评估不可省略** —— pass^k 比 pass@1 更重要，关键决策类 Agent 容忍阈值为 0%
6. **Trace 输出是前提** —— 没有结构化执行轨迹，过程评测无从下手，设计阶段就应纳入
7. **工程化才能持续** —— 嵌入 CI 流水线、自动触发、环境隔离、报告归档，让测评成为研发流程的一部分
