# Agent 风向：从堆 Token 到算 ROI

> 2024 年大家比谁的 Agent 能调更多工具、吃更多 Token；2025 年风向变了——CIO 开始问"这东西到底替我省了多少钱"，创业公司的 pitch deck 里出现了 ROI 曲线，Anthropic 发了 Agent 最大化的经济学论文。Agent 的竞争维度正在从"能力天花板"转向"单位成本产出"，这不是退步，是行业走向成熟的必经之路。

---

## 一、风向变了：三个信号

### 1.1 CIO 的问题变了

一年前 CIO 问的是："Agent 能做什么？"

现在问的是："Agent 做这件事，比人做便宜多少？比不用 Agent 快多少？出错率多少？出错成本多少？"

这不是个别的挑剔，而是规模化部署的自然门槛。Demo 阶段不谈成本，生产阶段必须谈。任何一个要进采购审批流程的技术，最终都要回答同一个问题：**投入产出比**。

### 1.2 创业公司的叙事变了

2024 年的 Agent 创业公司 pitch："我们支持 50+ 工具调用、10 万 Token 上下文、多轮自主决策。"

2025 年的 pitch："我们的 Agent 用 1/3 的 Token 完成同等任务，单次调用成本 $0.02，人效提升 4 倍，ROI 回收周期 3 个月。"

叙事变化的背后是客户成熟度在变。早期客户为"能做"买单，成熟客户为"划算"买单。

### 1.3 头部厂商的论文变了

Anthropic 在 2025 年发布了《How to Build an Agent that Actually Works》，核心不是调参技巧，而是**Agent 最大化的经济学分析**——什么时候该用 Agent，什么时候不该用，以及如何计算 Agent 的边际收益。

OpenAI 的 Realtime Agent、Google 的 A2A 协议，都在往"可计量、可调度、可编排"的方向走，而不是"更大上下文、更多工具"。

## 二、堆 Token 时代：我们是怎么走到这一步的

### 2.1 Token 是 Agent 的燃料，也是 Agent 的负债

Agent 的运行逻辑：接收任务 → 拆解子目标 → 逐个调用工具 → 汇总结果 → 返回答案。每一步都需要 LLM 推理，每一步都在烧 Token。

一个典型的 Agent 任务链路：

| 步骤 | Token 消耗 | 说明 |
|------|-----------|------|
| 任务理解 + 拆解 | 2K–5K | 理解用户意图，规划执行路径 |
| 工具选择 + 参数填充 | 1K–3K / 次 | 每次工具调用前的推理 |
| 结果解析 + 状态更新 | 1K–2K / 次 | 解析工具返回，更新上下文 |
| 错误恢复 + 重试 | 2K–5K / 次 | 出错后的诊断和重规划 |
| 最终汇总 + 格式化 | 2K–5K | 生成最终回答 |

一个 5 步工具调用的任务，轻松烧掉 20K–50K Token。如果中途出错重试 2 次，直接翻倍。

### 2.2 "Token 越多越强"的幻觉

早期 Agent 开发中有一个普遍的误区：**Token 消耗越多 = Agent 越努力 = 结果越好**。

这就像加班文化——工时长不等于产出高。Agent 的 Token 消耗和任务完成质量之间不是线性关系，而是一条先升后平的曲线：

- **Token 太少**：推理不充分，工具选择错误，任务失败
- **Token 刚好**：推理充分，路径最优，任务成功
- **Token 太多**：冗余推理、无效工具调用、自我重复、最终结果并没有更好

关键在于找到"刚好"的那个点，而不是无脑堆上去。

### 2.3 成本爆炸的真实案例

一个做客服 Agent 的团队分享过他们的数据：

| 指标 | V1（堆 Token） | V2（算 ROI） |
|------|---------------|-------------|
| 单次对话平均 Token | 12,000 | 4,200 |
| 工具调用次数 | 6.8 | 2.3 |
| 任务完成率 | 78% | 82% |
| 单次对话成本 | $0.18 | $0.06 |
| 日处理量（同预算） | 5,500 | 16,700 |

V1 用了 3 倍的 Token，完成率反而更低。原因很简单：过多的工具调用引入了更多出错机会，冗余推理增加了幻觉概率，长上下文导致注意力稀释。

**成本不是省出来的，是设计出来的。**

## 三、算 ROI 时代：Agent 经济学的核心框架

### 3.1 Agent ROI 的定义

ROI（Return on Investment）在 Agent 语境下的定义：

**Agent ROI = （人工完成成本 - Agent 完成成本） / Agent 完成成本**

其中：

- **人工完成成本** = 人工时长 × 时薪 + 出错率 × 出错修复成本
- **Agent 完成成本** = Token 消耗 × 单价 + 工具调用成本 + 出错率 × 出错修复成本 + 人工审核成本

关键洞察：**Agent 的成本不只是 API 调用费**。工具调用有成本（API 费用、计算资源），出错有成本（修复、回滚、客户投诉），人工审核有成本（Review 时间）。只看 Token 单价是严重低估。

### 3.2 四种成本，四种优化方向

| 成本类型 | 占比（典型场景） | 优化方向 |
|---------|----------------|---------|
| LLM 推理成本（Token） | 30–50% | 模型路由、Prompt 压缩、缓存复用 |
| 工具调用成本 | 20–40% | 减少无效调用、批量合并、结果缓存 |
| 错误修复成本 | 10–30% | 提高首次成功率、早期校验、优雅降级 |
| 人工审核成本 | 5–15% | 自动化验证、置信度过滤、分级审核 |

**最大的杠杆不在最显眼的地方**。很多人只盯着 Token 单价优化，但工具调用成本和错误修复成本往往才是大头。

### 3.3 Agent 经济学的三条定律

**定律一：边际收益递减**

Agent 每增加一个工具、每多一轮推理，带来的增量收益在递减，但增量成本在递增。存在一个最优停止点，超过它就是在亏钱。

```
边际收益（新增工具/推理轮次）> 边际成本 → 继续
边际收益 = 边际成本 → 停止，这是最优解
边际收益 < 边际成本 → 亏损，必须停止
```

**定律二：可靠性即经济性**

Agent 的出错率对 ROI 的影响是指数级的，不是线性的。因为出错不只是"重做一次"的成本——还有排查时间、客户信任损失、数据污染修复。

| 首次成功率 | 等效成本倍数 | 说明 |
|-----------|------------|------|
| 95% | 1.05x | 偶尔重试，几乎无感 |
| 90% | 1.11x | 每 10 次重试 1 次，可接受 |
| 80% | 1.25x | 每 5 次重试 1 次，开始肉疼 |
| 70% | 1.43x | 每 3 次重试 1 次，不可接受 |
| 50% | 2.00x | 一半要重做，ROI 直接归零 |

**把首次成功率从 80% 提到 90%，比把 Token 单价砍一半对 ROI 的贡献更大。**

**定律三：简单任务不该用复杂 Agent**

不是所有任务都值得上 Agent。Anthropic 的论文给出了明确的决策框架：

| 任务特征 | 推荐方案 | 原因 |
|---------|---------|------|
| 单步、确定性高 | 直接 API 调用 | 无需推理，Agent 是浪费 |
| 少步、路径固定 | 编排工作流（Workflow） | 路径已知，不需要自主决策 |
| 多步、路径不确定 | Agent | 需要动态规划，Agent 有价值 |
| 多步、跨系统、需判断 | Agent + 人工兜底 | 高价值 + 高风险，人机协作 |

**Agent 是"路径不确定时"的工具，不是"所有场景"的默认选择。** 用 Agent 做一个单步 API 调用能完成的事，就像用直升机去买牛奶。

## 四、从堆 Token 到算 ROI：五个实操策略

### 4.1 模型路由：不同任务用不同模型

不是每一步都需要 GPT-4o / Claude Opus。一个 Agent 的执行链路中，任务理解和最终汇总需要强模型，工具参数填充和格式化用弱模型就够了。

| 阶段 | 推荐模型级别 | Token 占比 |
|------|------------|-----------|
| 任务理解 + 拆解 | 强（Opus / GPT-4o） | 15% |
| 工具选择 + 参数填充 | 中（Sonnet / GPT-4o-mini） | 40% |
| 结果解析 + 状态更新 | 中（Sonnet / GPT-4o-mini） | 30% |
| 最终汇总 + 格式化 | 强（Opus / GPT-4o） | 15% |

效果：同等任务完成率下，**Token 成本降低 40–60%**。

这就是"法拉利问题"的 Agent 版——用最贵的模型做最简单的步骤，是最大的浪费。

### 4.2 Prompt 压缩与缓存

Agent 的上下文中有大量重复内容：系统指令、工具描述、历史对话摘要。这些不需要每次都完整传入。

- **系统 Prompt 缓存**：Anthropic 和 OpenAI 都支持 Prompt Caching，重复前缀只计费一次，成本降低 50–80%
- **工具描述按需加载**：不把 50 个工具的描述全塞进上下文，只传本次任务可能用到的 5–10 个
- **历史摘要替代原文**：超过 10 轮的对话历史，用摘要替代逐轮原文

### 4.3 减少无效工具调用

无效调用是 Token 的最大杀手。常见原因：

- **工具描述不清晰**，Agent 选错工具，发现不对再换，白白浪费 2 轮
- **缺少前置校验**，参数没填对就调用，API 报错，再重试
- **过度谨慎**，明明一步能完成的事，Agent 调了 3 个工具"确认"

优化手段：

1. **工具描述要精确**：写清楚适用场景、输入约束、输出格式，减少误选
2. **参数校验前置**：在调用工具前，先做 Schema 校验，不合法就不调
3. **设置调用预算**：单次任务最多 N 次工具调用，超过就停止并报告
4. **结果缓存**：相同参数的工具调用，短时间内的结果直接复用

### 4.4 提高首次成功率

根据定律二，可靠性即经济性。提高首次成功率是 ROI 最大的杠杆：

- **Few-shot 示例**：在 Prompt 中加入 2–3 个典型执行路径示例，Agent 的路径选择准确率从 60% 提升到 85%+
- **分步校验**：每完成一个子目标就校验结果，而不是全做完才发现第一步就错了
- **工具 Mock 测试**：上线前用 Mock 数据跑一遍完整链路，确保路径正确
- **置信度门槛**：Agent 对下一步不确定时，主动问用户而不是猜

### 4.5 建立成本仪表盘

你无法优化你看不见的东西。Agent 的成本必须是可观测的：

| 指标 | 采集方式 | 作用 |
|------|---------|------|
| 单任务 Token 消耗 | Trace 日志 | 发现异常消耗 |
| 工具调用次数 & 成功率 | Trace 日志 | 定位无效调用 |
| 首次成功率 | 多次运行统计 | 衡量可靠性 |
| 单任务总成本 | Token × 单价 + 工具成本 | 计算 ROI |
| 人均日处理量 | 业务统计 | 衡量人效提升 |

建议在 Agent 的 Trace 输出中，每一步都记录 Token 消耗和工具调用详情，这样事后可以按任务类型、模型版本、时间维度做成本分析。

## 五、Agent 的三种经济模型

不同的 Agent 部署场景，ROI 的计算方式完全不同：

### 5.1 替代型 Agent

**目标：替代人工完成重复性任务**

ROI = （人工成本 - Agent 成本） / Agent 成本

关键指标：首次成功率、单任务成本、日处理量

典型场景：客服、数据录入、文档生成、代码 Review

> 这类 Agent 的 ROI 最容易算，也最容易量化。但要注意：替代型 Agent 的出错成本不只是重做，还有客户流失风险。首次成功率低于 90% 就不该上线。

### 5.2 增强型 Agent

**目标：让人 + Agent 的组合比单独人更强**

ROI = （人机组合产出 - 纯人工产出） / Agent 成本

关键指标：任务完成质量提升、决策准确率提升、时间节省

典型场景：研发辅助、投资分析、医疗诊断辅助、法律文书审查

> 这类 Agent 不替代人，而是放大人的能力。ROI 的核心不是"省了多少"，而是"多做了什么"——原来做不了的分析深度、原来覆盖不了的场景数量。

### 5.3 探索型 Agent

**目标：发现人没想到的方案或机会**

ROI = （发现价值 - 探索成本） / 探索成本

关键指标：新发现数量、发现转化率、探索效率

典型场景：科研辅助、市场机会发现、漏洞挖掘、创意生成

> 这类 Agent 的 ROI 最难算，因为"发现"的价值高度不确定。但可以换一种思路：把探索成本控制在可接受的预算内（比如每天 $5），只要偶尔发现一个高价值机会就回本了。这是"期权思维"而非"债券思维"。

## 六、行业趋势：从能力竞赛到效率竞赛

### 6.1 模型厂商在做什么

| 厂商 | 动作 | 信号 |
|------|------|------|
| Anthropic | Prompt Caching、模型路由建议、Agent 经济学论文 | 从"卖更多 Token"到"帮客户用更少 Token" |
| OpenAI | Realtime Agent、Structured Output、GPT-4o-mini | 低成本高速度成为卖点 |
| Google | Gemini Flash、A2A 协议 | Agent 间协作效率优先于单 Agent 能力 |
| 开源 | Llama 3、Qwen 2.5、DeepSeek | 本地部署零 API 成本，ROI 天然高 |

所有厂商都在往"更便宜、更快、更可控"的方向走，而不是"更大上下文、更多参数"。

### 6.2 企业在做什么

从"能不能做"到"划不划算"的转变正在发生：

- **金融行业**：Agent 必须通过合规审计，出错率要求 < 0.1%，成本要低于人工 + 合规流程
- **电商行业**：客服 Agent 的 ROI 计算精确到每个对话，A/B 测试对比人工和 Agent 的转化率
- **研发团队**：Coding Agent 的价值不在于"写了多少行代码"，而在于"减少了多少 Review 轮次"和"缩短了多少交付周期"
- **SaaS 公司**：Agent 功能的定价从"按调用次数"转向"按解决的问题数"，和客户对齐 ROI

### 6.3 什么会死，什么会活

| 会死 | 会活 |
|------|------|
| 靠堆 Token 炫技的 Demo | 有成本仪表盘的生产 Agent |
| 万能 Swiss Army Knife Agent | 专精单一场景的高 ROI Agent |
| 不计成本的"自主决策" | 有预算和兜底的人机协作 |
| 只看 pass@1 的评测 | 看 pass^k + Token + 延迟的综合评测 |
| 按 Token 定价 | 按价值定价 |

## 七、给不同角色的建议

### 给 Agent 开发者

1. **先算账再写代码**：在开发前就估算单任务 Token 消耗和工具调用次数，如果算出来比人工贵，换方案
2. **模型路由是必修课**：不是每个步骤都需要最贵的模型，学会分级使用
3. **Trace 是成本优化的前提**：没有结构化执行日志，你根本不知道 Token 花在哪了
4. **首次成功率 > Token 单价**：砍 Token 单价省的是小钱，提高首次成功率省的是大钱

### 给技术管理者

1. **把 Agent 成本纳入预算**：不是"API 费用"，而是"总拥有成本"——包括开发、运维、出错修复、人工审核
2. **建立 ROI 仪表盘**：每个 Agent 都应该有成本、成功率、人效提升的实时看板
3. **不要用 Agent 解决不该用 Agent 的问题**：单步 API 调用能解决的事，不要上 Agent
4. **给 Agent 设预算上限**：单任务 Token 上限、日调用预算、月度总预算，超了就熔断

### 给创业者

1. **ROI 是护城河**：在 Agent 同质化严重的市场，成本效率是最可持续的差异化
2. **按价值定价而非按 Token 定价**：客户不关心你烧了多少 Token，关心你解决了什么问题
3. **从替代型场景切入**：ROI 最容易算、最容易说服客户买单
4. **成本结构决定商业模式**：如果你的 Agent 单任务成本是 $0.5，你只能服务愿意付 $5+ 的客户；如果是 $0.01，你可以服务长尾市场

---

## 八、结语

从堆 Token 到算 ROI，不是 Agent 的退步，而是行业的成人礼。

2024 年我们在探索 Agent 的能力边界——"它能不能做到？"
2025 年我们在探索 Agent 的经济边界——"它做到划不划算？"

这两个问题同样重要，但第二个问题决定了 Agent 能不能从 Demo 走向生产，从尝鲜走向标配。

**最好的 Agent 不是最聪明的 Agent，而是最划算的 Agent。**
