# Agent 工程是什么，怎么落地：从"能跑通一次"到"能跑生产"

> 这两年很多团队都有同一个困惑：Demo 跑得很惊艳，一上生产就翻车。Agent 会幻觉、会死循环、会乱调工具、会把简单的事拆成十步还做不对。问题不在模型不够强，而在大家把 Agent 当"聊天机器人 + 函数调用"来用，没把它当成一个需要工程化的系统来对待。这篇文章想讲清楚两件事：Agent 工程到底是什么，以及一个 Agent 从 Demo 到生产之间，到底要补哪些工程能力。

---

## 一、为什么需要一个叫"Agent 工程"的东西

### 1.1 LLM 工程和 Agent 工程不是一回事

过去两年大家都在做 LLM 工程：Prompt 调优、RAG 检索、模型路由、上下文压缩。这套打法的核心假设是 **单轮调用、确定性输出**——问一句答一句，错了重问就行。

Agent 不一样。Agent 是一个**会自己决定下一步做什么**的系统：它接收一个目标，自己拆任务、选工具、读结果、判断要不要重试，循环往复直到完成或放弃。这带来三个质变：

| 维度 | LLM 工程的假设 | Agent 工程的现实 |
|------|--------------|----------------|
| 调用次数 | 单轮或少数几轮 | 几十到上百轮 |
| 路径 | 预先编排好的固定路径 | 运行时动态决定 |
| 失败模式 | 答错了重问 | 中途任何一步崩了，整条链作废 |
| 成本 | 一次 Token | 累积 Token + 工具调用 + 时间 |
| 观测 | 看一次输出 | 要看整条执行轨迹 |

LLM 工程优化的是"单次调用质量"，Agent 工程优化的是"一条长链路的可靠性"。前者像优化一个函数，后者像运维一个分布式系统。**两者的失败模式、成本结构、可观测性要求完全不同，套用同一套打法必然翻车。**

### 1.2 Agent 翻车的典型场景

先看几个最常见的生产翻车点，后面所有的工程手段都是冲着这些点去的：

1. **幻觉级联**：第一步 Agent 猜了一个不存在的 API 字段，后面所有步骤都基于这个错字段往下走，错得越来越离谱。
2. **工具选择漂移**：明明该调工具 A，Agent 因为上下文里某句话误导，去调了 B，B 返回不对，Agent 又去调 C，越走越远。
3. **死循环烧钱**：某一步一直失败，Agent 不会换策略，只会用同一套参数重试，Token 烧光预算还没收敛。
4. **上下文爆炸**：每轮把工具返回的全量结果塞回上下文，跑 20 轮之后上下文几十万 Token，注意力稀释，延迟和成本双双爆炸。
5. **静默失败**：Agent 输出了一个看起来合理但完全错误的结果，没有报错、没有兜底，人如果不 Review 就直接流到下游业务。

这五种翻车，没有一种是"模型再强一点"能解决的。它们都是**工程问题**——需要边界设计、状态管理、校验、熔断、可观测性。这就是为什么需要"Agent 工程"这个独立的工程学科。

### 1.3 Agent 工程的定义

给个能用的定义：

> **Agent 工程是把 LLM 作为核心推理引擎，围绕它构建任务编排、工具治理、状态管理、错误控制、可观测性和成本治理的一整套工程实践，使一个自主决策系统能够在长链路、高不确定性、高累积成本下保持可靠运行。**

拆开看四个关键词：

- **长链路**：不是单次调用，是几十轮的执行
- **高不确定性**：路径运行时决定，不能预先编排
- **高累积成本**：每一步都花钱，错一步赔一串
- **可靠运行**：要可度量、可干预、可兜底

这四个词决定了 Agent 工程的所有技术选型。**它不是"把 LLM 用得更花"，是"让一个会自己决策的系统跑得稳"。**

## 二、Agent 的工程解剖：六个子系统

一个生产级 Agent 不是一个大 Prompt，而是六个子系统组合而成。把它们拆开看，每个都有自己的工程问题。

### 2.1 任务规划层

负责把用户的高层目标拆成可执行的子任务序列。这是 Agent 和 Workflow 的分水岭：Workflow 的步骤是人预先写死的，Agent 的步骤是模型运行时规划的。

| 规划策略 | 适合场景 | 工程风险 |
|---------|---------|---------|
| 一次性规划 | 任务结构稳定 | 计划脱离实际，执行时发现走不通 |
| 边走边规划 | 任务高不确定 | 规划本身烧 Token，可能反复横跳 |
| 模板化规划 | 同类任务高频复用 | 模板覆盖不全，遇到边界就崩 |
| 分层规划 | 复杂跨域任务 | 层间上下文丢失，子目标和总目标脱节 |

工程上的关键决策：**规划要不要可持久化**。一次性规划把计划存在内存里，崩了就没了；持久化规划把计划写进数据库，每完成一步更新状态，崩了能恢复、能审计、能让人介入修改。生产 Agent 几乎都该选后者。

### 2.2 工具治理层

Agent 调用的工具不是"越多越好"。工具数量和选择准确率是反向关系——上下文里塞 50 个工具描述，Agent 选错的概率比塞 10 个高一倍。

工具治理要解决四件事：

1. **工具发现**：根据当前任务动态选 5–10 个相关工具进上下文，而不是全量塞
2. **参数校验**：调用前用 Schema 校验参数，不合法直接拦截，不让无效调用打到下游
3. **结果解析**：工具返回的原始数据往往很大，要先压缩/抽取再喂回 Agent
4. **权限分级**：只读工具可以放开，写操作工具必须有人工确认或预算闸门

| 工具类型 | 失败成本 | 治理强度 |
|---------|---------|---------|
| 只读查询 | 低 | 放开 |
| 写入操作 | 中 | 预算 + 日志 |
| 不可逆操作（删库、发款） | 高 | 人工双确认 |
| 外部副作用（发邮件、推送） | 高 | 审批 + 限频 |

**工具治理的核心理念：让 Agent 能调的工具有边界，让它调错时损失有上限。** 一个能调任意工具的 Agent 不是强大，是危险。

### 2.3 状态与上下文管理

这是 Agent 工程里最容易被忽略、却最容易翻车的一层。

LLM 是无状态的，但 Agent 必须有状态——它要知道自己走到哪一步、前面调了什么、结果是什么、还剩什么没做。状态管理的工程问题有三个：

**第一，上下文窗口管理。** 一个跑 30 轮的 Agent，如果把每轮的工具返回都原文塞回上下文，很快爆掉。必须做摘要、截断、外部存储的组合策略：

| 上下文内容 | 处理方式 |
|----------|---------|
| 系统指令 | 全量保留，走 Prompt 缓存 |
| 当前任务描述 | 全量保留 |
| 最近 3–5 轮工具返回 | 原文保留 |
| 更早的工具返回 | 摘要后替换 |
| 大块原始数据 | 存外部，上下文只放指针 |

**第二，状态持久化。** 中途崩了能不能从断点续跑？这要求每一步的输入、输出、决策都落库。崩溃恢复不是奢侈品，是长链路 Agent 的必需品——一个跑 20 分钟的任务崩在第 18 分钟，不能让用户从头再来。

**第三，多 Agent 间的状态共享。** 如果是多个 Agent 协作，它们之间的状态怎么传？传全文太重，传摘要会丢信息。这本质是个分布式状态同步问题，和微服务里的最终一致性是同一类难题。

### 2.4 错误控制层

Agent 比传统系统更难处理错误，因为它的错误是**语义级**的——不是抛异常，而是"看起来对了其实错了"。

错误控制要分三层设计：

- **前置校验**：调用工具前检查参数、权限、预算，把明显错误的调用拦在门外
- **中置校验**：每完成一个子目标，用另一个轻量模型或规则校验中间结果是否合理
- **后置校验**：最终输出前，用独立的评审 Agent 或人工抽查做最后把关

| 错误类型 | 检测方式 | 兜底动作 |
|---------|---------|---------|
| 工具调用失败 | 异常捕获 | 重试 / 换工具 / 报告 |
| 参数错误 | Schema 校验 | 让 Agent 重新生成参数 |
| 路径走偏 | 中间结果校验 | 回滚到上一个检查点 |
| 最终结果错 | 评审 Agent / 人工 | 标记不可信，走人工流程 |
| 死循环 | 步数计数器 | 强制停止，转人工 |

**没有错误控制的 Agent 不能上生产。** 它不是"不够智能"，是"不可信"。一个偶尔输出错答案但有兜底的 Agent，比一个经常输出对答案但无法兜底的 Agent，更适合生产。

### 2.5 成本治理层

Agent 是累积成本系统——每一步都在烧钱，错一步赔一串。成本治理不是"省 Token"，是"让钱花在刀刃上"。

| 成本杠杆 | 效果 | 实施难度 |
|---------|------|---------|
| 模型分级路由 | 省 40–60% | 中 |
| Prompt 缓存 | 省 50–80%（重复前缀） | 低 |
| 工具结果缓存 | 省 20–40%（重复调用） | 中 |
| 步数预算 | 防失控 | 低 |
| 上下文压缩 | 省 30–50%（长链路） | 高 |
| 失败熔断 | 防异常烧钱 | 中 |

成本治理的第一原则：**单任务成本必须有上限且可观测**。一个没有预算闸门的 Agent，迟早会因为某个异常任务烧掉一天的预算。熔断不是省钱，是保命。

### 2.6 可观测性层

传统系统的监控看的是 QPS、延迟、错误率。Agent 系统要看的是**执行轨迹**——这一步想了什么、调了什么、返回了什么、为什么这么决策。

可观测性的核心是结构化的 Trace 日志，每一步至少要记录：

| 字段 | 作用 |
|------|------|
| step_id / parent_id | 还原执行树 |
| 模型版本 | 复现 + 归因 |
| 输入 Prompt 摘要 | 排查误判 |
| 工具调用 + 参数 | 定位无效调用 |
| 工具返回摘要 | 排查解析问题 |
| Token 消耗 | 成本归因 |
| 耗时 | 延迟归因 |
| 中间校验结果 | 路径走偏告警 |

**没有 Trace 的 Agent 是黑箱，黑箱不能上生产。** 排查一个线上 Agent 故障，80% 的时间花在"它到底干了什么"上，Trace 解决的就是这个问题。

## 三、落地路线：从 Demo 到生产的四个阶段

讲完系统结构，讲怎么一步步落地。见过太多团队想一步到位建"全功能 Agent 平台"，结果半年没产出。正确的姿势是分阶段，每阶段都交付能用的东西。

### 3.1 阶段一：单 Agent + 固定工具集（2–4 周）

目标：让一个 Agent 在固定场景里稳定完成单类任务。

| 要建的能力 | 具体动作 |
|----------|---------|
| 工具集 | 选 5–10 个高频工具，写清楚描述和 Schema |
| Prompt | 系统指令 + 3–5 个 Few-shot 示例 |
| 状态 | 内存态即可，先不持久化 |
| 错误控制 | 工具异常重试 + 步数上限 |
| 观测 | 结构化日志，能看到每步调了什么 |
| 成本 | 步数预算 + Token 计数 |

**这个阶段不要做的事**：不要上多 Agent 协作，不要上动态工具发现，不要上复杂的状态持久化。先把单场景跑稳，让 Agent 在 90% 的同类任务上能完成。

判断能不能进下一阶段：**连续一周、该场景任务首次成功率 ≥ 85%**。低于这个数说明连场景边界都没摸清，加复杂度只会更乱。

### 3.2 阶段二：状态持久化 + 错误兜底（2–4 周）

目标：让 Agent 能跑长任务，崩了能恢复，错了能兜住。

| 要建的能力 | 具体动作 |
|----------|---------|
| 状态持久化 | 每步决策和结果落库，支持断点续跑 |
| 中置校验 | 子目标完成后用轻量模型校验 |
| 兜底流程 | 校验失败 → 回滚 / 转人工 / 降级 |
| 上下文管理 | 历史轮次摘要，外部存储大块数据 |
| Trace | 完整执行轨迹可回放 |
| 成本仪表盘 | 单任务成本 + 异常告警 |

**这个阶段的核心是可靠性，不是能力扩展。** 很多人会忍不住加新工具、加新场景，忍住。先把第一阶段的场景做到 95%+ 成功率、单任务成本可控、崩了能恢复，再谈扩展。

判断能不能进下一阶段：**首次成功率 ≥ 92%，单任务 P95 成本稳定在预算内**。

### 3.3 阶段三：多场景 + 模型路由（4–6 周）

目标：把同一个 Agent 框架扩展到多个业务场景，并开始优化成本。

| 要建的能力 | 具体动作 |
|----------|---------|
| 场景路由 | 按任务类型分发到不同 Prompt + 工具集 |
| 模型分级 | 强模型做规划，中模型做执行，弱模型做校验 |
| 工具动态加载 | 按场景选相关工具子集进上下文 |
| 评审 Agent | 独立轻量模型做最终结果校验 |
| A/B 框架 | Prompt / 模型 / 工具集的灰度对比 |
| 成本治理 | Prompt 缓存 + 工具结果缓存 + 熔断 |

这个阶段才真正进入"工程化"——开始有版本管理、灰度发布、A/B 对比、成本优化。前面两个阶段是在让 Agent 跑通，这个阶段是在让 Agent 跑得又稳又省。

判断能不能进下一阶段：**多场景平均成功率 ≥ 90%，单任务成本比阶段一降 40%+**。

### 3.4 阶段四：多 Agent 协作 + 自治理（6+ 周）

目标：让多个 Agent 分工协作，处理跨域复杂任务。

| 要建的能力 | 具体动作 |
|----------|---------|
| Agent 间通信 | 共享上下文 + 任务交接协议 |
| 角色分工 | 规划 Agent / 执行 Agent / 评审 Agent |
| 编排层 | 决定哪个 Agent 接哪个子任务 |
| 状态同步 | 多 Agent 间的最终一致性 |
| 全局预算 | 跨 Agent 的总成本上限 |
| 全链路 Trace | 跨 Agent 的执行树串联 |

**这个阶段最容易过度设计。** 多 Agent 协作的复杂度是指数级上升的——两个 Agent 协作的调试成本不是单 Agent 的两倍，是三四倍。除非单 Agent 真的搞不定（任务跨多个系统、需要不同专家视角），不要轻易上多 Agent。大部分业务场景，一个 Agent + 好的工具集就够了。

### 3.5 阶段切换的红线

| 红线 | 含义 |
|------|------|
| 没量成功率就加复杂度 | 自我感动，不知道真不靠谱 |
| 成功率不到 85% 就上多场景 | 场景越多越烂，全是债 |
| 没有熔断就放开预算 | 一次异常烧掉一天预算 |
| 没 Trace 就上生产 | 出了故障只能干瞪眼 |
| 单 Agent 没跑稳就上多 Agent | 复杂度爆炸，永远调不完 |

**Agent 工程的落地节奏，慢就是快。** 每个阶段都把可靠性指标打实，比抢进度重要。Demo 看的是上限，生产看的是下限，工程的所有努力都在抬高下限。

## 四、技术选型的几个关键决策

### 4.1 自研还是用框架

市面上有 LangGraph、AutoGen、CrewAI、Anthropic SDK 等一堆 Agent 框架。选型逻辑很简单：

| 情况 | 推荐 |
|------|------|
| 团队没 Agent 经验，要快速验证 | 用框架，跑通 Demo |
| 场景简单、工具少、链路短 | 用框架，够用 |
| 场景复杂、要深度定制可观测性 | 自研核心 + 借用框架组件 |
| 要上生产、有严格合规要求 | 自研，框架当参考 |

框架的抽象往往假设"标准场景"，但生产场景几乎都不标准。框架最大的价值是**帮你快速理解 Agent 的结构**，而不是替你跑生产。见过太多团队被框架的抽象绑住，想加一个自定义的 Trace 都要绕三层封装。

**务实路线**：用框架跑通 Demo 学习结构，第二阶段开始自研核心循环，把框架当参考实现。

### 4.2 模型选型与路由

不是所有步骤都需要最贵的模型。生产 Agent 的模型选型原则：

| 步骤类型 | 模型级别 | 理由 |
|---------|---------|------|
| 任务理解 + 规划 | 强 | 推理质量决定整条链路 |
| 工具选择 + 参数填充 | 中 | 有 Schema 约束，容错高 |
| 结果解析 + 状态更新 | 中 | 结构化任务，中等模型够 |
| 中间校验 | 中或弱 | 只做合理性判断 |
| 最终汇总 + 格式化 | 强 | 决定输出质量 |
| 评审 | 独立于执行模型 | 避免同源偏见 |

**关键原则：执行链路和评审链路用不同模型**。同一个模型既执行又评审，容易对自己的输出有偏见，看不出自己的错。评审用同级别但不同厂商的模型，效果最好。

### 4.3 同步还是流式

Agent 的执行往往是长链路，一个任务跑几分钟很常见。用户体验和工程上都要考虑流式：

| 维度 | 同步 | 流式 |
|------|------|------|
| 用户体验 | 等几分钟没反馈，以为挂了 | 实时看到 Agent 在干什么 |
| 超时风险 | 长任务容易触发网关超时 | 心跳保活 |
| 调试 | 出错只能看最终结果 | 实时观察每一步 |
| 工程复杂度 | 低 | 中 |

**生产 Agent 默认该走流式**，哪怕只是把每一步的进度推给前端。这不只是体验问题，是工程问题——长任务没心跳，会被各种中间件当成死连接掐掉。

### 4.4 同步调用还是异步任务

| 任务时长 | 推荐 | 理由 |
|---------|------|------|
| < 30 秒 | 同步 | 简单，体验好 |
| 30 秒 – 5 分钟 | 流式同步 | 有反馈，不超时 |
| > 5 分钟 | 异步任务 + 回调 | 不能让用户干等 |

长任务一定要做成异步任务：用户提交 → 返回任务 ID → Agent 后台跑 → 完成后回调或推送。这跟传统后台任务没区别，别因为"这是 AI"就忽略工程常识。

---

## 五、团队和组织准备

### 5.1 需要什么角色

Agent 工程不是纯算法活，也不是纯后端活，它需要几种能力组合：

| 角色 | 核心职责 | 能力背景 |
|------|---------|---------|
| Agent 工程师 | 核心循环、状态管理、工具治理 | 后端 + 系统设计 |
| Prompt 工程师 | 系统指令、Few-shot、工具描述 | LLM 应用 + 领域知识 |
| 评测工程师 | 测试集、评分器、回归 | 测试 + 数据分析 |
| 平台工程师 | Trace、成本仪表盘、部署 | DevOps + 可观测性 |
| 领域专家 | 判断结果对错、定义边界 | 业务 |

小团队可以一人多角，但这五类能力都要有人覆盖。最常见的坑是**只招 Agent 工程师，没有评测和领域**——结果 Agent 跑起来了，但没人知道它到底准不准，准了也不知道为什么准。

### 5.2 评测体系是落地的前提

Agent 工程里最被低估的投入是评测。没有评测，所有的优化都是玄学。

评测体系至少要建三层：

| 层级 | 内容 | 频率 |
|------|------|------|
| 单元级 | 单个工具调用的参数正确率 | 每次提交 |
| 集成级 | 典型任务的端到端成功率 | 每天 |
| 回归级 | 历史失败案例的重放 | 每次发版 |

评分方式要组合用：

- **确定性评分器**：结果和期望完全匹配（适合结构化输出）
- **Rubric 评分器**：LLM 按评分标准打分（适合开放式输出）
- **人工评分**：领域专家抽样（适合判断类任务）

**没有评测体系的 Agent 优化，就是在黑箱里调旋钮。** 你不知道改了 Prompt 是变好还是变差，只能靠感觉。感觉不可靠，数据才可靠。

### 5.3 上线的门槛清单

给一个能直接用的上线 checklist：

- [ ] 典型任务首次成功率 ≥ 90%
- [ ] 单任务 P95 成本在预算内，有熔断
- [ ] 完整 Trace 可回放，能看到每步决策
- [ ] 崩溃可恢复，能从断点续跑
- [ ] 不可逆操作有人工双确认
- [ ] 评审 Agent 或人工兜底已就位
- [ ] 回归测试集通过率 ≥ 95%
- [ ] 成本仪表盘和告警已上线
- [ ] 降级方案已演练（Agent 挂了能切人工）

**没打勾就别上生产。** 这不是洁癖，是事故教训。每一条背后都是一类真实翻车。

## 六、常见误区

### 6.1 "模型够强就不需要工程"

这是最危险的误区。模型变强会抬高能力上限，但不会自动解决状态管理、成本治理、可观测性、错误兜底这些工程问题。一个更强的模型配上没有工程治理的框架，只会让翻车更隐蔽、更贵、更难排查。

**模型能力是上限，工程能力是下限。生产看下限。**

### 6.2 "Agent 就是带工具调用的 LLM"

工具调用只是 Agent 的一个能力，不是 Agent 的定义。一个真正的 Agent 系统还包括规划、状态、校验、兜底、观测。只做工具调用不做其他的，等于造了发动机没造刹车和仪表盘——能跑，但不敢上高速。

### 6.3 "先上生产再补工程"

Demo 思维。Agent 在生产环境的失败是累积的——第一次错可能只是用户体验差，第十次错就是预算烧光、数据污染、客户流失。工程债在 Agent 场景的利息特别高，因为每一步都在花真金白银。

**先补工程再上生产，顺序不能反。**

### 6.4 "Agent 能做所有事"

不能。Agent 适合"路径不确定、需要动态判断"的任务。路径确定的任务用 Workflow，单步能完成的任务用直接 API 调用。用 Agent 做一个单步 API 能完成的事，是用直升机去买牛奶——能买到，但贵且蠢。

**Agent 是"不确定路径"的工具，不是"所有场景"的默认选项。** 该用 Workflow 的地方别用 Agent，该用规则的地方别用 LLM。

## 七、结语：Agent 工程是新的软件工程

回到最初的问题：Agent 工程是什么？

它不是"用 LLM 写应用"，那是 LLM 应用开发。Agent 工程是**构建一个以 LLM 为推理核心的自主决策系统，并让它在长链路、高成本、高不确定性下可靠运行**。

这意味着它继承的不只是 LLM 工程的问题，更继承了分布式系统、运维、安全、成本管理、可观测性这一整套传统软件工程的遗产。一个成熟的 Agent 工程师，不仅要懂 Prompt 和模型，还要懂状态机、幂等性、熔断、Trace、预算控制——这些听起来一点也不"AI"的东西，恰恰是 Agent 能不能上生产的分水岭。

落地路径其实很朴素：先单场景跑稳，再补状态和兜底，再做多场景和成本优化，最后才谈多 Agent 协作。每一步都有明确的可靠性门槛，没达到就别往下走。**慢就是快，在 Agent 工程里尤其成立。**

一个朴素的判断标准：如果你公司的 Agent，Trace 能回放、崩了能恢复、成本有上限、错了能兜底、成功率有数据——那它已经比市面上 80% 的所谓"Agent 产品"工程化程度高了。Agent 工程不酷，它就是把软件工程的老道理，重新用在一个会自己决策的新系统上。

**Demo 比上限，生产比下限，工程就是抬高下限的那门手艺。**
