# OpenSpec 和 Superpowers 是什么，跟 OpenCode 有什么区别

> 2025–2026 年，AI 编程工具赛道出了三个现象级开源项目：OpenCode（180K stars）、Superpowers（240K stars）、OpenSpec（57K stars）。很多人把它们混为一谈："不都是让 AI 写代码的吗？"其实它们在架构中扮演的角色完全不同——一个管"在哪运行"，一个管"怎么干活"，一个管"先对齐再干"。本文厘清三个项目的定位、工作流和互补关系。

---

## 一、一句话定义

| 项目 | 一句话 | 类比 |
|------|--------|------|
| **OpenCode** | 一个开源的终端 AI 编程 Agent，是你能跑的**运行时** | 操作系统 |
| **Superpowers** | 一套技能方法论，教 Agent **怎么干活**才不会翻车 | 开发方法论（TDD + 敏捷） |
| **OpenSpec** | 轻量 Spec 框架，让人和 Agent 在写代码前**先对齐需求** | PRD + 设计评审 |

**OpenCode 是舞台，Superpowers 是表演方法，OpenSpec 是剧本。** 三者不是竞品，是不同层的东西——甚至可以全用在同一个项目里。

---

## 二、OpenCode：开源的终端 AI 编程 Agent

### 2.1 它是什么

OpenCode 是 [anomalyco](https://github.com/anomalyco) 开发的开源 AI 编程 Agent，180K GitHub stars，MIT 协议。它运行在终端里（也有桌面 App 和 IDE 插件），可以直接连接 75+ LLM 供应商的任何模型。

核心定位：**让 AI 在你本地的终端里直接读写代码、执行命令、查文件，像一个真实的结对编程搭档。**

### 2.2 关键能力

| 能力 | 说明 |
|------|------|
| 终端原生 | 在 WezTerm / Alacritty / Ghostty / Kitty 里跑，全键盘操作 |
| 任意模型 | 不绑模型，支持 Claude、GPT、Gemini 及 75+ 供应商 |
| LSP 集成 | 自动加载语言服务，让 LLM 理解类型系统 |
| 双模式 | Build 模式（全权限）/ Plan 模式（只读、先分析再建议） |
| 多会话 | 同一项目并行跑多个 Agent |
| /undo /redo | 改坏了直接回退 |
| 分享链接 | 把对话生成可分享的链接 |
| AGENTS.md | 每个项目放一份说明文件，Agent 启动时自动读取 |
| MCP / Skills / Rules | 扩展工具、定制技能和规则 |
| 桌面 App (Beta) | macOS / Windows / Linux |

### 2.3 典型工作流

```
cd my-project
opencode

/init                    → Agent 分析项目，生成 AGENTS.md
<TAB> 切 Plan 模式       → Agent 只读，帮你做方案
"我们想给 /settings 加身份校验"
<TAB> 切回 Build 模式    → Agent 开始改代码
/undo                    → 改坏了就回退
/share                   → 分享对话链接给同事
```

**OpenCode 的本质：它是一个运算环境。** 它不关心你用什么开发方法论、有没有写设计文档——它只管执行：读文件、改代码、跑命令、调工具。

## 三、Superpowers：教 Agent 怎么干活才不会翻车

### 3.1 它是什么

Superpowers 是 [Jesse Vincent](https://blog.fsck.com)（Prime Radiant）创建的开源技能框架，240K GitHub stars，MIT 协议。它不是一个独立的编程工具，而是一套**安装到编程 Agent 里的技能和方法论**，让 Agent 从根本上改变行为模式。

核心定位：**把 TDD、YAGNI、DRY、分步设计、子 Agent 驱动开发等软件工程最佳实践，编码成 Agent 自动遵守的工作流。**

### 3.2 技能库

Superpowers 包含 14 个技能，覆盖完整开发链路：

| 阶段 | 技能 | 做什么 |
|------|------|--------|
| 启动 | using-superpowers | 引导 Agent 理解整个技能体系 |
| 设计 | brainstorming | 苏格拉底式对话，追问需求，输出设计文档 |
| 环境 | using-git-worktrees | 创建隔离开发分支，验证测试基线 |
| 规划 | writing-plans | 把设计拆成 2–5 分钟一个的小任务 |
| 执行 | subagent-driven-development | 每个任务起一个子 Agent 独立完成 + 两阶段 Review |
| 执行 | test-driven-development | 强制 RED-GREEN-REFACTOR 循环 |
| 执行 | executing-plans | 批量执行 + 人类检查点 |
| 质检 | requesting-code-review | 对照 Plan 做 Review，严重问题阻塞进度 |
| 质检 | verification-before-completion | 确保问题真的修好了 |
| 收尾 | finishing-a-development-branch | 决定合并/PR/保留/丢弃 |
| 调试 | systematic-debugging | 四阶段根因分析流程 |
| 元技能 | writing-skills | 让 Agent 自己创建新技能 |
| 并行 | dispatching-parallel-agents | 并发多个子 Agent |

**注意：这些技能是 Agent 行为规则的集合，不是你自己去执行的流程。** 你装上 Superpowers 之后，Agent 会自觉在写代码之前先出设计、写测试再写实现、写完做 Review——不需要你催。

### 3.3 核心理念

- **测试先行**，永远先写测试再写实现
- **YAGNI**（你不会需要它）：不提前写用不上的代码
- **DRY**（不要重复自己）
- **子 Agent 驱动开发**：每个任务起干净上下文的子 Agent，执行完销毁
- 复杂任务走完整流程（设计 → 规划 → 执行 → 评审），简单修改不啰嗦

### 3.4 它怎么和 Agent 配合

Superpowers 被设计成可以安装到多种 Agent 上：Claude Code、Codex、Cursor、Gemini CLI、Copilot CLI、Kimi Code、**OpenCode** 等。

在 OpenCode 里安装 Superpowers 的方式：

```
Fetch and follow instructions from https://raw.githubusercontent.com/obra/superpowers/refs/heads/main/.opencode/INSTALL.md
```

装上之后，Agent 的行为就变了——它会主动走完整开发流程，从设计到 TDD 到 Review。

**Superpowers 的本质：它是软件工程方法论的 Agent 编码。** 它不是工具，是教工具怎么干活的一套规则和技能。

## 四、OpenSpec：先对齐需求，再写代码

### 4.1 它是什么

OpenSpec 是 [Fission AI](https://github.com/Fission-AI) 开发的开源 Spec 驱动开发框架，57K GitHub stars，MIT 协议。它的核心思路是：**在让 AI 写任何代码之前，先生成一份人能 Review 的 Spec（规格说明），对齐之后再动手。**

核心定位：**一个活在代码仓库里的轻量规划层，让需求和实现之间有了一份白纸黑字的契约。**

### 4.2 工作流

OpenSpec 用四个命令覆盖完整链路：

| 命令 | 做什么 | 产出 |
|------|--------|------|
| `/opsx:explore` | 不写代码，分析现有代码、推演方案 | 探索报告 |
| `/opsx:propose` | 生成变更提案（设计 + Spec + 任务） | proposal.md + design.md + tasks.md + spec deltas |
| `/opsx:apply` | 按任务清单逐项实现 | 代码变更 |
| `/opsx:archive` | 归档变更，更新 Spec 基线 | 归档目录 |

### 4.3 Spec 文件的形态

每次 `/opsx:propose` 会产生一个变更目录，里面最重要的部分是 **Spec Delta**：

```diff
# auth-session/spec.md

### Requirement: Session expiration
- The system SHALL expire sessions after 24 hours.
+ The system SHALL support configurable session expiration periods.

#### Scenario: Default session timeout
- GIVEN a user has authenticated
- WHEN 24 hours pass without activity
+ WHEN 24 hours pass without "Remember me"
- THEN invalidate the session token

+ #### Scenario: Extended session with remember me
+ - GIVEN user checks "Remember me" at login
+ - WHEN 30 days have passed
+ - THEN invalidate the session token
+ - AND clear the persistent cookie
```

这份 Delta 记录了**这次变更对系统需求的修改**。Reviewer 不用看代码就能知道改了什么。合并后 Spec 会更新到基线，成为活文档。

### 4.4 Spec 存在哪里

直接存在代码仓库里：

```
openspec/
├── specs/                ← 所有功能模块的 Spec 基线
│   ├── auth-login/spec.md
│   ├── auth-session/spec.md
│   └── checkout-cart/spec.md
└── changes/              ← 进行中的变更
    └── add-remember-me/
        ├── proposal.md
        ├── design.md
        ├── tasks.md
        └── specs/
            └── auth-session/spec.md  ← Spec Delta
```

**Spec 和代码一起版本管理，一起 Code Review，一起部署。** 不会像传统的 PRD 那样写完就过时。

### 4.5 OpenSpec 支持哪些 Agent

OpenSpec 自称支持 25+ AI 编码工具，包括 Claude Code、Cursor、Codex、GitHub Copilot、**OpenCode**、Windsurf、Gemini CLI、Cline、RooCode、Amazon Q 等。

这意味着你可以在 OpenCode 里用 OpenSpec 的 `/opsx:` 命令——Agent 会按 OpenSpec 的流程来工作。

**OpenSpec 的本质：它是代码仓库里的规划层。** 它不执行代码，不教 Agent 怎么干活。它只做一件事：确保人机之间在代码出现之前，就"要做什么"达成一致。

## 五、三者对比：一张表看清关系

| 维度 | OpenCode | Superpowers | OpenSpec |
|------|----------|-------------|----------|
| 定位 | AI 编程 Agent 运行时 | 开发方法论 & 技能框架 | Spec 驱动规划层 |
| Creator | Anomaly | Jesse Vincent / Prime Radiant | Fission AI |
| Stars | 180K | 240K | 57K |
| 独立运行 | ✅ 可以 | ❌ 需要宿主 Agent | ❌ 需要宿主 Agent |
| 写代码 | ✅ 直接改代码 | ❌ 告诉 Agent 怎么改 | ❌ Spec 是产物不是代码 |
| 规划能力 | 有 Plan 模式 | ✅ brainstorming + writing-plans | ✅ explore + propose |
| TDD 强制 | ❌ | ✅ 强制 RED-GREEN-REFACTOR | ❌ |
| Spec/PRD | ❌ | 有设计文档 | ✅ 核心能力 |
| 安装方式 | `npm i -g opencode-ai` | 插件安装（多 Agent 兼容） | `npm i -g @fission-ai/openspec` |
| 能在一起用吗 | — | ✅ 可装进 OpenCode | ✅ 可在 OpenCode 里用 |

### 关键误解澄清

- ❌ "Superpowers 是 OpenCode 的竞品" → **错**。Superpowers 是方法论插件，可以装在 OpenCode 里。
- ❌ "OpenSpec 是另一种 AI 编程工具" → **错**。OpenSpec 是 Spec 框架，运行在已有的 AI Agent 之上。
- ❌ "三个挑一个用就行" → **不准确**。它们是不同层，可以组合：**OpenCode 跑 Agent，OpenSpec 管规划，Superpowers 管方法论。**

## 六、架构关系：它们怎么叠在一起

```
                    ┌─────────────────────────┐
                    │      Superpowers        │  ← 方法论层
                    │  "怎么干活才不出错"      │
                    │  TDD / 设计 / Review    │
                    ├─────────────────────────┤
                    │       OpenSpec          │  ← 规划层
                    │  "先对齐再动手"          │
                    │  Spec / Proposal / Task │
                    ├─────────────────────────┤
                    │       OpenCode          │  ← 运行时层
                    │  "读文件、改代码、跑命令" │
                    │  LLM / LSP / MCP / Git  │
                    └─────────────────────────┘
```

三者各管一层，互不冲突：

- **OpenCode 在最底层**：提供 Agent 运行环境，连接模型、读写文件、执行命令
- **OpenSpec 在中间层**：在 Agent 动手之前，先产出可 Review 的 Spec 和任务清单
- **Superpowers 在最上层**：给 Agent 灌输 TDD、Review、子 Agent 等方法论习惯

## 七、怎么选：不同场景的组合建议

### 7.1 场景一：个人快速开发

**只装 OpenCode 就够了。**

你心里清楚要什么，不用写 Spec，不用走完整方法论。想改就改，错了 `/undo`。

### 7.2 场景二：复杂功能开发（多人协作）

**OpenCode + OpenSpec。**

团队需要一个共同的需求真相源。用 OpenSpec 先出 Spec，Review 完再让 Agent 实现。Spec 随代码入库，新同事拉下来就能知道每个模块的设计意图。

### 7.3 场景三：追求工程质量（TDD、Review 等）

**OpenCode + Superpowers。**

你想要 Agent 遵守严谨的开发纪律：先写测试、小步提交、自动 Review。Superpowers 把这套方法论装进 Agent 里，不需要你手动催。

### 7.4 场景四：大型工程（最全组合）

**OpenCode + OpenSpec + Superpowers。**

三者打通后的理想流程：

1. **OpenSpec** `/opsx:explore` → 探索方案
2. **OpenSpec** `/opsx:propose` → 产出 Spec + 设计 + 任务
3. 人工 Review Spec，确认对齐
4. **Superpowers** 接管执行：brainstorming 已就位 → writing-plans 做细拆 → subagent-driven-development 逐个任务 TDD 实现 → code review 自动检查
5. **OpenCode** 作为运行时执行所有操作
6. **OpenSpec** `/opsx:archive` → 归档变更，更新 Spec 基线

这个组合目前还没有默认打通（需要一些手动衔接），但三者的设计理念天然互补。**一个管"要做什么"，一个管"怎么做好"，一个管"在哪做"。**

## 八、各自的局限

| 项目 | 主要局限 |
|------|---------|
| OpenCode | 本身没有强制的规划和方法论；Agent 行为依赖你和它对话的质量 |
| Superpowers | 强方法论意味着重流程；简单的一次性修改也会触发设计文档环节（可跳过但费 Token） |
| OpenSpec | 只做规划对齐，不改代码不教方法；Spec 质量取决于你的 Review 投入 |

**没有一个工具能包办一切。** OpenCode 给自由度，Superpowers 给纪律，OpenSpec 给共识。自由度和纪律是矛盾的，共识和速度也是矛盾的——怎么平衡，取决于你的项目阶段和团队文化。

---

## 九、结语：它们是不同层的答案

回到标题的问题：OpenSpec 和 Superpowers 是什么，跟 OpenCode 有什么区别？

**OpenCode 是 AI 编程的白纸**——你可以随便在上面画，怎么画都行，画坏了自己改。

**OpenSpec 是图纸**——动手之前先把线框画好，人看一眼说"对"再让 AI 填颜色。

**Superpowers 是画法**——一套经过验证的笔法、构图规则、质检流程，让你的画稳定、耐看、不出废稿。

三者不打架。你用 OpenCode 写代码的时候，可以同时装上 Superpowers 的技能，再配上 OpenSpec 的规划命令。这不是生态碎片化，这是软件工程的经典分层：**运行时、方法论、需求规格，从来就是三层。** AI 编程工具的发展，不过是用开源项目的形式，把这三层重新实现了一遍——而且每一层都给了一个让你自己选的选项，而不是一家厂商打包卖你全家桶。

选择权在你手上。选几层、怎么组合，这是 2026 年 AI 编程最有意思的地方。
