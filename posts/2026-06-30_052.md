# Harness 和 Hermes 分不清？Agent Skills 生态的两块基石

> 你有没有遇到过：给 AI Agent 写了一堆 Skill，结果换个平台全废了？OpenCode 的 Skill 不能给 Claude Code 用，Claude Code 的不能给 Codex 用——明明长得很像，就是不通。最近圈子里两个以 H 开头的概念特别火：**Harness**（Skill 运行时引擎）和 **Hermes**（Skill 互操作协议）。很多人把它们当竞品，其实它们一个是执行层，一个是标准层，合在一起才构成完整的 Agent Skills 生态。本文拆清两者的定位、架构、代码实现和未来走向。

---

## 一、一句话定义

| 概念 | 一句话 | 类比 | 比喻 |
|------|--------|------|------|
| **Harness** | 加载和执行 Skill 的运行时引擎 | JavaScript 引擎（V8 / SpiderMonkey） | 汽车发动机 |
| **Hermes** | 定义 Skill 格式和跨平台互操作的标准 | ECMAScript 规范 | 汽油规格（92/95/98 号都兼容） |

**Harness 回答"怎么跑"，Hermes 回答"长什么样"。** 没有 Hermes，每个平台各玩各的，Skill 无法共享；没有 Harness，标准再好也没地方执行。两者是生态的上下两层，不是竞品。

---

## 二、Harness：Agent Skill 的运行时引擎

### 2.1 它解决什么问题

2025 年以来，各个 AI 编程 Agent 都开始支持 Skill 系统：

| 平台 | Skill 支持方式 |
|------|--------------|
| **OpenCode** | `~/.config/opencode/skills/` 目录 + 内置 Harness |
| **Claude Code** | `.claude/skills/` + Claude 内置加载器 |
| **Codex** | `~/.codex/skills/` + Codex Skill runtime |
| **Copilot CLI** | `~/.config/copilot-cli/skills/` + Copilot Harness |

每个平台的 Skill 机制高度相似——都有一个目录存放 Skill 定义文件，都有一套指令注入逻辑，都支持条件触发。但**实现各不相同**：

- 文件格式不同（有的用 YAML 前置元数据，有的用独立配置）
- 加载时机不同（有的启动时全量加载，有的按需）
- 上下文注入策略不同（有的全量注入，有的大量压缩）
- 工具映射不同（同一个"读文件"操作，不同平台工具名不一样）

**Harness 就是负责处理这些差异的那层软件**：它从文件系统读取 Skill，解析内容，注入到 Agent 的 System Prompt 里，并把 Skill 里声明的操作映射到平台实际可用的工具。

### 2.2 Harness 的核心职责

```
┌─────────────────────────────────────────────┐
│                 Agent 对话窗口               │
├─────────────────────────────────────────────┤
│              System Prompt                   │
│  ┌─────────────────────────────────────┐    │
│  │  Skill Instructions (由 Harness 注入) │    │
│  │  - 触发条件 / 工作流 / 规则          │    │
│  └─────────────────────────────────────┘    │
├─────────────────────────────────────────────┤
│                LLM 推理                     │
├─────────────────────────────────────────────┤
│          Harness (运行时引擎)                │
│  ┌───────────┐ ┌──────────┐ ┌───────────┐  │
│  │ 发现 Skill │ │ 加载解析  │ │ 注入上下文 │  │
│  └───────────┘ └──────────┘ └───────────┘  │
│  ┌───────────┐ ┌──────────┐ ┌───────────┐  │
│  │ 工具映射   │ │ 权限控制  │ │ 版本管理   │  │
│  └───────────┘ └──────────┘ └───────────┘  │
├─────────────────────────────────────────────┤
│              平台原生工具                    │
│       (fs/exec/grep/web/search...)          │
└─────────────────────────────────────────────┘
```

具体来说，Harness 干五件事：

1. **Skill 发现**：从配置目录（如 `~/.config/opencode/skills/`）扫描所有 Skill 定义
2. **元数据解析**：读取 Skill 的名称、描述、触发条件、优先级等元信息
3. **指令注入**：在 Agent 启动时或对话轮次中，将匹配的 Skill 指令注入 System Prompt
4. **工具适配**：把 Skill 里声明的抽象操作（如"读取文件"）映射到平台实际工具（OpenCode 的 `Read`、Claude Code 的 `View` 等）
5. **权限控制**：决定哪些 Skill 可以自动触发，哪些需要用户确认

### 2.3 一个 Harness 的最小实现

以 OpenCode 为例，Skill 加载的核心逻辑：

```javascript
// Harness 的核心：扫描、解析、注入
class SkillHarness {
  constructor(skillDir) {
    this.skillDir = skillDir;
    this.skills = [];
  }

  // 1. 发现：扫描 Skill 目录
  async discover() {
    const entries = await fs.readdir(this.skillDir);
    for (const name of entries) {
      const skillPath = path.join(this.skillDir, name);
      const stat = await fs.stat(skillPath);
      if (stat.isDirectory()) {
        const manifest = await this.loadManifest(skillPath);
        this.skills.push({ name, path: skillPath, ...manifest });
      }
    }
  }

  // 2. 加载：读取 Skill 的完整内容
  async loadSkill(skill) {
    const skillFile = path.join(skill.path, 'SKILL.md');
    return await fs.readFile(skillFile, 'utf-8');
  }

  // 3. 匹配：判断用户输入是否触发该 Skill
  match(userInput, skill) {
    return skill.triggers?.some(t =>
      userInput.toLowerCase().includes(t.toLowerCase())
    ) ?? false;
  }

  // 4. 注入：将 Skill 指令拼入 System Prompt
  async buildSystemPrompt(userInput) {
    const basePrompt = this.baseSystemPrompt;
    let skillContext = '';

    for (const skill of this.skills) {
      if (this.match(userInput, skill)) {
        skillContext += await this.loadSkill(skill) + '\n';
      }
    }

    return skillContext
      ? `${skillContext}\n---\n${basePrompt}`
      : basePrompt;
  }
}
```

这个 50 行的实现已经覆盖了 Harness 最核心的四步。不同平台的 Harness 差异主要在：
- 是否支持进度通知（如"正在加载 Skill X..."）
- 是否支持 Skill 卸载/热重载
- 上下文窗口管理和 Skill 内容压缩策略

### 2.4 Harness 之间的差异就是碎片化的来源

因为 Harness 是**平台特定的**，同一个 Skill 文件在不同平台上的行为可能不一致：

| 差异点 | 举例 |
|--------|------|
| 工具名 | OpenCode 叫 `Read`，Claude Code 叫 `View`，Codex 叫 `read_file` |
| 权限模型 | OpenCode 支持 `approved` 目录白名单，Claude Code 用 `permissions.allow` |
| 触发方式 | OpenCode 自动匹配关键词，Codex 需要显式 `/skill` 调用 |
| Skill 格式 | 部分平台用 `SKILL.md` + `manifest.json`，部分用内嵌 YAML frontmatter |

**这就是 Hermes 要解决的问题。**

## 三、Hermes：Agent Skill 的互操作协议

### 3.1 名字的由来

Hermes（赫尔墨斯）是希腊神话中的信使之神，穿有翼凉鞋，负责在神祇之间传递消息。**Agent Skills 生态里的 Hermes 也是这样**：它不做执行，只管"传递"——定义一套 Skill 的**通用格式**，让 Skill 能在不同平台的 Harness 之间流通。

### 3.2 Hermes 协议的核心

Hermes 本质上是一份**规范文档 + 校验工具链**，定义了三层东西：

#### 第一层：Skill 包格式

```
my-skill/
├── SKILL.md              # 核心指令（Markdown，必须）
├── hermes.json          # Hermes 元数据（必须）
├── assets/              # 配套资源（可选）
│   ├── templates/
│   └── references/
├── tests/               # 技能验证（可选）
└── examples/            # 使用示例（可选）
```

其中 `hermes.json` 的结构：

```json
{
  "hermes": "1.0",
  "skill": {
    "name": "systematic-debugging",
    "version": "1.2.0",
    "description": "Systematic debugging workflow for any bug",
    "triggers": ["bug", "error", "fix", "broken", "failing"],
    "priority": "high",
    "category": "workflow",
    "author": "superpowers",
    "license": "MIT"
  },
  "requirements": {
    "tools": ["read_file", "search_code", "execute_command"],
    "context_tokens": 4000,
    "permissions": ["filesystem_read", "shell_exec"]
  },
  "interop": {
    "tool_aliases": {
      "read_file": ["Read", "View", "read", "cat"],
      "search_code": ["Grep", "grep", "search_content"],
      "execute_command": ["Bash", "Terminal", "exec", "run"]
    }
  }
}
```

#### 第二层：工具别名映射

这是 Hermes 最关键的设计——**工具别名表**。Skill 作者写的操作名是抽象的（如 `read_file`），Hermes 把它映射到各平台的实际工具名：

```json
{
  "interop": {
    "tool_aliases": {
      "read_file": {
        "opencode": "Read",
        "claude_code": "View",
        "codex": "read_file",
        "copilot_cli": "read"
      },
      "search_code": {
        "opencode": "Grep",
        "claude_code": "Glob",
        "codex": "search_content",
        "copilot_cli": "grep"
      }
    }
  }
}
```

这样 Skill 作者只需要用 `read_file`、`search_code` 等标准工具名写指令，Hermes 在加载时会自动翻译成当前平台的实际工具名。

#### 第三层：Skill 指令规范

Hermes 不强制 Skill 的**内容**写什么——怎么写是 Skill 作者的自由。但它定义了一些约定：

1. **`## When to use`** 章节：定义触发条件
2. **`## Workflow`** 章节：定义执行步骤
3. **`## Platform notes`** 章节：平台特殊说明（可选）
4. **`// 平台名: 替代工具`** 行内注释：就地标注工具替代

### 3.3 Hermes 适配器的工作原理

给定一个符合 Hermes 的 Skill，各平台的 Harness 通过一个**适配器**加载它：

```javascript
// Hermes Adapter（逻辑在各个 Harness 里实现）
class HermesAdapter {
  constructor(platformId) {
    this.platformId = platformId; // 'opencode' | 'claude_code' | ...
  }

  // 加载 Hermes Skill 并适配当前平台
  async load(skillDir) {
    const manifest = await this.loadManifest(skillDir);
    const rawContent = await fs.readFile(
      path.join(skillDir, 'SKILL.md'), 'utf-8'
    );

    // 核心转换：把标准工具名替换为当前平台工具名
    const adapted = this.translateTools(rawContent, manifest.interop.tool_aliases);

    return {
      name: manifest.skill.name,
      triggers: manifest.skill.triggers,
      content: adapted,
      permissions: manifest.requirements.permissions
    };
  }

  translateTools(content, aliases) {
    let result = content;
    for (const [standardName, mappings] of Object.entries(aliases)) {
      const platformTool = mappings[this.platformId];
      if (platformTool && platformTool !== standardName) {
        // 替换文档中的标准工具名为平台工具名
        result = result.replaceAll(standardName, platformTool);
      }
    }
    return result;
  }
}
```

### 3.4 Hermes 和 Harness 的关系

```
                     Hermes 协议
                  (定义 Skill 通用格式)
                          │
                          │ 适配
                          ▼
    ┌──────────────────────────────────────────┐
    │          Hermes Adapter Layer            │
    │    (工具名翻译 / 权限转换 / 格式适配)      │
    └──────┬───────────┬───────────┬──────────┘
           │           │           │
           ▼           ▼           ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ OpenCode │ │ClaudeCode│ │  Codex   │
    │ Harness  │ │ Harness  │ │ Harness  │
    └──────────┘ └──────────┘ └──────────┘
         执行        执行        执行
```

**Harness 是执行层，Hermes 是标准层，Adapter 是连接层。**

## 四、核心差异对照表

| 维度 | Harness | Hermes |
|------|---------|--------|
| **定位** | 运行时引擎 | 互操作协议 |
| **谁实现** | 各 Agent 平台 | 社区标准 + 各平台适配器 |
| **关心什么** | 加载、执行、注入、权限 | 格式、别名、兼容、分发 |
| **是否平台特定** | **是**（每个平台有自己的） | **否**（跨平台通用） |
| **类比** | Docker Engine | OCI Image Spec |
| **类比 2** | JavaScript 引擎 | ECMAScript |
| **类比 3** | 汽车发动机 | 汽油规格 |
| **输出** | 可执行的 Agent 行为 | 可被任意 Harness 加载的 Skill 包 |
| **关键问题** | "这个 Skill 怎么跑？" | "这个 Skill 能不能在别处跑？" |

## 五、为什么不能只有一个

### 5.1 如果只有 Harness 没有 Hermes

- Skill 作者要给每个平台写一份适配版
- 开源社区分裂成 N 个 walled garden
- "选平台 = 选生态"，迁移成本极高
- 就像每个浏览器只能访问自己格式的网页

### 5.2 如果只有 Hermes 没有 Harness

- 有标准但没地方执行，标准是纸面上的
- 社区缺乏验证标准的实际运行环境
- 标准容易脱离实践，成为空中楼阁
- 就像定义了 ES2025 但没有任何浏览器实现它

### 5.3 两者并存的生态

这才是健康状态：

1. **Skill 作者**按 Hermes 标准写一份 Skill
2. **各平台**用 Harness + Hermes Adapter 加载它
3. **用户在任意平台**安装同一个 Skill，行为一致
4. **社区**围绕统一格式积累和分享 Skill，形成飞轮

这跟 Docker/OCI 的故事完全一样：Docker 是第一个 Harness（Docker Engine），OCI 是后来的 Hermes（容器镜像标准）。有了 OCI 之后，Podman、containerd、runc 都可以跑同一个镜像。

## 六、实际场景：写一个跨平台 Skill

假设要给 Agent 写一个"代码审查"Skill。如果用纯平台原生的写法：

```markdown
<!-- OpenCode 版本 -->
## When to use
用户说 "review"、"code review"

## Workflow
1. 用 Read 读取变更文件
2. 用 Grep 查找相关上下文
3. 用 Bash 运行 lint 命令
4. 输出结构化审查意见
```

```markdown
<!-- Claude Code 版本 -->
## When to use
用户说 "review"、"code review"

## Workflow
1. 用 View 读取变更文件
2. 用 Glob 查找相关上下文
3. 用 Bash 运行 lint 命令
4. 输出结构化审查意见
```

两份文件 80% 一样，就因为 `Read` vs `View`、`Grep` vs `Glob` 这种细碎差异要维护两份。如果平台增加到五个，维护成本线性增长。

用 Hermes 标准写法，一份文件就够了：

```markdown
<!-- Hermes 标准写法 -->
## When to use
用户说 "review"、"code review"

## Workflow
1. 用 `read_file` 读取变更文件
2. 用 `search_code` 查找相关上下文
3. 用 `execute_command` 运行 lint 命令
4. 输出结构化审查意见
```

配一个 `hermes.json` 声明工具别名映射，适配器自动翻译：

```json
{
  "interop": {
    "tool_aliases": {
      "read_file": {
        "opencode": "Read",
        "claude_code": "View",
        "codex": "read_file",
        "copilot_cli": "read",
        "gemini_cli": "View"
      },
      "search_code": {
        "opencode": "Grep",
        "claude_code": "Glob",
        "codex": "search_content",
        "copilot_cli": "grep",
        "gemini_cli": "Search"
      },
      "execute_command": {
        "opencode": "Bash",
        "claude_code": "Bash",
        "codex": "exec",
        "copilot_cli": "run",
        "gemini_cli": "Shell"
      }
    }
  }
}
```

**写一次，到处跑。** 这就是 Hermes 的价值——它把 Skill 从"平台附属品"变成了"独立资产"。

## 七、Skill 分发的未来：从目录到注册表

Hermes 如果只解决"格式统一"，效果有限。真正让生态起飞的是**统一的分发渠道**：

```
┌──────────────────────────────────────────┐
│          Skill Registry (类似 npm)        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │ 调试 Skill  │ │ CI Skill │ │ 前端 Skill │  │
│  │ v1.2.0   │ │ v3.1.0  │ │ v2.0.0  │  │
│  └──────────┘ └──────────┘ └──────────┘  │
└──────────────────────────────────────────┘
                    │
          Hermes 协议 (统一格式)
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
OpenCode       Claude Code       Codex
Harness         Harness          Harness
```

一旦有了标准的格式（Hermes）+ 集中的分发（Registry）+ 各自的运行（Harness），Skill 生态就进入了跟 npm/PyPI/crates.io 一样的正循环：

1. 开发者写一个 Skill，发布到 Registry
2. 用户在任意平台 `install skill-name` 安装
3. 平台的 Harness 加载并执行

目前这个生态还在早期，但方向已经很清晰了。

## 八、现状和展望

### 当前阶段（2026 年中）

| 层面 | 状态 |
|------|------|
| Harness | 各平台**都有**，稳定运行中 |
| Hermes 标准格式 | **提案阶段**，社区在收敛 |
| Registry | 尚无统一注册表，GitHub + 手动安装为主 |
| 跨平台兼容 | 需手动适配，工具名翻译靠人工 |

### 下一步

1. **Hermes 标准定稿**：社区统一 Skill 格式和工具别名表
2. **各平台内置 Hermes Adapter**：Harness 原生支持 Hermes 格式加载
3. **Skill Registry 上线**：集中式或去中心化的 Skill 分发平台
4. **生态爆发**：一旦 Hermes + Registry 就位，Skill 数量会指数增长

### 对开发者的建议

- **现在就开始用 Skill 系统**，不要等标准完美了再入坑——Harness 已经很稳定了
- **写 Skill 时注意工具名的可移植性**：在自己的文档里注明"本步骤对应 OpenCode 的 X 工具"，方便未来迁移
- **关注 Hermes 提案的进展**：一旦定稿，第一时间把自己的 Skill 转为 Hermes 格式
- **不要纠结选哪个平台**：有了 Hermes，Skill 是跨平台的资产，选平台看别的维度（性能、模型支持、UX）

## 九、总结

```
问：到底谁是"真正的"Skill 引擎？
答：Harness 是引擎，Hermes 是燃油标准。

问：我该学哪个？
答：都该了解。你现在用 Harness，未来所有 Skill 都会是 Hermes 格式。

问：能不能二选一？
答：不能。没有 Harness，Skill 无法执行；没有 Hermes，Skill 无法流通。
两者是生态的两块基石，不是竞品，不存在"谁取代谁"。
```

**一句话**：**Harness 让 Skill 能跑，Hermes 让 Skill 能到处跑。**
