# 什么是skill，如何从0开发一个skill

## 1. 引言

你有没有发现，现在大家讨论AI时，说得最多的已经不是大模型本身了——而是"AI Agent"。

当OpenAI推出GPTs，当Anthropic开放Claude 3的工具调用能力，当各家科技巨头都在布局Agent生态，一个词越来越频繁地出现在我们视野里：**skill**。

很多人都有过这样的体验：问大模型一个问题，它说得头头是道，但仔细一看，里面到处都是"幻觉"——数据错了，时间错了，甚至连引用的资料都是编出来的。让它帮你查天气，它只能给你编一个结果；让它帮你算汇率，它用的可能是去年的数据；让它帮你订机票，它更是无能为力，只能告诉你"我不能直接帮你订票"。

为什么会这样？

原因很简单：大模型擅长"生成内容"，但不擅长"完成任务"。它训练的时候见过海量文本，所以能说出很多东西，但它没有办法实时连上网获取最新信息，更没有办法直接调用外部服务帮用户完成实际操作。

这就是skill要解决的问题。

Skill，中文可以翻译为"技能"，就是给AI大模型/AI Agent装上"手脚"和"眼睛"，让它能够获取实时信息、调用外部服务、真正完成用户交给它的具体任务。有了skill，AI就不再只是一个"聊天机器人"，而是一个能真正解决问题的"智能助手"。

今天，我们就从头开始聊一聊：到底什么是skill？它为什么这么重要？以及，如果你想开发一个属于自己的skill，应该怎么做？

不管你是产品经理想理解这个新兴领域，还是开发者想动手实践，这篇文章都会给你一个完整的框架。

## 2. 什么是skill

说了这么多，那到底什么是skill？

简单来说：**Skill 是 AI 系统中的一个独立能力模块，它封装了完成特定任务的逻辑，通过标准化接口被 AI Agent 调用。**

如果用我们熟悉的生活场景做类比，你可以这样理解：

- 如果把 AI Agent 想象成一个餐厅，那么 skill 就是餐厅里的每个厨师。每个厨师只专注做好一道菜，宫保鸡丁厨师就做宫保鸡丁，鱼香肉丝厨师就做鱼香肉丝。客人点什么菜，就叫对应的厨师来做。

- 如果把 AI Agent 想象成一个公司，那么 skill 就是公司里的各个部门：市场部负责品牌推广，技术部负责写代码，财务部负责管钱。每个部门只负责自己专业领域的事情，有事情找对应部门就好。

### Skill 的核心特征

一个好的 skill，通常具备这几个特点：

**1. 单一职责**

一个 skill 通常只做好一件事。不会有一个 skill 既帮你查天气，又帮你订机票，还帮你写代码。就像一个厨师不会既做中餐又做西餐还做面点，专业的人做专业的事，单一职责让 skill 更聚焦，更容易维护。

**2. 可插拔**

Skill 是独立的模块，需要的时候加载进去，不需要就可以拿掉，不会影响 AI Agent 整体的运行。你今天想加一个"搜索"skill，加上就行；明天觉得不需要了，删掉就好，不用改其他代码。

**3. 可复用**

同一个 skill 可以被不同的 Agent 使用。只要遵循相同的接口标准，你开发的 weather skill，任何人都可以拿到自己的 Agent 里去用，不需要重新开发。

**4. 独立开发**

Skill 可以单独开发、测试、更新，不需要和 AI Agent 一起发布。开发者可以独立迭代自己的 skill，不依赖主版本的发布节奏。

### Skill 和其他概念的区别

很多人会把 skill 和插件、工具、功能这些概念搞混，我们来理一理它们的区别：

#### Skill vs 插件

插件（Plugin）这个概念很早就有了，传统的插件更多是**扩展宿主程序的功能**，给用户提供更多操作入口。比如浏览器插件，一般是给浏览器增加新功能，直接和用户交互。

而 skill 不同，skill 不是给用户用的，**是给 AI Agent 用的**。用户不直接调用 skill，而是告诉 Agent 自己想要什么，由 Agent 决定什么时候调用哪个 skill。

简单说：
- 插件：用户 → 插件 → 完成任务
- Skill：用户 → Agent → Skill → 完成任务

#### Skill vs 工具（Tool）

工具（Tool）这个词现在也用得很多，很多地方说的"工具调用"其实和我们说的 skill 很接近。

区别在于：**工具更偏向底层能力，而 skill 可能是多个工具的组合**。

比如你要做一个"规划周末出行"的 skill，它可能需要先后调用"查询天气"工具、"查询交通"工具、"查询酒店"工具，把这些工具的结果整合起来，给用户一个完整的出行方案。这个整合了多个工具的完整能力模块，就是一个 skill。

#### Skill vs 能力（Capability）

能力（Capability）是一个更宽泛的概念，大模型本身就具备很多能力，比如推理能力、写作能力。这些是大模型天生就有的，不需要额外调用外部模块。

而 skill 通常指**额外扩展的能力**，是大模型本身不具备，需要通过外部模块来实现的能力。当然，也有人把大模型本身的能力叫做"基础skill"，这只是定义上的区别，核心思想还是一致的。

### 一句话总结

> **Skill = 封装了特定能力 + 遵循标准接口 + 可被 AI Agent 动态调用的独立模块**。

理解了什么是skill，下一章我们来聊聊，为什么skill这么重要，它带来了什么核心价值。

## 3. Skill的核心价值

为什么现在大家都在讲skill？这个架构到底好在哪里？

我们从几个维度来看skill带来的核心价值。

### 1. 解决大模型天生的局限

我们在引言里说过，大模型不是完美的，它天生有几个局限：

- **不能获取实时信息**：训练完成后，大模型的知识就固定了，它不知道今天发生了什么，更不知道最新的天气、股价、新闻。
- **不能操作外部系统**：大模型只能生成文字，没法直接帮你订机票、发邮件、在数据库里改数据。
- **容易产生幻觉**：遇到不知道的问题，它宁愿瞎编也不承认不知道，因为它就是靠概率生成下一个词的。

Skill正好解决了这些问题：

- 要实时信息？通过skill调用搜索API或者天气API，拿到最新数据。
- 要操作系统？通过skill调用对应的接口，直接完成操作。
- 要避免幻觉？让skill先拿到准确数据，大模型只负责把数据整理成自然语言，从根源减少幻觉。

一句话：**大模型负责"思考"，skill负责"行动"**。专业的事情交给专业的模块做，分工合作才能把事情做好。

### 2. 让产品更易于扩展

产品做迭代的时候，最让人头疼的是什么？就是牵一发而动全身。你想加一个新功能，可能要改到处都是代码，测试起来也麻烦，一不小心就把原来的功能弄崩了。

Skill架构从根本上解决了这个问题：

- 加新能力 = 加一个新skill，不需要改Agent的核心代码
- 删除能力 = 删除一个skill，不影响其他功能
- 修改能力 = 只改这个skill本身，其他地方不用动

对于产品经理来说，这意味着什么？意味着你可以更快地验证想法，小步快跑试错。想加一个新功能试试看？先做一个skill，用户觉得好用就留着，不好用就拿掉，风险很低。

### 3. 促进分工协作

在一个AI产品团队里，不是每个人都懂所有事情：

- 有的人擅长做Agent框架，做规划、做调度、做用户交互
- 有的人擅长做垂直领域的能力，比如懂金融的做金融skill，懂医疗的做医疗skill
- 有的人擅长工程化，做基础设施，稳定性保障

Skill架构天然适合分工协作：只要大家都遵循同一个接口标准，框架团队只管框架，skill团队只管skill，最后拼在一起就能用。

往大了说，这就是生态。当标准建立起来了，每个人都可以开发skill，每个人都可以用别人开发好的skill，整个生态就会越来越繁荣。想象一下：如果有一万个不同的skill，用户可以按需组合，那能拼出多少种不同的AI产品？

### 4. 按需使用，节省成本

大模型推理是要钱的，GPU资源也是有限的。如果把所有能力都塞进一个大模型里，那这个模型会变得无比巨大，调用起来又慢又贵。

Skill架构是**模块化、可插拔**的：用户问什么问题，Agent只需要加载需要用到的skill，不需要把所有skill都加载进来。用户就问个天气，没必要把代码审查、文档编写这些skill都加载进来。

这样做的好处是：

- 启动更快，响应更快
- 节省内存，节省GPU资源
- 总体成本更低

对于用户来说，体验更好；对于厂商来说，成本更低，双赢。

### 5. 能力可以沉淀和复用

一个好的skill开发出来，可以被很多不同的Agent、很多不同的产品复用。

比如你开发了一个查询汇率的skill：

- 可以给出行Agent用
- 可以给海淘Agent用
- 可以给理财Agent用

不用每个产品都重新开发一遍，一次开发，多处使用，这就是复用的价值。整个行业的效率都会因此提升，大家不用重复造轮子，只需要专注在创造新的价值上。

### 价值总结

Skill不是玄学，也不是为了赶时髦玩概念，它真真切切解决了实际问题：

| 问题 | Skill带来的解决方案 |
|------|-------------------|
| 大模型天生有局限，不能联网不能操作 | 大模型思考，skill行动 |
| 产品扩展难，改一处影响全身 | 模块化，可插拔，独立迭代 |
| 团队协作效率低，每个人都要懂所有 | 分工协作，专业的人做专业的事 |
| 资源浪费，成本高 | 按需加载，节省资源 |
| 重复造轮子，效率低 | 一次开发，多处复用 |

下一章，我们来看看skill都有哪些常见类型。

## 4. Skill的常见分类

Skill可以从多个维度来分类，我们分别来看。

### 按功能分类

这是最直观的分类方式，按照skill能做什么事情来分：

#### （1）信息获取类

这类skill的作用就是从外部获取信息，回答大模型不知道的问题。常见例子：

- **搜索skill**：搜索引擎搜索，获取最新信息
- **天气skill**：查询实时天气
- **汇率skill**：查询最新汇率
- **新闻skill**：获取热点新闻
- **知识库skill**：查询公司内部知识库

特点：输入一个查询，返回相关信息，大多是读操作。

#### （2）数据处理类

这类skill对数据进行加工处理，输出处理后的结果：

- **摘要skill**：把长文章浓缩成摘要
- **数据分析skill**：对数据进行统计分析，生成报表
- **翻译skill**：把文字翻译成另一种语言
- **OCR skill**：从图片中识别文字

特点：输入数据，输出加工后的数据，不涉及外部系统操作。

#### （3）操作执行类

这类skill会对外部系统进行写操作，真正"做事"：

- **邮件skill**：发送邮件
- **日程skill**：创建日历日程
- **CRM skill**：在CRM系统创建客户记录
- **代码执行skill**：运行代码，输出结果
- **订票skill**：帮助用户订购机票酒店

特点：会改变外部系统的状态，需要权限验证，有副作用。

#### （4）复合能力类

这类skill整合了多个基础skill，完成一个相对完整的任务：

- **出行规划skill**：整合天气、交通、酒店推荐，给用户完整方案
- **简历筛选skill**：搜索简历、提取关键信息、打分排序
- **周报生成skill**：拉取Git提交记录、整理工作内容、生成周报

特点：内部会调用多个基础skill，把结果整合后一次性输出给用户，对用户来说更方便。

### 按复杂度分类

按skill的复杂程度，可以分为三个层级：

#### （1）基础工具型

最简单的skill，就是对一个API的简单封装：

- 输入参数固定
- 调用一个API
- 返回结果

比如查询天气就是典型的基础工具型skill：你给城市，我调用天气API，返回温度、湿度、风向就结束了。

开发难度低，复用性高。

#### （2）流程编排型

需要多个步骤，有逻辑流程控制：

- 用户输入可能不完整，需要追问用户
- 需要按顺序调用多个工具
- 根据前一步的结果决定下一步做什么

比如帮用户订机票，可能需要：
1. 问用户去哪里、什么时候走
2. 查询航班信息
3. 用户选择航班后调用支付
4. 支付完成出票

这种有流程有判断的skill，就是流程编排型。

#### （3）Agent型

最复杂的一种skill，skill本身就是一个Agent：

- 自己有规划能力
- 可以自主决定调用哪些工具
- 可以处理复杂的开放问题

比如"帮我做一个市场调研"就是一个复杂任务，这个skill自己会拆解问题，一步步搜索信息，最后给出完整报告。

这种skill其实就是一个Agent，被更大的Agent作为能力调用。

### 分类总结

分类不是目的，帮助我们理解skill的多样性就好。不管怎么分类，核心思想不变：**把特定能力封装成独立模块，遵循标准接口，方便调用。**

下一章，我们来看点有意思的——通过具体案例对比，看看什么样的skill是好skill，什么样的skill是不好的skill。

## 5. 好skill vs 坏skill：案例对比

说了这么多概念，我们来看几个实际例子，对比一下好skill和坏skill的区别。

### 案例一：查询天气

**❌ 坏skill设计：「查询北京天气」**

这个skill名字叫"查询北京天气"，只能查北京，不能查其他城市。

问题出在哪？
- 职责太窄，只能用在特定场景
- 扩展性差，要查上海还要再做一个skill
- 完全没有复用价值

**✅ 好skill设计：「查询天气」**

输入参数：城市名称
输出结果：该城市的实时天气信息

支持查询任何城市，不局限于特定城市。

点评：单一职责，但不绑定特定实例，复用性好，任何需要查询天气的场景都能用。

### 案例二：帮用户规划旅行

**❌ 坏skill设计：「规划周末北京三日游」**

把"用户问需求→查天气→查景点→查交通→生成攻略"整个流程硬编码写死，只能做北京三日游，换个城市就不行。

问题出在哪？
- 绑定了特定场景，复用性差
- 一旦要修改流程，整体都要改
- 其实可以拆成多个基础skill，让Agent去组合

**✅ 好skill设计：拆分成多个小skill**

- `查询天气`：查任何城市天气
- `查询景点`：查任何城市景点
- `查询交通`：查询城市间交通
- `生成攻略`：整合信息生成攻略

需要的时候Agent组合调用，不同场景可以灵活搭配。

点评：单一职责，灵活组合，比做成一个大而全的skill更有用。

### 案例三：处理用户数据

**❌ 坏skill设计：「生成销售报表」**

这个skill做了太多事情：
1. 从数据库拉取销售数据
2. 做统计计算
3. 生成PDF文件
4. 发送邮件给主管

所有步骤写在一个skill里。

问题出在哪？
- 几个不同职责混在一起
- 如果只想生成报表不想发邮件，用不了
- 如果换个邮箱服务商，整个skill都要改

**✅ 好skill设计：拆分成多个skill**

- `拉取销售数据`：只负责从数据库取数据
- `统计销售数据`：只负责计算统计
- `生成PDF报表`：只负责生成PDF
- `发送邮件`：只负责发邮件

需要完整流程就依次调用，只需要其中一步就单独用。更加灵活。

### 从案例看好skill的设计原则

通过上面几个对比，我们总结出好skill的几个设计原则：

| 原则 | 说明 |
|------|------|
| **单一职责** | 一个skill只做好一件事，不要什么都干 |
| **参数化** | 把变量做成参数，不要硬编码写死 |
| **可组合** | 小skill比大skill好，更容易组合出更多能力 |
| **无依赖** | 尽量减少对其他skill的强依赖，解耦 |
| **输入输出清晰** | 定义清楚输入什么，输出什么，方便Agent调用 |
| **错误处理** | 出错了返回清晰的错误信息，方便Agent处理 |

### 反模式：哪些设计要避免

开发skill时，这些坑要尽量避免：

1. **上帝skill**：一个skill什么都干，几千行代码，改都没法改
2. **过窄skill**：只能解决一个极其特定的问题，复用性几乎为零
3. **不稳定依赖**：依赖的服务经常变，skill也跟着经常变
4. **接口模糊**：输入输出不清晰，Agent不知道怎么调用
5. **缺失错误处理**：调用失败就直接崩了，不返回错误信息

记住一句话：**做小不做大，做简不做繁，做通用不做特定**。这是skill设计的黄金法则。

下一章，我们进入实战环节，准备从0开始开发一个skill了。

## 6. 从0开发一个skill：准备工作

从现在开始，我们一步步动手开发一个完整的skill——`weather`查询天气skill。看完这部分，哪怕你是新手，也能照着做出自己的第一个skill。

先说说我们需要准备什么。

### 前置知识

你不需要是资深开发工程师，但需要有一些基础：

- 会用基本的Git命令
- 会写一点基础的Python/JavaScript代码（我们这个例子用Python）
- 知道API是什么，会调用简单的HTTP接口
- 看得懂基本的JSON格式

就这些，足够了。

### 开发环境

你需要在电脑上装好这些：

1. **Python 3.8+**：我们用Python开发
2. **Git**：用来版本管理
3. **代码编辑器**：VS Code或者你习惯的任何编辑器
4. **一个支持MCP协议的Agent框架**：我们遵循Model Context Protocol标准接口，换任何框架都能用

如果你没有开发环境，也没关系，核心流程和思路是一样的。

### 获取API密钥

我们需要调用第三方天气API来获取数据，现在用免费的OpenWeatherMap API做例子：

1. 打开 [OpenWeatherMap](https://openweathermap.org/api) 注册账号
2. 免费版就够开发用了，注册后拿到你的API密钥
3. 保存好这个密钥，后面要用

## 7. 开发实战：一步步教你写一个weather skill

现在开始写代码，我们一步一步来。

### 第一步：填写skill元信息

首先，我们在`skill.json`里填写这个skill的元信息。这是给Agent框架看的，告诉框架这个skill叫什么名字，能做什么，需要什么参数。

```json
{
  "name": "weather",
  "description": "查询指定城市的实时天气信息",
  "version": "1.0.0",
  "author": "Your Name",
  "inputs": {
    "city": {
      "type": "string",
      "description": "要查询天气的城市名称，比如：Beijing",
      "required": true
    }
  },
  "output": {
    "type": "object",
    "description": "天气信息，包含温度、天气状况、湿度、风速"
  }
}
```

解释一下每个字段：
- `name`: skill的唯一名称，简短清晰
- `description`: 一句话说明这个skill做什么，Agent会根据这个描述决定什么时候调用它
- `version`: 版本号，方便迭代
- `inputs`: 定义输入参数，每个参数说明类型、描述、是否必填
- `output`: 定义输出格式和描述

这个文件很重要，Agent框架就是靠这个来理解你的skill。description一定要写清楚，不然Agent不知道什么时候该用它。

### 第二步：写依赖文件

在`requirements.txt`里写上我们需要依赖的Python包：

```txt
requests
python-dotenv
```

- `requests`: 用来发HTTP请求调用天气API
- `python-dotenv`: 用来从`.env`文件读取API密钥，不会把密钥写在代码里

安装依赖：
```bash
pip install -r requirements.txt
```

### 第三步：写主程序代码

现在到了最关键的`main.py`，我们来写主要逻辑：

```python
import os
import requests
from dotenv import load_dotenv

# 加载环境变量
load_dotenv()
API_KEY = os.getenv("OPENWEATHER_API_KEY")

def get_weather(city: str) -> dict:
    """
    获取指定城市的实时天气
    
    Args:
        city: 城市名称
        
    Returns:
        包含天气信息的字典
    """
    # API endpoint
    url = "https://api.openweathermap.org/data/2.5/weather"
    
    # 请求参数
    params = {
        "q": city,
        "appid": API_KEY,
        "units": "metric"  # 使用摄氏度
    }
    
    try:
        response = requests.get(url, params=params, timeout=10)
        response.raise_for_status()
        data = response.json()
        
        # 提取我们需要的信息
        result = {
            "city": city,
            "temperature": data["main"]["temp"],
            "feels_like": data["main"]["feels_like"],
            "condition": data["weather"][0]["main"],
            "description": data["weather"][0]["description"],
            "humidity": data["main"]["humidity"],
            "wind_speed": data["wind"]["speed"],
            "success": True
        }
        
        return result
        
    except requests.exceptions.RequestException as e:
        return {
            "success": False,
            "error": f"获取天气信息失败: {str(e)}"
        }

def main(inputs: dict) -> dict:
    """
    skill入口函数，Agent会调用这个函数
    
    Args:
        inputs: 包含输入参数的字典，格式由skill.json定义
        
    Returns:
        处理结果，格式由skill.json定义
    """
    city = inputs.get("city")
    
    if not city:
        return {
            "success": False,
            "error": "缺少必填参数: city"
        }
    
    return get_weather(city)

# 本地测试用
if __name__ == "__main__":
    result = main({"city": "Beijing"})
    print(result)
```

代码解释：

1. **`main`函数是入口**：Agent调用skill的时候，只会调用这个`main`函数，把参数传进来，拿到返回结果。所以你只要保证这个函数的输入输出符合定义就可以了。

2. **错误处理**：不管是参数缺失还是API调用失败，我们都返回清晰的错误信息，带上`success: False`，Agent知道调用失败了，可以做相应处理。不要直接抛出异常让程序崩溃。

3. **API密钥安全**：密钥从环境变量读取，不写死在代码里，也不会提交到Git，安全第一。

## 8. 测试与调试

开发完了，测试很重要。我们来看看怎么测试你的skill，常见问题怎么排查。

### 测试分层

测试skill可以分几个层次来做：

#### （1）本地单元测试

最基础的测试，就是在本地直接调用入口函数，验证返回对不对。就像我们上一章最后做的那样：

```python
if __name__ == "__main__":
    result = main({"city": "Beijing"})
    print(result)
```

这一步能解决大部分问题：
- 参数对不对
- API调用通不通
- 格式对不对
- 有没有语法错误

建议你做几个测试用例：
- 正常情况：正确城市名，应该返回正确结果
- 参数错误：不传city，应该返回错误信息
- 城市不存在：应该返回API错误，被正确处理

#### （2）在Agent框架中集成测试

本地测试通过了，接下来就要放到真正的Agent框架里测试，看看Agent能不能正确识别和调用它。

主要验证这几点：

1. **元信息能被正确读取**：框架能看到你的skill，能读到name、description、inputs这些信息
2. **描述能被正确理解**：用户问天气的时候，Agent能正确判断需要调用你的weather skill
3. **参数能被正确提取**：Agent能从用户问题中提取出城市名，传给你的skill
4. **结果能被正确处理**：skill返回结果后，Agent能正确理解并用自然语言回答用户

大部分问题出在这一步，常见问题我们后面说。

#### （3）端到端测试

完整走一遍用户对话流程：

```
用户：北京今天天气怎么样？
→ Agent理解需求，决定调用weather skill
→ Agent提取参数city=Beijing
→ skill调用API，返回结果
→ Agent拿到结果，生成自然语言回答
→ 用户得到回答
```

整个流程跑通一遍，没有问题才算真正完成。

### 常见问题排查

开发调试的时候，你可能会遇到这些问题：

**问题1：Agent从来不调用我的skill**

可能原因：
- `skill.json`里的`description`写得不清楚，Agent不知道什么时候该用它
- 描述太泛，比如只写了"获取信息"，没说清楚是获取什么信息
- 元信息格式不对，框架读不出来

解决方法：把description写具体，比如"查询指定城市的实时天气"，这样LLM一看就懂什么时候该用。

**问题2：Agent提取参数错了**

可能原因：
- `skill.json`里参数描述写得不清楚
- 用户问题本身模糊，比如用户只说"今天天气怎么样"，没说城市

解决方法：
- 在参数description里写清楚要求
- 如果参数缺失，你的skill应该返回清晰的错误，Agent会追问用户

**问题3：调用skill报错了**

可能原因：
- 依赖没装全
- API密钥不对
- 网络不通
- 输入格式不对

排查方法：先本地跑一遍，看看错误信息。大部分问题本地就能发现。

**问题4：返回结果Agent看不懂**

可能原因：
- 返回格式不对
- 没有统一的success字段表示成功失败
- 错误信息不清晰

解决方法：严格按照约定的格式返回，成功失败都返回结构化信息，方便Agent处理。

### 调试技巧

分享几个调试技巧：

1. **加日志**：在关键位置打日志，看看参数对不对，API返回什么
2. **用print看中间结果**：开发阶段大胆print，知道哪一步出问题
3. **最小化复现**：先本地单元测试，再集成测试，缩小问题范围
4. **检查JSON格式**：元信息和返回结果都要保证是合法JSON

测试调试完了，确认没问题，我们就可以发布了。下一章说说怎么发布和维护。

## 9. 发布与维护

开发测试都完成了，现在该发布了。发布之后还要持续维护。我们来看看怎么做。

### 发布到skill仓库

一般来说，每个Agent框架都会有一个公共的skill仓库，你可以把你的skill提交上去，让其他人也能用。

大致流程是：

1. **推送到GitHub**：把你的代码推送到GitHub公开仓库
2. **添加标签**：加上`ai-skill`、`weather`这样的标签，方便搜索
3. **提交PR**：到官方skill仓库提交PR，把你的skill加进去
4. **审核通过**：维护者审核通过后，你的skill就正式发布了

如果没有公共仓库，你也可以自己把代码放在GitHub，别人可以直接克隆下来使用。

### 版本管理

skill也要做好版本管理，遵循语义化版本：

- **主版本号**：不兼容的接口修改，比如输入参数变了
- **次版本号**：新增功能，向下兼容
- **修订号**：bug修复，不新增功能

比如：
- `1.0.0` → 第一个正式版本
- `1.0.1` → 修复了几个bug
- `1.1.0` → 新增了支持国内城市中文名称的功能
- `2.0.0` → 换了API提供商，接口参数变了，不兼容旧版本

版本号写在`skill.json`里，每次更新记得升级版本号。

### 怎么维护

发布了不是就完事了，skill需要持续维护：

- **监控稳定性**：关注API服务商有没有变化，会不会经常超时
- **及时修复bug**：用户反馈问题及时修复
- **响应需求**：用户提的合理需求可以考虑实现
- **更新依赖**：定期更新依赖包，修复安全漏洞

一个维护活跃的skill才能长久被人使用。没人维护的skill，用的人自然就少了。

### 写好文档很重要

很多开发者不重视文档，但对于skill来说，文档特别重要：

- 用户想使用你的skill，一看README就知道怎么配置、怎么用
- 其他开发者想贡献代码，一看文档就懂了整体结构
- 产品经理想评估要不要集成，看文档就知道这个skill能做什么不能做什么

记住：**skill的文档比代码更重要**，因为很多用你skill的人根本不看代码，他们只看README。

发布维护讲完了，下一章我们来聊一个大家都关心的话题——skill的商业化机会。

## 10. Skill的商业化机会

当AI Agent生态起来之后，skill就变成了一个新的赛道，这里面有很多机会。我们来梳理一下。

### 1. 垂直领域skill

每个垂直行业都有特定的需求，这些需求通用skill满足不了，需要专业的垂直skill。这是最大的机会。

比如：
- **金融领域**：查询股票信息、获取财报、计算投资收益
- **法律领域**：查询法条、分析合同、生成法律文书
- **医疗领域**：查询药品信息、解读检查报告
- **电商领域**：查询商品信息、对比价格、下单
- **企业内部**：查询内部数据、走审批流程、操作内部系统

这些垂直领域的skill，有很强的专业性，不是随便什么人都能做，壁垒比较高。如果你懂某个行业，开发一个好用的垂直skill，会很有价值。

### 2. 按调用付费

商业模式也很清晰，最直接的就是按调用次数付费：

- 免费额度：每个月免费调用多少次
- 超出付费：超出部分按次收费，几分钱到几毛钱一次不等
- 包月套餐：固定费用， unlimited调用

对于开发者来说，只要你的skill有用，就能源源不断获得收入。而且是被动收入，开发完一次，一直收费。

### 3. 企业定制

企业用户有特定的需求，愿意花钱定制专门的skill：

- 对接企业内部系统
- 定制业务逻辑
- 专属的稳定性保障

客单价从几千到几万不等，比To C按调用付费赚得多。如果你有企业资源，这是很好的方向。

**现在很多企业都在做AI转型，需要把内部能力包装成skill给Agent调用，这里需求很大。**

### 4. Skill市场平台

除了做skill，你也可以做skill市场平台，撮合开发者和用户：

- 开发者上传skill
- 用户购买使用
- 平台抽成

类似苹果App Store的模式，只不过卖的是AI skill。当生态起来了，平台价值很大。

当然，做平台门槛比较高，需要先有流量，有生态，适合有一定基础的团队做。

### 5. 开源商业化

很多skill可以先开源获得用户，然后卖企业版增值服务：

- 基础版开源免费
- 企业版提供更多功能、技术支持、SLA保障
- 按年收订阅费

这是现在很成熟的模式，既获得了社区用户，又有商业化收入。

### 机会总结

AI Agent生态才刚刚开始，skill作为生态中的核心模块，机会很多：

| 方向 | 门槛 | 潜在收益 | 适合 |
|------|------|----------|------|
| 垂直领域skill | 中（需要行业知识） | 中高 | 懂行业的开发者 |
| 通用工具skill | 低 | 中（薄利多销） | 个人开发者 |
| 企业定制 | 中高 | 高 | 有企业资源的团队 |
| Skill市场平台 | 高 | 很高 | 有流量有基础的团队 |
| 开源+订阅 | 中 | 中高 | 开源社区背景团队 |

不管你是个人开发者还是团队，都能找到适合自己的机会。

现在这个领域还是蓝海，早点进场，更容易占坑。下一章我们做个总结，分享一些经验。

## 11. 最佳实践与经验总结

最后，我们把开发skill的最佳实践总结一下，方便大家参考。

### 设计层面

1. **小而美比大而全好**：做小skill，做好一件事就行，不要什么都想做。小skill更容易复用，更容易组合。

2. **接口清晰，参数明确**：输入输出一定要定义清楚，description写明白，LLM才能正确调用。

3. **错误处理要完善**：失败了不要直接崩溃，返回清晰的错误信息，让Agent知道怎么处理。

4. **单一职责**：一个skill只做一件事，这是最基本也是最重要的原则。

### 开发层面

1. **依赖最小化**：不要依赖太多乱七八糟的包，依赖越少，冲突越少，维护越简单。

2. **API密钥不要放代码里**：用环境变量或者配置文件，一定要加入`.gitignore`，别不小心把密钥提交到GitHub了。

3. **写好单元测试**：核心逻辑加几个测试用例，改代码不 regression。

4. **语义化版本管理**：版本号说清楚，升级的时候用户知道有什么变化。

### 文档层面

1. **README要写清楚**：干什么用，怎么配置，输入输出什么格式，举个例子，这几点一定要有。

2. **更新CHANGELOG**：每个版本改了什么，写清楚，用户升级的时候知道要注意什么。

3. **许可证要加上**：开源不开源，说清楚，避免法律问题。

### 经验感悟

从这几年做skill的经验来看，我有几点感悟：

**第一，skill生态才刚刚开始**。现在大家都在讲Agent，但真正丰富的skill生态还没建立起来，机会很多。

**第二，专业领域的skill最值钱**。通用skill比如查天气搜新闻，谁都能做，壁垒低。垂直专业领域，你懂行业，你做出来的skill就是有价值，别人抢不走。

**第三，稳定比什么都重要**。用户用你的skill，最基础的要求就是稳定，别经常挂。稳定了，用户才敢用在生产环境。
